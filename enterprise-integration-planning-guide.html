Planning the Integration of a New System into the Enterprise Landscape

A step-by-step architect-level guide for onboarding a new SaaS or vendor application across Supply Chain, Product Creation, Finance, and Direct-to-Consumer domains.

1
Phase One
Discovery & Landscape Assessment

Before writing a single diagram, understand what already exists — on both sides of the integration boundary.

  • Audit the vendor's integration capabilities: REST APIs, webhooks, EDI, SFTP, or pre-built connectors
  • Map existing enterprise integration patterns already in use across your target domains
  • Identify the system of record for every shared entity (Product, Order, Inventory, Financial record)
  • Engage the vendor's technical team to understand their data model, rate limits, and versioning policy
  • Review the Enterprise Integration Architecture team's current standards and approved patterns
Note
Engage architecture governance early. All external vendor integrations must align with the CDT architectural standards before design work begins. A late governance review is the most common cause of rework.
Watch out
Avoid assuming data model parity. Vendor data models rarely match internal canonical models. Document mismatches during discovery — don't defer them to the build phase.
Deliverables
  • Vendor integration capability matrix
  • System-of-record map per entity type
  • Architecture standards alignment checklist
2
Phase Two
Domain & Data Flow Mapping

Model what data needs to flow, in which direction, and at what frequency — for each domain pair.

Domain Entities exchanged Direction Trigger
Supply chain / planning Inventory levels, purchase orders, forecasts, shipment status Bidirectional Event-driven + nightly batch
Product creation / PLM BOMs, material specs, colorways, product attributes Inbound to vendor On product lifecycle event
Finance / ERP (SAP) GL entries, cost centers, invoices, payment status Bidirectional Scheduled batch + event triggers
DTC / eCommerce Orders, catalog data, pricing, customer data Bidirectional Real-time event-driven
Tip
For each row in the table above, also define the latency SLA (e.g., "inventory updates must propagate within 5 minutes during peak trading") and the failure behavior (retry? dead-letter? alert?). These drive pattern selection in Phase 3.
Deliverables
  • Data flow diagram per domain (entities, direction, frequency)
  • Latency SLA matrix
  • Failure behavior decision log
3
Phase Three
Integration Pattern Selection

Map each data flow to the right architectural pattern. Mismatched patterns are the leading cause of reliability and maintainability problems.

Pattern When to use Platform tooling
Async event-driven Real-time triggers: order placed, inventory threshold crossed Azure Service Bus Topics / queues
Sync API call Request-response lookups: price check, stock availability query Azure APIM
Scheduled batch Bulk syncs: overnight PLM → ERP product data reconciliation Azure Logic Apps Azure Data Factory
Orchestration Multi-step workflows crossing two or more domains Logic Apps stateful
Choreography Loosely coupled domain events where no central coordinator is needed Service Bus pub/sub
Key decision
Orchestration vs. choreography is the most consequential architectural decision in this phase. For flows crossing SAP + PLM + Supply Chain simultaneously, orchestration provides better observability and error handling. For high-volume DTC event streams, choreography is more resilient and scalable.
Deliverables
  • Pattern decision record per integration flow
  • Architecture Decision Record (ADR) for orchestration vs. choreography
  • Integration topology diagram
4
Phase Four
API & Canonical Data Model Design

This is the core technical design phase. Decisions here govern every downstream implementation choice.

  • Define canonical data models for all shared entities: Product Order InventoryUnit CostRecord
  • Design vendor-to-canonical transformation mappings — document every field mismatch explicitly
  • Establish API versioning strategy (URI-based /v1/ vs. header-based) and publish via APIM
  • Define backward-compatibility rules to protect downstream consumers from breaking changes
  • For SAP flows: choose the right interface mechanism
SAP interface type Use case Notes
IDoc / BAPI Legacy batch integrations, EDI-style messaging Mature but verbose; prefer for existing SAP-to-SAP patterns
SAP Integration Suite Managed SAP-native middleware Best for SAP Cloud / S/4HANA; aligns with SAP roadmap
REST API via APIM Modern event-driven or sync flows Preferred for new integrations; provides gateway-level governance
Tip
Design the canonical model first, independently of both the vendor's model and the internal ERP model. The canonical model is a contract, not a copy of either side. This decoupling is what allows either side to evolve without breaking the other.
Deliverables
  • Canonical data model definitions (OpenAPI / JSON Schema)
  • Field mapping documents per integration flow
  • API versioning policy document
  • SAP interface selection rationale
5
Phase Five
Security & Governance Review

No external system connects without passing a security and governance gate. This is a hard checkpoint, not a formality.

  • Authentication: OAuth 2.0 client credentials or API keys managed via APIM policies
  • Authorization: role-based access scoped per domain — Supply Chain data ≠ Finance data access
  • Data residency review: where does the vendor store data? Are global brand requirements met across US, EMEA, APAC?
  • PII & sensitive data handling: customer order data (DTC) requires explicit classification and handling rules
  • Vendor security posture review: SOC 2 Type II, ISO 27001, penetration testing cadence, incident response SLA
  • Network topology review: VNet integration, private endpoints, or public API with IP allowlisting?
Watch out
DTC flows carry customer PII. Any integration touching the Direct-to-Consumer domain must go through a full data privacy review — including GDPR (EMEA), CCPA (California), and any applicable local data laws. This cannot be retrofitted after go-live.
Deliverables
  • Security review sign-off from InfoSec
  • Data classification matrix per entity and domain
  • Vendor security assessment results
  • Network topology diagram with access control annotations
6
Phase Six
Non-Functional Requirements

Define NFRs explicitly before build begins. They drive infrastructure sizing, SLA commitments, and on-call responsibilities.

NFR category Questions to answer
Throughput Peak transactions per minute? What is the seasonal multiplier (e.g., holiday peak for DTC)?
Latency Acceptable end-to-end delay per flow? P95 and P99 targets, not just averages.
Reliability Retry logic configured? Dead-letter queues monitored? Idempotency keys on all write operations?
Scalability Auto-scale triggers defined in Azure? Maximum concurrency limits set on consumers?
Observability Correlation IDs propagated end-to-end? Dashboards configured? Alert thresholds set?
Disaster recovery RPO and RTO targets per domain? Failover topology documented and tested?
Tip
Model peak load at 3–5× the expected average for DTC-connected flows. E-commerce workloads are highly spiky and integration layers are frequently the first bottleneck to surface under load.
Deliverables
  • NFR specification document, signed off by product and engineering leads
  • Capacity model and Azure resource sizing estimate
  • SLA commitments documented per integration flow
7
Phase Seven
Build, Test & Operate

Design-first, test-first. Build sequence matters — starting with infrastructure before contracts is the most common source of integration debt.

Build sequence
  • Publish API contracts in Azure APIM using design-first OpenAPI specifications
  • Build transformation logic and integration flows in lower environments (dev → staging)
  • Map vendor data to canonical model — handle field mismatches explicitly with documented transformations
  • Test with production-like volumes, including deliberate failure scenarios
  • Promote to production with staged rollout and feature flags where possible
Testing requirements
Test type What it validates
Unit tests Transformation logic: field mappings, data type coercions, null handling
Contract tests Vendor schema vs. canonical model; detect breaking changes before they reach production
End-to-end tests Full flow per domain: SaaS → integration layer → target system → response
Performance tests Validate NFR targets under peak load, with real-world message sizes
Chaos / resilience tests Vendor downtime, malformed payloads, SAP unavailability, Service Bus throttling
Operate
  • Instrument all integration flows with correlation IDs for end-to-end distributed tracing
  • Configure dead-letter queue alerting and automated requeue procedures
  • Build and publish runbooks for the top 5 expected failure scenarios per domain
  • Onboard to observability dashboards with alert routing to the on-call team
  • Schedule quarterly vendor API change reviews to catch deprecation notices early
Note
Observability is not optional. An integration without correlation IDs, dead-letter monitoring, and structured logging is unoperatable at scale. Build this in from day one — retrofitting observability into a running integration is expensive and error-prone.
