Python Senior Engineer Interview Q&A

20 most commonly asked questions with answers and code examples

Core Python
1. What is the GIL and how does it affect multithreading? core python ▼
Answer

The Global Interpreter Lock (GIL) is a mutex in CPython that allows only one thread to execute Python bytecode at a time — even on multi-core machines. This means Python threads do not achieve true parallelism for CPU-bound tasks.

# GIL allows concurrency for I/O-bound but NOT CPU-bound tasks
import threading, multiprocessing

# Threading works for I/O — GIL released during I/O wait
def fetch_data():
    response = requests.get("https://api.example.com")  # GIL released

# Threading does NOT help CPU-bound — GIL is held
def crunch():
    return sum(range(10_000_000))  # GIL never released

# Use multiprocessing for CPU-bound — each process has its own GIL
with multiprocessing.Pool() as pool:
    results = pool.map(crunch, [1, 2, 3, 4])
Interview tip: GIL is a CPython detail — Jython and PyPy don't have it. For CPU-bound use multiprocessing. For I/O-bound use threading or asyncio.
2. Difference between deepcopy and shallow copy? core python ▼
Answer

A shallow copy creates a new object but references the same nested objects. A deepcopy creates a completely independent copy including all nested objects.

import copy

original = {"name": "Alice", "scores": [90, 85, 92]}

# Shallow copy — nested list is SHARED
shallow = copy.copy(original)
shallow["scores"].append(100)
print(original["scores"])  # [90, 85, 92, 100] ← MUTATED!

# Deep copy — fully independent
deep = copy.deepcopy(original)
deep["scores"].append(999)
print(original["scores"])  # [90, 85, 92] ← unchanged ✅

# Assignment is neither — just another reference
alias = original
alias["name"] = "Bob"
print(original["name"])  # "Bob" — same object!
Interview tip: Assignment = same object. copy() = new container, shared contents. deepcopy() = fully independent clone.
3. How do Python decorators work? Write one from scratch. core python ▼
Answer

A decorator is a function that takes another function, wraps it with extra behaviour, and returns the wrapped version. The @syntax is just syntactic sugar for function reassignment.

from functools import wraps
import time

def timer(func):
    @wraps(func)  # preserves __name__, __doc__
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} took {time.time()-start:.3f}s")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(1)

slow_function()  # slow_function took 1.001s

# Decorator with arguments
def retry(times=3):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for i in range(times):
                try: return func(*args, **kwargs)
                except Exception: pass
        return wrapper
    return decorator
Interview tip: Always use @wraps(func) — without it func.__name__ becomes "wrapper" which breaks logging and debugging.
4. What are generators and how do they differ from lists? core python ▼
Answer

Generators are lazy iterators that yield one item at a time on demand. They use O(1) memory regardless of size, while lists store everything in memory at once.

import sys

# List — all in memory
squares_list = [x**2 for x in range(1_000_000)]
print(sys.getsizeof(squares_list))  # ~8MB

# Generator — lazy, one at a time
squares_gen = (x**2 for x in range(1_000_000))
print(sys.getsizeof(squares_gen))   # ~120 bytes!

# Generator function with yield
def fibonacci():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b

fib = fibonacci()
print(next(fib))  # 0
print(next(fib))  # 1
print(next(fib))  # 1
Interview tip: Generators are single-pass — once exhausted they are empty. Use them for large datasets, infinite sequences, or pipelines. Use lists when you need random access or multiple iterations.
5. Explain *args and **kwargs with examples. core python ▼
Answer

*args captures extra positional arguments as a tuple. **kwargs captures extra keyword arguments as a dictionary. Together they make functions accept any number of arguments.

# *args — variable positional arguments
def add(*args):
    return sum(args)

add(1, 2, 3)        # 6
add(1, 2, 3, 4, 5)  # 15

# **kwargs — variable keyword arguments
def greet(**kwargs):
    for key, val in kwargs.items():
        print(f"{key}: {val}")

greet(name="Alice", role="Engineer")

# Combined — order matters: pos, *args, keyword-only, **kwargs
def full(pos, *args, keyword_only, **kwargs):
    print(pos, args, keyword_only, kwargs)

full("a", "b", "c", keyword_only="x", extra="y")
# a ('b','c') x {'extra': 'y'}
Interview tip: Common use — wrapper functions that forward all arguments. Order must be: positional, *args, keyword-only, **kwargs.
6. What is a context manager? Write one using __enter__ and __exit__. core python ▼
Answer

A context manager defines setup and teardown logic for a "with" block. __enter__ runs on entry and returns the resource. __exit__ runs on exit — even if an exception occurs.

class DatabaseConnection:
    def __init__(self, url):
        self.url = url

    def __enter__(self):
        print("Opening connection")
        self.conn = connect(self.url)
        return self.conn

    def __exit__(self, exc_type, exc_val, exc_tb):
        print("Closing connection")
        self.conn.close()
        return False  # False = don't suppress exceptions

with DatabaseConnection("postgres://...") as conn:
    conn.execute("SELECT * FROM users")

# Cleaner with contextlib
from contextlib import contextmanager

@contextmanager
def timer(label):
    import time; start = time.time()
    yield
    print(f"{label}: {time.time()-start:.2f}s")
Interview tip: Return True from __exit__ to suppress exceptions, False to propagate. The @contextmanager approach is cleaner for simple cases.
7. What is the difference between is and == in Python? core python ▼
Answer

== checks value equality (same data?). "is" checks identity (same object in memory?). Always use == for value comparison and "is" only for None, True, False.

a = [1, 2, 3]
b = [1, 2, 3]
print(a == b)   # True — same values
print(a is b)  # False — different objects

c = a
print(a is c)  # True — same object

# Python caches small ints (-5 to 256)
x = 256; y = 256
print(x is y)  # True — cached!
x = 257; y = 257
print(x is y)  # False — not cached

# Always use "is" for None
if result is None:   # correct ✅
    pass
Interview tip: Never use "is" to compare strings or integers in general code — Python's interning is an implementation detail and produces surprising results.
OOP
8. Difference between @staticmethod, @classmethod, and instance methods? oop ▼
Answer

Instance methods receive self (the instance). Class methods receive cls (the class) and can modify class state. Static methods receive neither — they are utility functions scoped inside the class.

class Employee:
    company = "Nike"

    def __init__(self, name, salary):
        self.name = name
        self.salary = salary

    # Instance method — access to self
    def get_info(self):
        return f"{self.name} at {self.company}"

    # Class method — alternative constructor pattern
    @classmethod
    def from_dict(cls, data):
        return cls(data["name"], data["salary"])

    # Static method — utility, no self or cls
    @staticmethod
    def validate_salary(salary):
        return salary > 0

emp = Employee.from_dict({"name": "Alice", "salary": 90000})
Interview tip: @classmethod is commonly used for alternative constructors (from_dict, from_json). @staticmethod is for utility functions that logically belong to the class but don't need class/instance state.
9. Explain MRO (Method Resolution Order) and how super() works. oop ▼
Answer

MRO defines the order Python searches for methods in an inheritance chain using the C3 linearization algorithm. super() follows MRO to call the next class — not necessarily the direct parent.

class A:
    def hello(self): print("A")

class B(A):
    def hello(self):
        print("B")
        super().hello()

class C(A):
    def hello(self):
        print("C")
        super().hello()

class D(B, C):  # Diamond inheritance
    pass

# MRO: D → B → C → A
print([c.__name__ for c in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']

D().hello()
# B
# C  ← super() in B calls C, not A!
# A
Interview tip: super() means "call the next class in MRO" — not "call my parent". This is the key insight most candidates miss in diamond inheritance questions.
10. What are dunder/magic methods? Give examples beyond __init__. oop ▼
Answer

Dunder methods (double underscore) let you define how objects behave with built-in Python operations — comparisons, arithmetic, iteration, string representation, length, and more.

class Vector:
    def __init__(self, x, y): self.x, self.y = x, y
    def __repr__(self): return f"Vector({self.x}, {self.y})"
    def __str__(self):  return f"({self.x}, {self.y})"
    def __add__(self, other): return Vector(self.x+other.x, self.y+other.y)
    def __len__(self):  return 2
    def __eq__(self, other): return self.x==other.x and self.y==other.y
    def __iter__(self):
        yield self.x
        yield self.y

v1 = Vector(1, 2); v2 = Vector(3, 4)
print(v1 + v2)   # (4, 6)  — uses __add__
print(len(v1))   # 2       — uses __len__
x, y = v1        # unpacking — uses __iter__
Interview tip: Always implement __repr__ — it's used in debugging, logging, and the REPL. If you only implement one, implement __repr__.
11. What is the difference between __str__ and __repr__? oop ▼
Answer

__repr__ is for developers — unambiguous, ideally eval()-able. __str__ is for end users — readable and friendly. print() calls __str__ first, then falls back to __repr__.

class Product:
    def __init__(self, name, price):
        self.name = name; self.price = price

    def __repr__(self):
        return f"Product(name={self.name!r}, price={self.price!r})"

    def __str__(self):
        return f"{self.name} - ${self.price:.2f}"

p = Product("Air Max", 129.99)
print(p)        # Air Max - $129.99       ← __str__
repr(p)         # Product(name='Air Max', price=129.99) ← __repr__
print([p])      # [Product(...)]  ← always __repr__ in containers
Interview tip: In lists and dicts Python always calls __repr__ not __str__. Always define __repr__. Add __str__ only when you need a different user-friendly format.
Async & Concurrency
12. Difference between asyncio.gather() and asyncio.as_completed()? async ▼
Answer

gather() runs all coroutines concurrently and returns results in submission order after ALL complete. as_completed() yields results one by one in completion order — you process each as it arrives.

import asyncio

async def fetch(url, delay):
    await asyncio.sleep(delay)
    return f"data from {url}"

# gather — wait for ALL, results in submission order
results = await asyncio.gather(
    fetch("/users", 3),
    fetch("/orders", 1),
    fetch("/products", 2),
)
# all 3 done before you see any result
# order: [users, orders, products] — submission order

# as_completed — results arrive as they finish
coros = [fetch("/users", 3), fetch("/orders", 1), fetch("/products", 2)]
for coro in asyncio.as_completed(coros):
    result = await coro
    print(result)  # orders at 1s, products at 2s, users at 3s
Interview tip: Use gather() when you need all results together. Use as_completed() when you want to update a progress bar or stop early when you find what you need.
13. What happens if you use a blocking call inside async def in FastAPI? async ▼
Answer

A blocking call inside an async def route freezes the entire asyncio event loop. While it runs, no other requests can be processed — turning your async server into a single-threaded synchronous one.

# BAD — blocks entire event loop
@app.get("/bad")
async def bad_route():
    result = requests.get("https://api.example.com")  # BLOCKS!
    time.sleep(2)   # BLOCKS all other requests!
    return result.json()

# GOOD — non-blocking async
@app.get("/good")
async def good_route():
    async with httpx.AsyncClient() as client:
        result = await client.get("https://api.example.com")
    return result.json()

# ALSO OK — sync route runs in threadpool automatically
@app.get("/sync")
def sync_route():
    result = requests.get("https://api.example.com")  # safe in def
    return result.json()
Interview tip: async def routes must ONLY use async-compatible libraries (httpx, asyncpg, aioredis). If you must use a blocking library, use a regular def route and let FastAPI handle it in a thread pool.
14. How does asyncio.Semaphore work and when would you use it? async ▼
Answer

A Semaphore limits how many coroutines can run a section of code simultaneously. Others wait until a slot is freed. Essential for rate-limiting API calls or controlling database connection pool usage.

import asyncio

semaphore = asyncio.Semaphore(2)  # max 2 at a time

async def fetch(url):
    async with semaphore:       # acquire slot
        print(f"Fetching {url}")
        await asyncio.sleep(1)
        return f"data from {url}"
    # slot released automatically

async def main():
    urls = [f"/api/{i}" for i in range(5)]
    results = await asyncio.gather(*[fetch(url) for url in urls])
    # Only 2 run at a time despite 5 scheduled

# Real world: AWS Bedrock rate limiting
bedrock_sem = asyncio.Semaphore(5)
async def invoke_bedrock(prompt):
    async with bedrock_sem:
        return await bedrock.invoke_model(prompt)
Interview tip: Semaphore vs Lock — Lock = Semaphore(1). Use Semaphore(N) when you have a known capacity limit like API rate limits, DB connection pools, or thread pool size.
15. Difference between threading, multiprocessing, and asyncio? async ▼
Answer

Threading uses OS threads — good for I/O-bound but limited by GIL for CPU. Multiprocessing uses separate processes — true CPU parallelism, no GIL. Asyncio uses a single thread with cooperative coroutines — most efficient for high-concurrency I/O.

# Threading — I/O bound, limited by GIL for CPU work
import threading
threads = [threading.Thread(target=fetch_data) for _ in range(10)]
for t in threads: t.start()

# Multiprocessing — CPU bound, true parallelism
import multiprocessing
with multiprocessing.Pool(4) as pool:
    results = pool.map(crunch_numbers, data)

# Asyncio — I/O bound, single thread, most scalable
import asyncio
results = await asyncio.gather(*[fetch(url) for url in urls])
Interview tip: I/O + moderate concurrency → threading. I/O + high concurrency (1000s) → asyncio. CPU-bound → multiprocessing. FastAPI + many API calls → asyncio is ideal.
Performance
16. How does @lru_cache work and what are its trade-offs? performance ▼
Answer

@lru_cache stores function results in a dictionary keyed by arguments. On repeated calls with the same arguments it returns instantly. LRU evicts the Least Recently Used entry when the cache is full.

from functools import lru_cache, cache

# Without cache — O(2^n)
def fib_slow(n):
    if n < 2: return n
    return fib_slow(n-1) + fib_slow(n-2)

# With cache — O(n)
@lru_cache(maxsize=128)
def fib_fast(n):
    if n < 2: return n
    return fib_fast(n-1) + fib_fast(n-2)

fib_fast(50)
print(fib_fast.cache_info())
# CacheInfo(hits=48, misses=51, maxsize=128, currsize=51)
fib_fast.cache_clear()

# @cache = lru_cache(maxsize=None), Python 3.9+
@cache
def expensive(n): ...
Interview tip: Arguments must be hashable — can't cache calls with list or dict args. Trade-off: speed vs memory growth. Use cache_clear() to invalidate stale data.
17. When would you use a generator instead of a list comprehension? performance ▼
Answer

Use a generator when the dataset is large, you only need to iterate once, or you are building a processing pipeline. Use a list when you need random access, multiple iterations, or len().

import sys

squares_list = [x**2 for x in range(1_000_000)]
squares_gen  = (x**2 for x in range(1_000_000))
print(sys.getsizeof(squares_list))  # ~8MB
print(sys.getsizeof(squares_gen))   # ~120 bytes

# Generator pipeline — process 10GB log with constant memory
def read_file(path):
    with open(path) as f:
        for line in f: yield line.strip()

def filter_errors(lines):
    return (l for l in lines if "ERROR" in l)

pipeline = filter_errors(read_file("app.log"))
for record in pipeline:
    process(record)  # constant O(1) memory!
Interview tip: Generators are single-pass — once exhausted they are empty. If you need to iterate multiple times or need len(), use a list. Otherwise prefer generators for memory efficiency.
18. What are __slots__ in Python classes and why use them? performance ▼
Answer

__slots__ restricts instance attributes to a fixed set and stores them in a compact array instead of a per-instance __dict__. This reduces memory by ~40-50% and speeds up attribute access.

# Without __slots__ — each instance has a __dict__
class Point:
    def __init__(self, x, y):
        self.x = x; self.y = y

# With __slots__ — no __dict__, fixed attributes only
class PointSlotted:
    __slots__ = ['x', 'y']
    def __init__(self, x, y):
        self.x = x; self.y = y

import sys
p1 = Point(1, 2)
p2 = PointSlotted(1, 2)
print(sys.getsizeof(p1))  # 48 + dict overhead ~232 bytes
print(sys.getsizeof(p2))  # 56 bytes — much smaller!

p1.z = 3  # OK — has __dict__
p2.z = 3  # AttributeError — no __dict__!
Interview tip: Use __slots__ when creating millions of instances (ML feature vectors, game entities). Trade-off: you lose dynamic attribute assignment and some __dict__-based features like default pickle.
Data Structures
19. When would you use a deque instead of a list? data structures ▼
Answer

Use a deque when you need fast O(1) append and pop from BOTH ends. list.insert(0) and list.pop(0) are O(n) because they shift all elements. deque.appendleft() and deque.popleft() are O(1).

from collections import deque

# List — O(n) at the front
lst = [1, 2, 3]
lst.insert(0, 0)  # O(n) — shifts everything
lst.pop(0)        # O(n) — shifts everything

# deque — O(1) at both ends
dq = deque([1, 2, 3])
dq.appendleft(0)  # O(1) ✅
dq.popleft()      # O(1) ✅
dq.append(4)      # O(1) ✅

# maxlen — automatic sliding window
last_5 = deque(maxlen=5)
for i in range(10):
    last_5.append(i)
print(last_5)  # deque([5, 6, 7, 8, 9], maxlen=5)

# BFS graph traversal — always use deque!
queue = deque([start_node])
while queue:
    node = queue.popleft()  # O(1) FIFO
Interview tip: deque is the go-to for BFS — always use popleft() not pop() to maintain FIFO order. The maxlen parameter is great for sliding window problems.
20. Explain the time complexity of dict, set, and list operations. data structures ▼
Answer

dict and set use hash tables — O(1) average for lookup, insert, delete. List uses a dynamic array — O(1) for append/pop at end, O(n) for insert/search/delete at arbitrary positions.

# DICT — hash table, O(1) average
d = {"key": "value"}
d["key"]          # O(1) lookup
d["new"] = 1      # O(1) insert
del d["key"]      # O(1) delete
"key" in d        # O(1) membership

# SET — hash table, O(1) average
s = {1, 2, 3}
s.add(4)          # O(1)
3 in s            # O(1) ← much faster than list!
s1 & s2           # O(min(len(s1), len(s2))) intersection

# LIST — dynamic array
lst = [1, 2, 3]
lst.append(4)     # O(1) amortized
lst.pop()         # O(1)
lst[2]            # O(1) index access
lst.insert(0, 0)  # O(n) — shifts elements!
3 in lst          # O(n) — linear search!

# Practical tip — convert list to set for repeated membership checks
items = list(range(1_000_000))
items_set = set(items)
999999 in items      # O(n) slow
999999 in items_set  # O(1) instant
Interview tip: If checking membership repeatedly, convert list to set first. Worst case for dict/set is O(n) due to hash collisions but extremely rare in practice.
