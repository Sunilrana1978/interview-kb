Part A — Implementation Lifecycle

1. Planning & Design

Understand the corpus model. A corpus is a single ingested collection — think of it as a folder of related files optimized for similarity search. You can query one corpus for targeted retrieval, or several at once for broader recall. Decide upfront:

  • One corpus per domain/topic vs. one corpus with metadata filtering
  • Which corpora a given agent needs to query — the VertexAiRagRetrieval tool accepts a list of RagResource entries, so a single retrieval tool can span multiple corpora if broader recall is the goal
  • Whether retrieval needs tuning per use case (a factual-lookup agent wants a tight distance threshold; an exploratory research agent wants a looser one and higher top-k)

Vector store tier decision (see Section 13 for detail). By default, RAG Engine uses RagManagedDb, a fully managed Spanner instance offered in two tiers: - Basic — cost-effective, fixed configuration (~0.1 nodes); ideal for experiments, small data, latency-insensitive workloads - Scaled — production-grade with autoscaling (1–10 nodes); designed for large datasets and performance-sensitive apps

Choose this at design time, not after load-testing surprises you — moving from Basic to Scaled tier is a capacity change you want planned, not reactive.

Source inventory. RAG Engine ingests from local files, Cloud Storage, or Google Drive. Map each source to:

Source Ingestion path Refresh mechanism
Local files One-time rag.upload_file() Manual re-upload
Cloud Storage rag.import_files() from gs:// URI Scheduled re-import job
Google Drive rag.import_files() from Drive resource Scheduled re-import job

Naming convention. Fix corpus display names early:

corpus-<domain>-<env>

2. Prerequisites & Environment Setup

Enable APIs and set environment:

gcloud services enable aiplatform.googleapis.com --project=YOUR_PROJECT_ID

export GOOGLE_GENAI_USE_VERTEXAI=True
export GOOGLE_CLOUD_PROJECT=YOUR_PROJECT_ID
export GOOGLE_CLOUD_LOCATION=us-central1

IAM roles:

Role Purpose Assign to
roles/aiplatform.user Create/manage corpora, import files, query Ingestion pipeline SA, platform team
roles/aiplatform.viewer Read-only inspection of corpora Auditors, read-only tooling
roles/storage.objectViewer Read source GCS bucket Ingestion SA (on the source bucket)

Agent Engine deployment identity. If deploying the agent to Vertex AI Agent Engine, deployment automatically grants the new agent identity permission to query its configured RAG corpus — verify this binding explicitly rather than assuming it, especially in a locked-down project where default grants may be restricted by org policy.

Staging resources. RAG Engine workflows typically provision: - A source GCS bucket for raw documents - A staging GCS bucket for Agent Engine deployment artifacts

Keep both environment-scoped (gs://source-docs-dev, gs://source-docs-prod), never shared across environments.

3. Provisioning the RAG Corpus

from vertexai.preview import rag
import vertexai

vertexai.init(project="YOUR_PROJECT_ID", location="us-central1")

corpus = rag.create_corpus(
    display_name="corpus-support-prod",
    description="Production support knowledge base",
)
print(corpus.name)  # projects/PROJECT/locations/us-central1/ragCorpora/CORPUS_ID

Vector DB tier configuration is set at corpus creation — confirm the tier parameter against current SDK documentation before provisioning, since this surface is under vertexai.preview and has moved between releases. Treat corpus creation as a Terraform-adjacent, script-driven step (wrap the Python call in your CI pipeline) rather than a manual console action, so corpus IDs are captured as pipeline outputs, not tribal knowledge.

Post-create validation: list files on the new corpus (should be empty) to confirm it's queryable before wiring any ingestion job to it.

4. Data Ingestion Pipeline

rag.import_files(
    corpus_name=corpus.name,
    paths=["gs://your-bucket/docs/"],
    chunk_size=512,
    chunk_overlap=100,
)

Ingestion pipeline stages:

Source (local / GCS / Drive)
   → Import (rag.import_files)
   → Chunk (configurable size/overlap)
   → Embed
   → Index → Corpus
   → Queryable
  • Chunk size and overlap are tunable here, unlike Gemini File Search or Vertex AI Search — this is the main reason to choose RAG Engine over those options. Start from a documented baseline (e.g., 512/100) and adjust based on eval results (Section 7), not intuition.
  • Re-import strategy: RAG Engine does not distinguish "incremental" vs. "full" the way Vertex AI Search does — plan your own logic for detecting changed/deleted source files and re-importing only what's needed, or accept periodic full re-imports for smaller corpora.
  • File-level tooling: the RAG corpus tools support listing and deleting individual files by corpus — build a small maintenance script around this rather than relying on full corpus rebuilds for every content correction.

5. Retrieval Configuration

from google.adk.tools.retrieval.vertex_ai_rag_retrieval import VertexAiRagRetrieval

retrieval_tool = VertexAiRagRetrieval(
    name="retrieve_documentation",
    description="Retrieve relevant documents from the knowledge base",
    rag_resources=[rag.RagResource(rag_corpus=corpus.name)],
    similarity_top_k=10,
    vector_distance_threshold=0.6,
)
Parameter Effect Tuning guidance
similarity_top_k How many chunks are retrieved Higher = more recall, more prompt-context cost; start at 5–10
vector_distance_threshold Minimum similarity to include a chunk Lower threshold = stricter relevance; tune against eval set, not by feel
rag_resources Which corpus/corpora to query Multiple entries broaden recall across corpora in one call

Document your chosen values and the eval results that justified them — these are the parameters most likely to be "tuned" ad hoc by a well-meaning engineer later without re-running the eval suite.

6. Agent Integration (ADK)

from google.adk.agents import Agent

root_agent = Agent(
    model="gemini-2.5-flash",
    name="rag_agent",
    instruction="You are a helpful assistant. Use the retrieval tool to answer questions based on the provided documents. Always cite the source document.",
    tools=[retrieval_tool],
)

Critical constraint: the VertexAiRagRetrieval tool can only be used by itself within an agent instance — it does not combine with other tools (like GoogleSearch) on the same agent. If you need both retrieval and general search, use an orchestrator agent that delegates to a RAG-specialist sub-agent and a search-specialist sub-agent, mirroring the pattern used for Gemini File Search.

Deployment to Agent Engine:

from vertexai.preview import reasoning_engines

remote_agent = reasoning_engines.ReasoningEngine.create(
    root_agent,
    requirements=["google-cloud-aiplatform[adk,agent_engines]"],
)

This packages the agent, pushes it to Agent Runtime, and (per Google's standard flow) automatically grants the new agent identity permission to query the configured RAG corpus — confirm this grant in IAM rather than assuming it silently succeeded.

7. Evaluation & Testing

Test type What it checks Cadence
Retrieval precision/recall @ k Right chunks retrieved for labeled questions Every chunking/embedding change
Answer groundedness + citation accuracy Generated answer traces to retrieved chunks, citations point to correct source Every prompt/model change
Threshold sensitivity How vector_distance_threshold changes affect recall vs. noise Before any parameter change ships
Multi-corpus interference If querying multiple corpora, does cross-corpus retrieval degrade relevance? Whenever a new corpus is added to a shared retrieval tool
Load/latency test End-to-end retrieval + generation latency under expected concurrency Before each production promotion

Build a fixed labeled question set per corpus and re-run it on every chunking, embedding, or threshold change — this is the guardrail against silent retrieval-quality drift, which is the dominant failure mode for tunable RAG systems.

8. CI/CD & Promotion Across Environments

Pipeline shape:

PR merged → CI script creates/updates corpus in dev (idempotent)
  → import test fixtures → run eval suite
  → promote agent config + corpus reference to staging
  → run eval suite + load test → manual approval
  → promote to prod → smoke test → monitor
  • Corpus creation and file import are Python/API calls, not Terraform resources today — wrap them in your CI pipeline as an explicit, idempotent step (check "does a corpus with this display name already exist" before creating) rather than a one-off manual script.
  • Each environment gets its own corpus created from the same pipeline definition — never point a staging agent at the production corpus ID, even temporarily for testing.
  • Store the corpus resource name as a pipeline output/secret consumed by agent deployment config, never hardcoded in agent source.

9. Security & Compliance Sign-off

Before go-live, confirm:

  • [ ] Ingestion service account has read-only access to source buckets, and aiplatform.user scoped appropriately
  • [ ] Agent Engine's auto-granted corpus access is confirmed in IAM, not assumed
  • [ ] Audit logging enabled on aiplatform.googleapis.com for corpus create/import/query operations
  • [ ] A documented process exists for removing a specific file from a corpus (for takedown/compliance requests)
  • [ ] Chosen RagManagedDb tier (Basic/Scaled) matches the data sensitivity and durability requirements agreed with stakeholders

Part B — Post-Deployment Operations

10. Monitoring & Observability

Key signals to dashboard (Cloud Monitoring / Vertex AI dashboards):

Metric Why
Retrieval latency (p50/p95/p99) User-facing performance; correlate spikes with corpus size growth
Import job success/failure Silent import failures leave the corpus stale with no visible symptom
Chunk count / corpus size trend Sudden drops indicate accidental file deletion; sudden spikes indicate duplicate ingestion
Retrieval empty-result rate Rising trend suggests threshold miscalibration or corpus gaps
RagManagedDb node utilization (Scaled tier) Approaching the 10-node autoscale ceiling signals a tier/capacity conversation

Logging. Enable Cloud Audit Logs (Data Access) on aiplatform.googleapis.com scoped to RAG corpus operations — every import_files, delete_file, and query call should be attributable.

Alerting thresholds (starting points): - Import job failure → page on-call (stale-corpus risk compounds silently, same as Vertex AI Search) - Empty-result rate > 15% over 1 hour → alert (usually a threshold or corpus regression) - RagManagedDb node utilization sustained > 80% (Scaled tier) → capacity review

11. Corpus Freshness & Reindexing

  • Maintain a freshness SLA per source, and monitor actual import-to-queryable latency against it — same discipline as Vertex AI Search, but note RAG Engine gives you no built-in incremental/full distinction, so your own tooling must track which source files have already been imported.
  • Since file-level delete/list tools exist on the corpus, prefer targeted re-import of changed files over full corpus rebuilds for routine content updates — full rebuilds should be reserved for scheduled reconciliation (e.g., monthly) to catch drift.
  • Log every ingestion run's source path list, chunk count delta, and corpus size before/after — this is your primary tool when a document seems to have "disappeared" from retrieval.

12. Incident Response & Runbooks

Runbook: Document not being retrieved 1. Confirm the file was actually imported — list files on the corpus and check for the expected file reference. 2. If present but not surfacing, check vector_distance_threshold — a recently tightened threshold is a common silent cause of "the document exists but never gets retrieved." 3. If absent, check the last ingestion job's logs for a parsing/format failure on that specific file.

Runbook: Retrieval quality regression (wrong/irrelevant chunks surfacing) 1. Check whether chunk_size, chunk_overlap, similarity_top_k, or vector_distance_threshold changed recently — diff against the documented baseline in Section 5. 2. Re-run the labeled eval suite against the current corpus state to quantify the regression. 3. Roll back the parameter change first (cheapest fix); only re-chunk/re-import if the regression persists after rollback.

Runbook: Agent errors combining retrieval with other tools 1. Confirm the agent definition doesn't combine VertexAiRagRetrieval with another tool on the same agent instance — this is a hard platform limitation, not a bug. 2. If both capabilities are needed, refactor to an orchestrator + specialist sub-agent pattern.

Runbook: Latency degradation on Scaled tier 1. Check RagManagedDb node utilization against the autoscale ceiling (10 nodes). 2. Check for a concurrent large import job competing with query serving. 3. If sustained and infrastructure-side, escalate to Google Cloud support with project ID, corpus resource name, and time window.

Escalation path: on-call engineer → platform team lead → Google Cloud support (for confirmed backend/infrastructure issues).

13. Scaling & Capacity Management (RagManagedDb tiers)

Tier Configuration Best for Scaling behavior
Basic Fixed, ~0.1 nodes Experiments, small corpora, latency-insensitive use No autoscaling — plan for a tier upgrade before production traffic
Scaled Autoscaling, 1–10 nodes Large datasets, performance-sensitive production apps Scales within its band; sustained ceiling utilization requires a capacity conversation with Google, not just a config change
  • Decide tier per environment: Basic in dev/staging is usually sufficient; Scaled is the default assumption for production once real user traffic is expected.
  • Re-evaluate tier choice whenever corpus size or query volume materially changes — a corpus that started as an experiment on Basic tier and quietly became production-critical is a common capacity-planning gap.

14. Cost Management

  • Cost drivers: corpus storage (via RagManagedDb), embedding generation on import, and query-time retrieval + generation (Gemini model calls).
  • Track cost per corpus, not just per project, so you can attribute spend to the domain/team that owns each corpus — mirrors the same discipline as Vertex AI Search.
  • Scaled-tier RagManagedDb has a materially different cost profile than Basic — confirm the tier matches actual need before defaulting to Scaled "just in case."
  • Review low-traffic corpora periodically; a corpus receiving few queries but frequent re-imports is a common silent cost sink, same failure pattern as under-used Vertex AI Search data stores.

15. Backup, Versioning & Rollback

  • The corpus is rebuildable from source (GCS/Drive) plus your ingestion pipeline definition — treat the source system + pipeline code as the real backup, not the corpus itself.
  • Version-control your ingestion pipeline script (chunk size, overlap, source paths) and your retrieval tool configuration (top_k, distance_threshold) — these are the actual "configuration" that needs rollback capability.
  • For a bad chunking/re-import, keep the previous corpus alive under a versioned name (corpus-support-prod-v2) until the new one is validated, then cut the agent over via a config change rather than deleting the old corpus immediately — this gives you an instant rollback path.
  • Document a file-level deletion procedure (for takedown/compliance) that confirms removal from the corpus index, not just from the source bucket.

16. Access Review & Governance Cadence

Activity Cadence
Review IAM bindings on ingestion and query-serving identities Quarterly
Confirm Agent Engine auto-granted corpus permissions are still scoped correctly Quarterly
Review corpus inventory (unused/orphaned corpora) Semi-annually
Audit log review for anomalous import/query patterns Monthly, or continuous via alerting

17. Change Management / Release Process

  1. All corpus provisioning, chunking config, and retrieval parameter changes go through PR review — no ad hoc console/notebook changes against production corpora.
  2. Any change to chunk_size, chunk_overlap, similarity_top_k, or vector_distance_threshold requires an eval suite run against the labeled question set before merge.
  3. Production promotions require a documented rollback plan — usually reverting the parameter change or cutting the agent back to the previous versioned corpus.
  4. Maintain a per-corpus changelog: what changed, when, why, eval results before/after, approver.

18. Troubleshooting Reference

Symptom Likely cause First check
Document never retrieved despite being imported Threshold too strict, or chunking split it awkwardly Compare vector_distance_threshold to eval baseline; inspect chunk boundaries
Retrieval quality regression after a change Chunk size/overlap or threshold change Diff against documented baseline; re-run eval suite
Agent throws a tool-combination error VertexAiRagRetrieval combined with another tool on one agent Refactor to orchestrator + sub-agent pattern
Rising latency on Scaled tier Node utilization near ceiling, or concurrent large import Check RagManagedDb utilization + recent import job status
Corpus size unexpectedly shrank Accidental file deletion or bad reconciliation logic in custom ingestion script Check ingestion job logs for delete calls

19. Appendix — CLI / Python Cheatsheet

from vertexai.preview import rag
import vertexai

vertexai.init(project="YOUR_PROJECT_ID", location="us-central1")

# Create corpus
corpus = rag.create_corpus(display_name="corpus-support-prod")

# Import files
rag.import_files(corpus_name=corpus.name, paths=["gs://your-bucket/docs/"],
                  chunk_size=512, chunk_overlap=100)

# List files in a corpus
rag.list_files(corpus_name=corpus.name)

# Delete a specific file
rag.delete_file(name="projects/.../ragFiles/FILE_ID")

# Delete a corpus entirely
rag.delete_corpus(name=corpus.name)

# Enable API
gcloud services enable aiplatform.googleapis.com

# Check IAM bindings
gcloud projects get-iam-policy YOUR_PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:sa-rag-ingest@*"

This surface lives under vertexai.preview — confirm exact method signatures, tier-configuration parameters, and quota limits against current Google Cloud documentation before running in production, since preview APIs move faster than stable ones.
