Interview Ready Guide

PySpark on AWS EMR
End-to-End Playbook

From cluster sizing to SparkSession, partitioning, salting, transformations, and actions — everything you need to explain confidently.

1

Assess Your Data Size First

Before spinning up any cluster, you must know the size of your raw data sitting in S3 or Snowflake. This drives every sizing decision downstream.

Key questions to answer:

How many GB or TB of raw data will PySpark read? How complex are your transformations — simple filters, or heavy joins and aggregations? What is your expected output data size? Is this a one-time batch or recurring job?

Rule of Thumb: Cluster memory ≥ 2× to 3× raw data size (for shuffles + intermediate data)
2

Choose the Right Node Type & Size

EMR clusters have three node types. Master Node — coordinates tasks, runs the Driver and Cluster Manager. One per cluster, doesn't need to be large. Core Nodes — run Executors and process data. These are your workhorses. Task Nodes — optional, temporary compute for burst workloads.

For most PySpark workloads on S3, use one Master + multiple Core nodes of the same instance type.

Data Size Instance Type vCPUs RAM # Core Nodes Use Case
10–50 GB r5.xlarge 4 32 GB 2–4 Dev / Small
50–500 GB r5.2xlarge 8 64 GB 4–10 Production
500 GB–5 TB r5.4xlarge 16 128 GB 10–30 Large Scale
5 TB+ r5.8xlarge 32 256 GB 20–50+ Enterprise
3

Decide Executors Per Node & Memory Allocation

On each Core Node, you need to reserve memory for the OS and YARN overhead, then divide the rest among Executors. A good rule: allocate 2–4 cores per Executor and leave 1 core + ~5 GB per node for the OS.

Example: r5.2xlarge — 8 vCPUs, 64 GB RAM

OS
5GB
Executor 1 — 13 GB
Executor 2 — 13 GB
Executor 3 — 13 GB
Overhead
~7GB
Available RAM = Total RAM − OS (5 GB) = 64 − 5 = 59 GB
Executors per Node = (8 vCPUs − 1) ÷ 2 cores per executor = 3 executors
Memory per Executor = 59 GB ÷ 3 ≈ 19 GB (keep ~13 GB usable, rest for overhead)

If you have 5 Core Nodes each with 3 Executors → 15 total Executors in your cluster.

4

Start the SparkSession

SparkSession is the entry point for your PySpark application. It connects to the already-running EMR cluster and discovers available Executors automatically.

pyspark_session.py 
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

# Create SparkSession — connects to EMR cluster automatically
spark = (SparkSession
    .builder
    .appName("MyPySparkJob")
    .# Executor config — matches your cluster sizing
    config("spark.executor.instances", "15")    # 5 nodes × 3 executors
    .config("spark.executor.cores", "2")         # 2 cores per executor
    .config("spark.executor.memory", "13g")      # 13 GB per executor
    .config("spark.driver.memory", "5g")         # Driver memory
    .config("spark.sql.shuffle.partitions", "200") # Default shuffle partitions
    .getOrCreate()
)
5

Read Data & Apply Partitioning

When PySpark reads from S3 or Snowflake, data is pulled into Executor memory as partitions. You can control how many partitions and how data is distributed.

read_and_partition.py 
# Read from S3 — PySpark auto-partitions based on file blocks
df = spark.read.parquet("s3://my-bucket/raw-data/")

# Check how many partitions were created automatically
print(df.rdd.getNumPartitions())

# Repartition by a column key — good for joins on customer_id
df_partitioned = df.repartition(60, "customer_id")
# 60 partitions = ~4 partitions per executor (15 executors)

# Or repartition by count only (no key)
df_repartitioned = df.repartition(100)

# Coalesce — reduce partitions (no shuffle, use before writing)
df_coalesced = df.coalesce(10)
Ideal partition size: 100–200 MB per partition
Number of partitions ≈ Total Data Size ÷ 128 MB (or 2–4× total executor cores)
6

Handle Skewed Data with Salting

If one key value (e.g., customer_id = 999) has millions of rows while others have hundreds, one Executor gets overloaded. Salting adds a random number to the key to spread data evenly.

salting_technique.py 
from pyspark.sql import functions as F

# Step 1: Define salt factor (how many buckets to split skewed key into)
SALT_FACTOR = 10

# Step 2: Add salt column to the skewed DataFrame
df_salted = df.withColumn(
    "salted_key",
    F.concat(
        F.col("customer_id").cast("string"),
        F.lit("_"),
        (F.rand() * SALT_FACTOR).cast("int").cast("string")
    )
)

# Step 3: Repartition on the salted key
df_salted = df_salted.repartition(60, "salted_key")

# Step 4: Perform transformations (groupBy, aggregation, etc.)
result = df_salted.groupBy("salted_key").agg(
    F.sum("amount").alias("total_amount")
)

# Step 5: Remove salt and re-aggregate to get final result
final = result.withColumn(
    "customer_id",
    F.split(F.col("salted_key"), "_")[0]
).groupBy("customer_id").agg(
    F.sum("total_amount").alias("total_amount")
).drop("salted_key")

When to Use Salting

When one key has far more rows than others. When joins cause one executor to hang. When you see skewed task durations in Spark UI.

Choosing Salt Factor

Salt factor = number of extra buckets. Use 10–20 for moderate skew, up to 100 for severe skew. Higher salt = better distribution but more overhead.

7

Transformations, Actions & Writing Output

Transformations are lazy — they build a plan but don't execute. Actions trigger actual computation across Executors.

transformations_actions.py 
# ── TRANSFORMATIONS (lazy, build DAG only) ──────────────────

# Filter rows
df_filtered = df.filter(F.col("status") == "active")

# Add new column
df_transformed = df_filtered.withColumn(
    "total_value", F.col("qty") * F.col("price")
)

# Join two DataFrames (partition both on join key first!)
df_orders = df_orders.repartition(60, "customer_id")
df_customers = df_customers.repartition(60, "customer_id")

df_joined = df_orders.join(
    df_customers, "customer_id", "inner"
)

# GroupBy aggregation
df_agg = df_joined.groupBy("customer_id").agg(
    F.sum("total_value").alias("lifetime_value"),
    F.count("*").alias("order_count")
)

# ── ACTIONS (trigger execution across all executors) ─────────

# Count rows — triggers full job
print(df_agg.count())

# Show sample — triggers partial execution
df_agg.show(10)

# ── WRITE OUTPUT back to S3 or Snowflake ────────────────────
(df_agg
    .coalesce(10)                        # reduce files before writing
    .write
    .mode("overwrite")
    .partitionBy("region")              # physical partition in S3
    .parquet("s3://my-bucket/output/")
)

End-to-End Flow — Interview Answer

Assess Data SizeCheck raw data size in S3/Snowflake. Determine transformation complexity.
Size the EMR ClusterChoose instance type (r5.2xlarge), number of Core nodes, executors per node, memory per executor.
Spin Up EMR ClusterLaunch 1 Master + N Core nodes. EMR provisions virtual servers on each node automatically.
Create SparkSessionYour PySpark script creates a SparkSession which discovers all available Executors on the cluster.
Read from S3 / SnowflakeData is pulled from centralized storage into Executor memory as distributed partitions (RDD underneath).
Repartition + Salt if SkewedRepartition by join key. Apply salting if data is skewed to distribute load evenly across Executors.
Apply TransformationsFilters, joins, aggregations — all lazy, building a DAG. No data moves yet.
Trigger an Actioncount(), show(), write() — triggers actual parallel execution across all Executors.
Write Output to S3 / SnowflakeFinal results collected from Executors, coalesced, and written back to persistent storage.

💡 Interview Tip — How to Answer Sizing Questions

  • Start with data size: "My raw data is X GB in S3."
  • Pick instance type: "I'd use r5.2xlarge with 8 vCPUs and 64 GB RAM."
  • Calculate executors: "Reserve 5 GB for OS, 2 cores per executor = 3 executors per node, 13 GB each."
  • Calculate nodes: "With 100 GB data, I need ~5 nodes for 15 total executors."
  • Mention partitioning key: "I'd partition by customer_id since my joins are on that key."
  • Mention salting: "If customer_id is skewed, I'd salt it with rand() × 10 to spread load."
  • Close with lazy evaluation: "All transformations are lazy — they only execute when I trigger an action like write()."
