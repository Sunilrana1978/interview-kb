AWS CSPM Services — Deep Dive

Technical detail, configuration mechanics, and production-style examples for each core AWS service used in Cloud Security Posture Management.

AWS Config

Foundation · Detective

How it works

Config runs a Configuration Recorder per account/Region that snapshots supported resource types and streams every change to an S3 bucket as a config item, with a full history and relationship graph (e.g., which SG is attached to which ENI). Config Rules (AWS-managed Lambda or custom Lambda/Guard rules) evaluate resources on a schedule or on-change and mark them COMPLIANT / NON_COMPLIANT. Conformance Packs bundle dozens of rules mapped to a standard (CIS AWS Foundations, PCI-DSS, NIST 800-53) and can deploy org-wide via Organization Conformance Packs.

Production example

RETAIL / E-COMMERCE PLATFORM A multi-account retail org (prod, staging, PCI-scoped payment account) deploys the Operational-Best-Practices-for-PCI-DSS conformance pack org-wide via Control Tower. The s3-bucket-public-read-prohibited and restricted-ssh managed rules run continuously. When a developer accidentally opens SG port 22 to 0.0.0.0/0 on an EC2 auto-scaling group during a hotfix, Config flags it NON_COMPLIANT within minutes, an EventBridge rule fires an SSM Automation document that revokes the ingress rule, and the event lands in Security Hub for audit trail — no human in the loop for the first response.

Key config knobs

  • recording-group — all-resource-types vs. scoped list (cost control at scale)
  • Aggregators for cross-account/cross-Region visibility from a security-tooling account
  • Remediation via SSM Automation documents attached directly to a rule
PITFALL Recording "all resource types" in every account gets expensive fast at Nike/retail scale (thousands of resources changing per hour). Most mature orgs scope the recorder to security-relevant resource types and rely on CloudTrail + Security Hub for the rest.

AWS Security Hub

Aggregator · Detective

How it works

Security Hub ingests findings from Config, GuardDuty, Inspector, Macie, IAM Access Analyzer, Firewall Manager, and third-party integrations (Trellix, CrowdStrike, Prisma Cloud, Qualys) — all normalized to the AWS Security Finding Format (ASFF). It runs its own standards checks (FSBP — Foundational Security Best Practices, CIS, PCI DSS, NIST 800-53) and computes a per-account security score. A delegated administrator account aggregates findings across the whole AWS Organization.

Production example

SOC / MTD INTEGRATION PATTERN A security-tooling account is set as Security Hub's delegated admin across ~40 member accounts. An EventBridge rule matches findings with Severity.Label IN [HIGH, CRITICAL] and routes them to an SQS queue consumed by a Lambda that posts a normalized alert into Trellix XDR (or a SOAR platform) for MTD analyst triage — this is exactly the aggregation pattern to know cold when explaining how AWS-native posture data feeds a Managed Threat Defense workflow.

Key mechanics

ConceptPurpose
ASFFCommon JSON schema so Config/GuardDuty/partner findings are queryable the same way
InsightsSaved queries, e.g. "EC2 instances with critical vulns exposed to internet"
Custom ActionsManual trigger to send a finding to an EventBridge target (ticketing, SOAR)
PITFALL Enabling all three standards (FSBP + CIS + PCI) on every account creates duplicate/overlapping findings for the same misconfiguration. Most teams pick FSBP as baseline and layer PCI/CIS only on in-scope accounts.

Amazon GuardDuty

Threat Detection

How it works

GuardDuty analyzes VPC Flow Logs, DNS query logs, and CloudTrail management/data events (no infrastructure to deploy — it's fed from existing logs) using threat-intel feeds and anomaly ML models. Optional protection plans extend it to EKS Audit Logs, S3 data events, RDS login activity, Lambda network activity, and Malware Protection (EBS snapshot scanning on suspicious EC2/container workloads).

Production example

CREDENTIAL COMPROMISE An IAM access key belonging to a CI/CD pipeline service account leaks into a public GitHub repo. Within minutes, GuardDuty's UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration and Recon:IAMUser/MaliciousIPCaller findings fire because API calls originate from a Tor exit node. The finding streams to Security Hub, EventBridge triggers a Lambda that disables the access key via iam:UpdateAccessKey, and Detective is used afterward to trace exactly which API calls the attacker made before containment.

Production example

CONTAINER WORKLOAD On an EKS cluster running customer-facing microservices, GuardDuty EKS Protection flags Execution:Kubernetes/ExecInSuspiciousDirectory when an attacker who compromised a pod runs a shell in /tmp. Malware Protection then auto-scans the underlying EBS volume and confirms a cryptominer binary, feeding a CRITICAL finding into Security Hub.
PITFALL GuardDuty is Region-scoped — teams that only enable it in us-east-1 miss activity in Regions they forgot they use. Enable org-wide, all-Region, via a delegated admin.

Amazon Inspector

Vulnerability Management

How it works

Inspector continuously scans EC2 instances (via SSM agent, no separate agent needed), ECR container images (on push and on a schedule for existing images), and Lambda function code/layers for known CVEs, using NVD data plus AWS's own scoring. It also runs a network reachability check for EC2 — flagging instances with a vulnerable package that's also reachable from the internet, which prioritizes remediation far better than raw CVSS score alone.

Production example

CONTAINER SUPPLY CHAIN A CI pipeline pushes a new microservice image to ECR. Inspector scans it on push and flags a CRITICAL CVE in a base-image OpenSSL package. The pipeline is gated on a Security Hub Insight query (no CRITICAL findings on the image digest) via a CodeBuild step that calls GetFindings — the deployment is blocked before it ever reaches production EKS.

Production example

NETWORK REACHABILITY PRIORITIZATION Out of 4,000 EC2 instances with medium/high CVEs, Inspector's reachability analysis shows only 30 are actually internet-reachable on the vulnerable port. Patch-Tuesday effort gets focused on those 30 first instead of triaging all 4,000 by CVSS score alone.
PITFALL Lambda scanning only covers packages Inspector recognizes in deployment packages/layers — custom-built binaries or unusual runtimes can slip through; don't treat it as a complete SBOM solution.

Amazon Macie

Data Protection

How it works

Macie uses managed data identifiers (and custom regex-based identifiers) to classify objects in S3 for PII, credentials, and financial data, and separately evaluates bucket-level security posture (public access, encryption, sharing outside the account) regardless of content scanning.

Production example

DATA EXPOSURE PREVENTION A data-engineering team stands up a new S3 bucket for a customer analytics export and misconfigures the bucket policy to allow account-wide read. Macie's automated sensitive-data discovery job flags the bucket both for public/cross-account access and for containing objects matching the CREDIT_CARD_NUMBER managed identifier. The combined severity trips a Security Hub CRITICAL finding same-day, well before a scheduled quarterly audit would have caught it.
PITFALL Full-bucket sensitive-data discovery jobs bill per GB scanned — running them unscoped across a multi-petabyte data lake gets expensive; most teams scope jobs to specific prefixes or sample percentages.

IAM Access Analyzer

Identity Posture

How it works

Access Analyzer uses automated reasoning (not simulation) to mathematically evaluate resource policies (S3, KMS, IAM roles, Lambda, SQS, Secrets Manager) and flag any that grant access to an entity outside a defined "zone of trust" (the account or the org). A separate Unused Access analyzer flags IAM roles/users with permissions or access keys unused for 90+ days — key for least-privilege cleanup.

Production example

CROSS-ACCOUNT TRUST DRIFT During a vendor offboarding, a cross-account IAM role trust policy that was scoped to a third-party's account ID is never removed after the contract ends. Access Analyzer continues to flag it as an external-access finding every scan cycle until it's cleaned up — providing an audit trail that the exposure existed and for how long, which matters for a SOC 2 finding response.

Production example

LEAST PRIVILEGE AT SCALE Unused Access findings across 200 IAM roles in a platform-engineering account show 40 roles with S3 full-access permissions that were never actually used to write, only read. Policies get right-sized to s3:GetObject/ListBucket only, cutting blast radius if any of those roles are later compromised.

AWS Organizations + Service Control Policies

Preventive Guardrails

How it works

SCPs are JSON policies attached at the Organization root, OU, or account level that set the maximum available permissions — even an account admin with a full-access IAM policy can't exceed what the SCP allows. Unlike Config/Security Hub (detective — flag after the fact), SCPs are preventive — the API call is denied outright.

Production example

DECOMMISSIONING GUARDRAIL During decommissioning of legacy Trellix-based services in AWS, an SCP is attached to the "legacy-security-tools" OU that denies ec2:RunInstances and cloudformation:CreateStack for any new resource creation — allowing only Delete*/Terminate* actions during the wind-down window, preventing anyone from accidentally standing up new workloads in an account that's mid-decommission.

Production example

REGION LOCKDOWN A retail org operating under data-residency requirements attaches an SCP denying all actions outside us-east-1 and us-west-2 at the root OU, closing off the possibility of shadow-IT resources spinning up in an unmonitored Region that Config/GuardDuty aren't even watching.
PITFALL SCPs don't grant permissions, only restrict — a common mistake is expecting an SCP allow-list to substitute for IAM policies. It's a ceiling, not a grant.

AWS Control Tower

Landing Zone Automation

How it works

Control Tower automates a multi-account landing zone: it provisions a management account, a log-archive account, an audit/security account, and standard OUs, then attaches a baseline of mandatory guardrails (SCPs) and strongly-recommended Config rules. Account Factory templates new-account creation so every account starts from the same compliant baseline.

Production example

MERGERS & ACQUISITIONS ONBOARDING When an acquired company's AWS accounts are enrolled into the parent org's Control Tower landing zone, Account Factory immediately applies the standard guardrail set (mandatory encryption, CloudTrail logging, no public S3) — instead of a manual months-long security review before the accounts can be trusted, the guardrails self-enforce from day one and drift is reported centrally.
PITFALL Control Tower guardrail drift (someone manually detaches an SCP) is detected but not always auto-remediated — dashboards need active monitoring, it isn't fully "set and forget."

AWS Firewall Manager

Network Policy

How it works

Firewall Manager centrally pushes and enforces WAF rule groups, Security Group policies, Network Firewall, and Shield Advanced protections across every account/resource in an OU — and can auto-remediate resources (e.g., a Security Group) that drift from the enforced baseline.

Production example

WAF BASELINE AT SCALE A retail org with 50+ public-facing ALBs across business-unit accounts uses Firewall Manager to push a common WAF rule group blocking SQLi/XSS patterns and rate-limiting to every ALB tagged internet-facing=true, org-wide — new ALBs that match the tag are auto-enrolled without any per-account WAF setup.
PITFALL Firewall Manager requires Organizations + a delegated admin and doesn't retroactively fix resources outside its policy scope tags — untagged resources silently fall outside enforcement.

Amazon Detective

Investigation

How it works

Detective ingests CloudTrail, VPC Flow Logs, and GuardDuty findings into a behavior graph spanning up to a year, letting an analyst pivot from "this IAM role triggered a GuardDuty finding" to "here's every API call, IP, and resource that role touched in the prior 30 days" without manually correlating raw CloudTrail logs.

Production example

MTD INCIDENT TRIAGE After GuardDuty flags anomalous API calls from an EC2 instance role, an MTD analyst opens the finding directly in Detective and sees the instance also made unusual calls to Secrets Manager 20 minutes earlier and communicated with a rare external IP — turning a single alert into a full attack-chain timeline in a few clicks instead of hours of manual CloudTrail Athena queries.

AWS Audit Manager

Compliance Evidence

How it works

Audit Manager maps live AWS activity (Config compliance state, CloudTrail events, resource inventories) to controls in a chosen framework (SOC 2, ISO 27001, PCI DSS) and continuously collects evidence into an assessment, instead of security teams manually screenshotting console pages before an audit.

Production example

SOC 2 AUDIT CYCLE Ahead of an annual SOC 2 Type II audit, the security team runs a standing Audit Manager assessment against the SOC 2 framework all year; by audit time, 80% of the evidence (encryption status, access reviews, logging configuration) is already collected and timestamped, cutting weeks off manual evidence-gathering.

AWS Trusted Advisor

Baseline Checks

How it works

Trusted Advisor runs lightweight account-level checks (root account MFA, exposed access keys, open security groups on common ports, S3 bucket permissions) without any setup, refreshed periodically. Business/Enterprise Support unlocks the full security check category plus programmatic access via the Support API.

Production example

QUICK POSTURE SNAPSHOT Before a new account is enrolled into full Config/Security Hub monitoring, a quick Trusted Advisor check immediately surfaces that root MFA isn't enabled and two Security Groups allow 0.0.0.0/0 on port 3389 — the fastest possible "are we already on fire" check during initial account assessment, well before deeper tooling is even deployed.

Trellix Integration Patterns

ePO · XDR · DXL ↔ AWS

Why this matters

AWS-native services generate posture and threat data; Trellix consumes, correlates, and acts on it as part of a broader MTD/XDR workflow. The integration is almost never "Trellix replaces Security Hub" — it's Trellix subscribing to AWS findings and, separately, AWS ingesting Trellix telemetry so both sides have a complete picture. There are three integration points worth knowing cold: Security Hub as the AWS-side aggregator, EventBridge as the transport, and DXL/ePO as the Trellix-side control plane.

Pattern 1 — Security Hub → EventBridge → Trellix XDR

This is the most common direction: AWS findings flow outward to Trellix for correlation with endpoint telemetry.

Config / GuardDuty / Inspector / Macie→ Security Hub (ASFF)→ EventBridge rule→ Lambda transformer→ Trellix XDR API / DXL broker
  • An EventBridge rule filters on source: aws.securityhub and detail.findings[].Severity.Label IN [HIGH, CRITICAL]
  • A Lambda reshapes the ASFF payload into the Trellix XDR ingestion schema (or publishes to a DXL topic if using the on-prem ePO bridge) and calls the Trellix API
  • Trellix correlates the cloud finding against endpoint/EDR telemetry for the same asset, producing a single fused incident instead of two disconnected alerts
PRODUCTION EXAMPLE — CLOUD + ENDPOINT CORRELATION GuardDuty flags Backdoor:EC2/C2Activity on an EC2 instance. The finding streams through Security Hub → EventBridge → Lambda → Trellix XDR. Trellix XDR already has ENS (Endpoint Security) telemetry from the same host showing a suspicious process spawned moments earlier. XDR auto-correlates the two into one incident, and because the host also runs the Trellix agent, an MTD analyst can trigger endpoint isolation directly from the fused incident — something GuardDuty alone can't do since it has no endpoint control plane.

Pattern 2 — Trellix ePO/DXL → AWS (inbound enrichment)

Less common but increasingly used: Trellix-side detections (ENS malware verdicts, MTD analyst-confirmed incidents) get pushed back into AWS so cloud-native remediation (SCP, SSM Automation, Lambda) can act — e.g., quarantining an IAM role or EC2 instance tied to a confirmed-malicious host.

  • DXL (Data Exchange Layer) publishes an event when ePO confirms a malware verdict or MTD closes an incident as malicious
  • An integration service (Lambda behind API Gateway, or a DXL-to-EventBridge bridge) translates the DXL message into a PutEvents call on a custom EventBridge bus
  • A downstream rule triggers SSM Automation to isolate the EC2 instance's Security Group or tag it for forensic snapshot — closing the loop from endpoint detection to cloud containment
PRODUCTION EXAMPLE — CONTAINMENT LOOP MTD confirms a ransomware precursor on an EC2-hosted Windows workload via ENS. The DXL event is bridged into a custom EventBridge bus; a rule matches on detail-type: trellix.malware.confirmed and runs an SSM Automation runbook that swaps the instance's Security Group to an "isolated" SG (deny all except forensics subnet) within seconds — faster than waiting for a human to jump from the Trellix console into the AWS console to do it manually.

Pattern 3 — CSPM posture feed for Trellix Cloud module

If using Trellix's cloud security/CSPM module (as opposed to just XDR/ENS), it typically reads AWS posture via a cross-account IAM role with read-only access to Config, Security Hub, and CloudTrail — the same read pattern as any third-party CSPM tool (Wiz, Prisma Cloud) rather than a push model.

  • Cross-account role with a trust policy scoped to Trellix's AWS account ID, granted securityhub:GetFindings, config:Describe*, config:Get*, read-only
  • External ID required in the trust policy (standard third-party access hardening — same pattern for any SaaS security vendor)
  • Trellix polls or subscribes via EventBridge partner event bus rather than maintaining its own change-detection logic

Architecture decision points

DecisionOptionsConsideration
Transport to TrellixEventBridge + Lambda vs. DXL broker vs. direct API pollingEventBridge is lowest-latency and serverless; DXL matters if on-prem ePO is still authoritative
DirectionAWS→Trellix only vs. bidirectionalBidirectional (containment loop) needs more careful IAM scoping — inbound write access to AWS is higher risk
Cross-account accessPush (AWS calls Trellix API) vs. pull (Trellix assumes a read role)Pull is simpler to secure (no AWS credentials given to Trellix); push allows near-real-time delivery
Finding dedupLet Trellix XDR dedupe vs. pre-filter in LambdaPre-filtering in Lambda (severity/resource-type) reduces noise and API cost before it ever reaches Trellix
PITFALL Standing up the containment loop (Pattern 2) without tightly scoping the IAM role the automation runs under is a real risk — a compromised or misfiring DXL bridge with broad ec2:*/iam:* permissions becomes its own attack surface. Scope the automation role to only the specific isolate/quarantine actions it needs (e.g., ec2:ModifyInstanceAttribute for SG swap only), not broad admin access.
