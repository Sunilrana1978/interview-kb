Deploying an ADK Agent to Vertex AI Agent Engine, and Plugging It Into Your React App

This guide walks through: deploying your ADK agent to Vertex AI Agent Engine (the managed Agent Runtime), building a small Cloud Run bridge that handles your Okta login and a custom domain, and connecting your existing React app's chat UI to it.

Why the bridge? Agent Engine's API lives on Google's own domain (aiplatform.googleapis.com) and expects a Google OAuth token — not your Okta token, and not something you can put your own DNS record in front of. A thin Cloud Run service in between lets you keep your custom domain, validate Okta, and call Agent Engine using Google credentials server-side.

Part 1 — Deploy the Agent to Agent Engine

Step 1: Install the SDK

pip install --upgrade google-cloud-aiplatform[agent_engines,adk] --break-system-packages

Step 2: Set environment variables

export GOOGLE_CLOUD_PROJECT=YOUR_PROJECT_ID
export GOOGLE_CLOUD_LOCATION=us-central1
export STAGING_BUCKET=gs://YOUR_STAGING_BUCKET
export GOOGLE_GENAI_USE_VERTEXAI=True

Step 3: Wrap your agent as an AdkApp

In your agent's project (the same agent.py you've been developing locally), wrap the root agent:

from google.adk.agents import Agent
from vertexai import agent_engines

root_agent = Agent(
    model="gemini-2.5-flash",
    name="cx_agent",
    instruction="You are a helpful customer experience assistant.",
    tools=[...],   # your existing tools
)

app = agent_engines.AdkApp(
    agent=root_agent,
    enable_tracing=True,   # sends traces to Cloud Trace automatically
)

Step 4: Test locally before deploying

import asyncio

async def test():
    async for event in app.async_stream_query(
        user_id="test-user",
        message="Hello, what can you help me with?",
    ):
        print(event)

asyncio.run(test())

Step 5: Deploy

Either via the ADK CLI:

adk deploy agent_engine \
  --project=$GOOGLE_CLOUD_PROJECT \
  --region=$GOOGLE_CLOUD_LOCATION \
  --staging_bucket=$STAGING_BUCKET \
  --trace_to_cloud \
  --display_name="cx-agent" \
  --description="Customer experience agent"

...or via the Python SDK directly:

import vertexai
from vertexai import agent_engines

client = vertexai.Client(project="YOUR_PROJECT_ID", location="us-central1")

remote_agent = client.agent_engines.create(
    agent=app,
    config={
        "requirements": ["google-cloud-aiplatform[agent_engines,adk]"],
        "staging_bucket": "gs://YOUR_STAGING_BUCKET",
    },
)
print(remote_agent.api_resource.name)
# projects/PROJECT_ID/locations/LOCATION/reasoningEngines/RESOURCE_ID

Deployment takes a few minutes. Save the printed resource name — you'll need it for every future call.

Step 6: Grant your service accounts access

Any identity that will call the agent (including the Cloud Run bridge you'll build next) needs the Vertex AI User role on the project:

gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
  --member="serviceAccount:YOUR_CLOUD_RUN_SA@YOUR_PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"

Step 7: Verify in the console

Go to Vertex AI > Agent Engine in the Google Cloud console. Your agent should be listed, and you can use the built-in Playground there to chat with it directly — useful for testing without touching your app.

Part 2 — Build the Cloud Run Bridge (Okta + Custom Domain)

This small service does three things: validates the Okta JWT from your React app, calls Agent Engine using its own Google Cloud identity, and streams the response back.

Step 1: Minimal backend (Python / FastAPI example)

import os
from fastapi import FastAPI, Request, HTTPException
from fastapi.responses import StreamingResponse
import vertexai
from vertexai import agent_engines
from jose import jwt
import httpx

app = FastAPI()

OKTA_ISSUER = "https://YOUR_OKTA_DOMAIN/oauth2/default"
OKTA_JWKS_URI = f"{OKTA_ISSUER}/v1/keys"
OKTA_AUDIENCE = "YOUR_OKTA_CLIENT_ID"

REASONING_ENGINE = "projects/PROJECT_ID/locations/LOCATION/reasoningEngines/RESOURCE_ID"

client = vertexai.Client(project=os.environ["GOOGLE_CLOUD_PROJECT"], location="us-central1")
remote_agent = client.agent_engines.get(REASONING_ENGINE)

async def verify_okta_token(auth_header: str) -> dict:
    if not auth_header or not auth_header.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Missing token")
    token = auth_header.split(" ")[1]
    async with httpx.AsyncClient() as http:
        jwks = (await http.get(OKTA_JWKS_URI)).json()
    try:
        claims = jwt.decode(token, jwks, audience=OKTA_AUDIENCE, issuer=OKTA_ISSUER)
        return claims
    except Exception:
        raise HTTPException(status_code=401, detail="Invalid token")

@app.post("/agent/query")
async def query_agent(request: Request):
    claims = await verify_okta_token(request.headers.get("authorization"))
    body = await request.json()

    async def event_stream():
        async for event in remote_agent.async_stream_query(
            user_id=claims["sub"],
            session_id=body.get("sessionId"),
            message=body["message"],
        ):
            yield f"data: {event}\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")

Step 2: Deploy to Cloud Run

gcloud run deploy cx-agent-bridge \
  --source . \
  --region=us-central1 \
  --service-account=YOUR_CLOUD_RUN_SA@YOUR_PROJECT_ID.iam.gserviceaccount.com \
  --no-allow-unauthenticated
--no-allow-unauthenticated means Cloud Run itself also requires a Google identity to invoke it. If your React app calls it directly over the internet, you'll usually front it with a load balancer (next step) and let your bridge's own Okta check be the access control, or use Cloud Run's IAM invoker binding if only internal services call it.

Step 3: Custom domain + managed SSL

gcloud compute addresses create agent-ip --global

gcloud compute ssl-certificates create agent-cert \
  --domains=agent.yourcompany.com \
  --global

# Point a serverless NEG + load balancer at your Cloud Run service,
# then attach agent-cert and the reserved IP to the HTTPS proxy.

Add an A record for agent.yourcompany.com in Cloud DNS pointing at the reserved IP.

Part 3 — Connect Your React App

Step 1: Call the bridge with the Okta token you already have

const token = await oktaAuth.getAccessToken();

const res = await fetch("https://agent.yourcompany.com/agent/query", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    sessionId: currentSessionId,
    message: userInput,
  }),
});

const reader = res.body.getReader();
const decoder = new TextDecoder();
let result = "";
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  result += decoder.decode(value);
  // parse SSE "data: ..." lines and update chat UI incrementally
}

Step 2: Chat UI components

Rather than building message bubbles, streaming indicators, etc. from scratch, drop in a React chat library (e.g. CopilotKit or assistant-ui) and point its data source at your /agent/query endpoint instead of writing custom rendering logic.

Step 3: Session IDs

Generate a sessionId client-side when a conversation starts (a UUID is fine) and keep sending the same one with every message. Agent Engine's session service keeps the conversation context server-side, so you never need to resend prior messages.
