01 — Overview

The cluster architecture

Kubernetes organises machines into a cluster with a Control Plane that manages everything and Worker Nodes that actually run your workloads.

Control plane
API Server
Central gateway for all requests. Every operation in the cluster passes through here.
etcd
Distributed key-value store. Holds the entire cluster state.
Scheduler
Watches for unscheduled pods and assigns them to the best available node.
Controller Manager
Runs reconciliation loops. Drives actual state toward desired state continuously.
Worker nodes
kubelet
Node agent. Receives pod specs from the API server and ensures containers are running.
kube-proxy
Maintains iptables / IPVS rules for service networking on every node.
Container runtime
Pulls images and runs containers. Usually containerd or CRI-O.
Pods
Smallest deployable unit. One or more containers sharing network and storage.

02 — Control plane

The API server

Every single operation in a Kubernetes cluster — from kubectl apply to a controller updating pod status — goes through the API server. It's stateless, horizontally scalable, and the sole writer to etcd.

Every request passes through four gates before it takes effect:

Authentication
→
Authorization
→
Admission control
→
Write to etcd
01

Authentication

Verifies who is making the request. Supports certificates, bearer tokens, OIDC, and service account tokens.

02

Authorization (RBAC)

Checks whether the authenticated identity is allowed to perform the requested action on the requested resource.

03

Admission control

Plugins that can mutate or reject requests before they're persisted. Enforces resource limits, pod security, and custom webhooks.

04

Watch & notify

Components like the scheduler and controllers subscribe to changes via long-lived watches — the event-driven backbone of Kubernetes.

Production tip Run 3 API server replicas behind a load balancer. Since all state lives in etcd, each replica is fully stateless and can handle any request independently.
03 — Control plane

etcd — cluster memory

A distributed key-value store built on the Raft consensus algorithm. Every Kubernetes object — every pod spec, every secret, every node status — lives here. The API server is the only component that reads from or writes to etcd directly.

Data is stored under hierarchical keys:

# Every resource has a predictable path
/registry/pods/default/my-pod
/registry/deployments/production/web-app
/registry/secrets/kube-system/bootstrap-token
/registry/services/specs/default/my-service

Critical Always run an odd number of etcd nodes — even numbers don't improve fault tolerance and can cause split-brain. Snapshot etcd regularly with etcdctl snapshot save and enable encryption at rest.
04 — Control plane

The scheduler

Watches for pods with no assigned node and picks the best one using a two-phase process: first eliminate nodes that can't run the pod, then score the rest and pick the winner.

Phase 1 — Filter
Resource fit
Enough CPU and memory available?
Taints & tolerations
Does the pod tolerate the node's taints?
Node affinity
Does the node match required labels?
Port conflicts
Are the required host ports free?
Phase 2 — Score (0–100)
Least loaded
Prefer nodes with more free resources.
Spreading
Balance replicas across zones and nodes.
Soft affinity
Prefer nodes matching optional rules.
Image locality
Prefer nodes that already have the image.
Remember The scheduler only places pods — it never runs them. Once it writes the node binding to the API server, the kubelet on that node takes over. The scheduler also never reschedules running pods; that requires the optional Descheduler component.
05 — Control plane

Controller manager

A single binary housing dozens of control loops — each watching for drift between desired and actual state, and taking quiet corrective action 24/7.

The reconciliation loop Every controller does the same thing: observe actual state → compare to desired state → if different, take action. This is Kubernetes' self-healing superpower.
D

Deployment

Creates and manages ReplicaSets. Orchestrates rolling updates by scaling new and old sets in opposite directions.

RS

ReplicaSet

Keeps the right number of pod replicas running. Immediately creates replacements when pods crash.

N

Node

Monitors node health. Marks nodes NotReady after ~40s of silence and evicts pods after ~5 minutes.

EP

Endpoints

Keeps the Endpoints object for each Service updated with currently healthy pod IPs.

J

Job

Ensures batch pods run to a specified number of successful completions, retrying on failure.

SA

ServiceAccount

Auto-creates default service account tokens in every new namespace so pods can authenticate.

08 — Networking

Cloud load balancing

Kubernetes integrates with your cloud provider to route external traffic to pods. There are four Service types and an Ingress layer for HTTP — each solving a different problem.

CI

ClusterIP

Internal only. Every service gets one automatically. Use this for all service-to-service communication inside the cluster.

NP

NodePort

Opens a port (30000–32767) on every node. Simple but requires an external load balancer in front for production use.

LB

LoadBalancer

Provisions an actual cloud load balancer with a public IP. One LB per service — simple but costly at scale.

IN

Ingress

One load balancer for all HTTP services. Routes by host and path. Handles TLS termination. Needs an Ingress controller (NGINX, Traefik, etc.).

ScenarioUse
Internal service-to-serviceClusterIP
Multiple HTTP/HTTPS servicesIngress + one LoadBalancer
Single TCP/UDP service exposed externallyLoadBalancer
Local dev / bare metalNodePort
External managed databaseExternalName
Readiness probes Always configure readiness probes. Without them, Kubernetes sends traffic to pods that are still starting up or stuck — causing real user errors. A pod only enters the load balancing rotation once its readiness probe passes.

10 — Deployment models

Ingress & service routing

Traffic inside a Kubernetes cluster is routed through a layered system: Services provide stable internal endpoints, Ingress rules control external HTTP entry, and kube-proxy stitches it all together with iptables rules on every node.

Request routing — from external client to pod

CI

ClusterIP

Every Service gets a stable virtual IP inside the cluster. DNS resolves svc-name.namespace.svc.cluster.local to this IP. kube-proxy then maps it to a live pod IP via iptables.

IN

Ingress rules

Match on host and path. One Ingress object can route dozens of paths to different backend services, all through a single load balancer IP.

EP

Endpoints

The Endpoints controller watches pods and keeps the list of healthy pod IPs for each Service updated in real time. kube-proxy reads this list to build iptables rules.

RP

Readiness probes

A pod is only added to the endpoint list — and starts receiving traffic — once its readiness probe returns HTTP 200. Crash-looping or slow-starting pods are automatically excluded.

The DNS name pattern every service follows:

# Full DNS name (works from any namespace)
orders-svc.orders.svc.cluster.local
#  │          │      │        │
#  service    ns     fixed    cluster domain

# Short form (within the same namespace)
orders-svc

# Cross-namespace (from inventory calling orders)
orders-svc.orders
Ingress vs LoadBalancer service A LoadBalancer Service provisions one cloud load balancer per service — expensive at scale. An Ingress uses one load balancer for all HTTP services, routing by host and path. On AKS with AGIC: one Application Gateway handles every domain namespace through a single public IP.
11 — Deployment models

Blue-green deployments

Run two identical production environments — blue (current) and green (new version). Switch all traffic from blue to green in a single atomic step. Instant rollback is flipping that switch back.

Blue-green — before, switch, and rollback

Why blue-green
Zero downtime
Traffic switches in milliseconds. No pod restarts, no rolling delays.
Instant rollback
Blue stays running. One patch command reverts to the previous version in under a second.
Pre-flight testing
Green runs smoke tests and synthetic checks before a single real user hits it.
Clean separation
Blue and green never share pods. No partial states mid-deployment.
Tradeoffs & when to use
Double the resources
Both environments run simultaneously — 2× pod count during the switchover window.
DB schema changes are hard
If v1.5 requires a breaking schema migration, blue (v1.4) may not be able to roll back safely.
Best for
High-traffic services where even a 30-second rolling window is unacceptable. Orders, payments, checkout flows.
Not needed for
Background workers, analytics pipelines, low-traffic internal services — rolling updates are fine.

Deployment YAML pattern — label-based switching:

# Two Deployments, same namespace, different labels
apiVersion: apps/v1
kind: Deployment
metadata:
  name: orders-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: orders
      version: green   # ← key differentiator
  template:
    metadata:
      labels:
        app: orders
        version: green
    spec:
      containers:
      - name: orders
        image: acr.io/orders:v1.5
---
# Service selector controls which version receives traffic
apiVersion: v1
kind: Service
metadata:
  name: orders-svc
spec:
  selector:
    app: orders
    version: blue    # ← patch this to "green" to switch
12 — Deployment models

Production rollout strategies

Kubernetes supports multiple rollout strategies, each making a different tradeoff between risk, speed, resource cost, and observability. The right choice depends on the service's criticality and traffic volume.

Strategy comparison — traffic and pod state during rollout

RU

Rolling update

Kubernetes default. Replaces pods one by one — new pod must pass readiness probe before the next old pod is terminated. Zero downtime, minimal resource overhead. Best for most services.

RC

Recreate

Terminates all v1 pods, then starts all v2 pods. Brief downtime but guarantees no two versions run simultaneously — useful when v1 and v2 are incompatible (e.g. shared singleton resource).

BG

Blue-green

Two full environments. Instant atomic switch via Service selector patch. Instant rollback. Best for payment flows, order processing — any service where a 30-second rolling window is unacceptable.

CA

Canary

Route a small percentage (1–5%) of real traffic to v2 while v1 serves the rest. Monitor error rates and latency. Gradually shift weight. Auto-rollback if metrics breach thresholds. Best for high-risk changes.

Rolling update — key Deployment parameters:

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # never reduce capacity below desired
      maxSurge: 1          # allow 1 extra pod above desired during rollout
  minReadySeconds: 10    # pod must stay ready for 10s before counting as healthy
  progressDeadlineSeconds: 300  # fail rollout if not complete in 5 min

Canary with AGIC / Istio traffic weights:

# Istio VirtualService — split 95% stable / 5% canary
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
spec:
  http:
  - route:
    - destination:
        host: orders-svc
        subset: stable
      weight: 95
    - destination:
        host: orders-svc
        subset: canary
      weight: 5    # increase gradually as confidence grows

Warehouse · ProcurementRolling updateInternal operational tools — brief pod churn acceptable

On AKS with Istio The Istio add-on on AKS makes canary deployments a first-class operation. Combined with KEDA autoscaling and Azure Monitor alerting, you can build a fully automated progressive delivery pipeline: deploy canary → watch error rate → auto-promote or auto-rollback based on thresholds — no manual intervention needed.
