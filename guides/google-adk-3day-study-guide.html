Day 1 — Foundations

Understand the mental model, get an agent running locally in under an hour, then master tools.

What is ADK, and why use it?

ADK is Google's open-source, code-first Python (also TypeScript/Go/Java/Kotlin) framework for building, evaluating, and deploying AI agents. You write plain Python — no drag-and-drop builder, no YAML orchestration files — and ADK supplies the runtime: it manages conversation sessions, tool invocation, multi-agent delegation, memory, and one-command deployment. It's optimized for Gemini but works with 100+ models through LiteLLM adapters, so you aren't locked in.

The framework reached 2.0 General Availability in June 2026, adding graph-based workflows and collaborative multi-agent support on top of the original agent/tool/session model.

FrameworkPhilosophyBest for
Google ADKOpinionated, code-first, native multi-agentGemini/GCP-first teams wanting fast, structured multi-agent apps
LangChain / LangGraphGeneral-purpose, model-agnosticMaximum flexibility, many third-party integrations
CrewAIRole-based agent collaborationSimple crew-style task delegation
Why it matters for you: if you've already touched Bedrock AgentCore, Strands Agents SDK, or MCP, the concepts here will feel familiar — Agent ≈ your LLM-backed worker, Tool ≈ function/action, Session ≈ conversation + state, Runner ≈ the orchestration loop. ADK's differentiator is how cleanly multi-agent delegation and workflow composition are built into the core, rather than bolted on.

Installation & environment setup

Requirements: Python 3.10+ (3.11/3.12 recommended for performance), a free Gemini API key from Google AI Studio.

# 1. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate      # macOS/Linux
# venv\Scripts\activate       # Windows

# 2. Install ADK
pip install google-adk

# 3. Scaffold a new agent project
adk create my_agent

# 4. Add your Gemini API key to my_agent/.env
echo "GOOGLE_API_KEY=your_key_here" > my_agent/.env

adk create scaffolds this structure:

my_agent/
├── agent.py        # root_agent definition — the only required piece
├── .env             # API key / project config
└── __init__.py

Core architecture: Agent, Runner, Session

Three concepts do almost all the work in ADK:

  • Agent — defines instructions, model, and tools. The unit of "intelligence".
  • Runner — executes an agent, manages the loop of: user input → LLM call → tool calls → response.
  • Session — the conversation thread: event history + a state dict that persists data across turns.
Fig 1 — The Runner drives the loop: user message → Session → Agent/LLM → optional Tool call → back into Session state → response to user.

Your first agent

Edit agent.py:

from google.adk import Agent

root_agent = Agent(
    name="greeting_agent",
    model="gemini-flash-latest",
    instruction="You are a helpful assistant. Greet the user warmly and ask how you can help.",
)

Run it two ways:

# Interactive terminal chat
adk run my_agent

# Local web UI with a chat interface + debug/event graph
adk web
Note: adk web is a development/debugging tool, not meant for production traffic. Run it from the parent directory that contains your agent folder.

Function tools & built-in tools

Tools are what turn a chatbot into an agent. ADK auto-generates a tool schema from your Python function's type hints and docstring — this is why Python 3.10+ type hints matter.

from google.adk.agents.llm_agent import Agent

def get_current_time(city: str) -> dict:
    """Returns the current time in a specified city.

    Args:
        city: Name of the city, e.g. "Portland".
    """
    # In production, call a real timezone API here
    return {"status": "success", "city": city, "time": "10:30 AM"}

root_agent = Agent(
    model="gemini-flash-latest",
    name="root_agent",
    description="Tells the current time in a specified city.",
    instruction="You are a helpful assistant. Use 'get_current_time' when asked about time.",
    tools=[get_current_time],
)

Built-in tools ship with the framework — e.g. web search:

from google.adk import Agent
from google.adk.tools import google_search

agent = Agent(
    name="researcher",
    model="gemini-flash-latest",
    instruction="You help users research topics thoroughly, citing sources.",
    tools=[google_search],
)
Day 1 exercise: Build a small "fact-checker" agent: give it the google_search tool, a low temperature, and an instruction that makes it skeptical and requires it to verify claims before answering. Run it with adk web and try a few true/false claims.

Day 2 — Multi-Agent Orchestration

Move from one agent to teams of specialized agents that delegate, chain, and collaborate.

Sub-agents & delegation

Instead of one giant prompt trying to do everything, ADK encourages composing small, specialized agents. A parent agent can hold a list of sub_agents and either delegate dynamically (LLM decides which sub-agent to call) or run them in a fixed structure using workflow agents.

Why split agents up: simpler design, higher reliability per task, and easier maintenance — you can fix one specialized agent without breaking the others.

Sequential, Parallel & Loop agents

These are deterministic workflow agents — their execution order is controlled by code, not by an LLM decision.

Fig 2 — The three deterministic workflow primitives: strict order (Sequential), fan-out/fan-in (Parallel), and iterative refinement (Loop).

SequentialAgent — fixed pipeline

from google.adk.agents import SequentialAgent, LlmAgent

code_writer = LlmAgent(
    name="CodeWriterAgent",
    instruction="Write Python code for: {specification}",
    output_key="generated_code",   # saved to session.state["generated_code"]
)
code_reviewer = LlmAgent(
    name="CodeReviewerAgent",
    instruction="Review this code for bugs and style:\n{generated_code}",
    output_key="review_comments",
)
code_refactorer = LlmAgent(
    name="CodeRefactorerAgent",
    instruction="Refactor {generated_code} using {review_comments}.",
    output_key="refactored_code",
)

root_agent = SequentialAgent(
    name="CodePipeline",
    sub_agents=[code_writer, code_reviewer, code_refactorer],
)

ParallelAgent — fan-out/fan-in

from google.adk.agents import ParallelAgent, SequentialAgent

fan_out = ParallelAgent(
    name="PreproductionTeam",
    sub_agents=[cast_research_agent, box_office_research_agent],
)
# A downstream agent reads both output_key values from shared state
root_agent = SequentialAgent(
    name="MoviePitchPipeline",
    sub_agents=[fan_out, file_writer_agent],
)

LoopAgent — iterative refinement

from google.adk.agents import LoopAgent, LlmAgent

generator = LlmAgent(
    name="Generator",
    instruction="Generate a SQL query. If given {feedback}, fix the errors and try again.",
    output_key="draft",
)
critic = LlmAgent(
    name="Critic",
    instruction="Check if {draft} is valid SQL. If correct, output 'PASS'. Else, explain the error.",
    output_key="feedback",
)

root_agent = LoopAgent(
    name="ValidationLoop",
    sub_agents=[generator, critic],
    max_iterations=5,   # hard stop; agents can also exit early via escalate=True
)

Session state & data flow

Agents pass data to each other through session.state — a shared dict. The cleanest way to write to it is the output_key parameter shown above; the cleanest way to read it is the {key} templating syntax inside an instruction string.

PrefixScope
(none)Current session only
temp:Current invocation only (shared across sub-agents in one turn)
user:Persists across all sessions for that user
app:Persists across all sessions, all users
Important: always mutate state via context.state[...] inside a callback/tool (the CallbackContext/ToolContext), never directly on a Session object fetched from the SessionService — the latter bypasses event tracking and can silently lose data.

Callbacks & guardrails

Callbacks hook into six checkpoints, grouped in three pairs: agent lifecycle (before_agent_callback / after_agent_callback — available on every agent type), model interaction (before_model_callback / after_model_callback — LlmAgent only), and tool execution (before_tool_callback / after_tool_callback — LlmAgent only). Every before_* callback can act as a gatekeeper: return None to proceed normally, or return an object to short-circuit the step.

def block_forbidden_content(callback_context, llm_request):
    last_text = llm_request.contents[-1].parts[0].text
    if "password" in last_text.lower():
        return LlmResponse(
            content=types.Content(role="model",
                                   parts=[types.Part(text="I can't help with that request.")])
        )
    return None  # proceed to the LLM as normal

guarded_agent = LlmAgent(
    name="SupportAgent",
    model="gemini-flash-latest",
    instruction="You are a customer support agent.",
    before_model_callback=block_forbidden_content,
)

Custom agents

When Sequential/Parallel/Loop aren't enough (e.g. conditional branching), subclass BaseAgent and implement _run_async_impl yourself — you get full control while still being able to nest the built-in workflow agents inside your custom one.

class StoryPipeline(BaseAgent):
    def __init__(self, name, story_generator, critic, reviser, grammar_check, tone_check):
        loop_agent = LoopAgent(name="CriticReviserLoop",
                                sub_agents=[critic, reviser], max_iterations=2)
        sequential_agent = SequentialAgent(name="PostProcessing",
                                            sub_agents=[grammar_check, tone_check])
        super().__init__(name=name, sub_agents=[story_generator, loop_agent, sequential_agent])

    async def _run_async_impl(self, ctx):
        async for event in self.story_generator.run_async(ctx):
            yield event
        async for event in self.loop_agent.run_async(ctx):
            yield event
        # custom conditional logic can branch here based on ctx.session.state
        async for event in self.sequential_agent.run_async(ctx):
            yield event
Day 2 exercise: Build a "writer's room": a story_generator LlmAgent feeding into a LoopAgent of critic + reviser (2–3 iterations), followed by a SequentialAgent doing grammar and tone checks. Use the ADK web UI's event graph to visualize the run.

Day 2 checklist

  • Built a SequentialAgent pipeline with output_key chaining
  • Built a ParallelAgent fan-out with a gather step
  • Built a LoopAgent with an exit condition
  • Used a before_model_callback as a guardrail
  • Understand temp:, user:, app: state prefixes

Day 3 — Production & Advanced Topics

Memory, evaluation, deployment, and security — what separates a demo from a shippable agent.

Long-term memory

Session state is short-term — it disappears once a session ends (unless you use user:/app: prefixes for simple values). For true long-term memory across sessions, ADK provides VertexAIMemoryBankService, which uses Gemini to extract and store key facts from a session, then retrieves them later via similarity search.

from google.adk.memory import VertexAiMemoryBankService

memory_service = VertexAiMemoryBankService(
    project="my-gcp-project", location="us-central1", agent_engine_id="my-engine-id"
)

async def save_to_memory(callback_context):
    await memory_service.add_session_to_memory(callback_context.session)

tutor_agent = LlmAgent(
    name="PythonTutor",
    model="gemini-flash-latest",
    instruction="Personalize this lesson using memories: {memory_results}",
    after_agent_callback=save_to_memory,   # persist facts once the session ends
)

Evaluation framework

ADK ships a built-in evaluation harness so you can regression-test agent behavior before every deploy — comparing an agent's tool calls and final responses against an expected trajectory.

# evalset.test.json — defines expected tool-call trajectory + expected response
adk eval my_agent eval_sets/greeting_eval_set.json
Practice: write at least one eval case per tool your agent has, plus 2–3 "adversarial" cases probing off-topic or unsafe requests.

Deployment: three tiers

Fig 3 — All three inherit managed auth, Cloud Trace observability, and enterprise security without agent code changes.
# One-command deploy to Cloud Run
adk deploy cloud_run --project=my-gcp-project --region=us-central1 my_agent

# Deploy to the fully managed Agent Platform
adk deploy agent_engine --project=my-gcp-project my_agent
Given your AWS background: think of Agent Platform ≈ a managed agent runtime similar in spirit to Bedrock AgentCore Runtime, Cloud Run ≈ your familiar serverless-container deploy path, and GKE ≈ EKS-equivalent for teams needing fine-grained infra control.

Security & prompt-injection defense

Treat every LLM output and every tool result as untrusted input. Practical layers in ADK:

  • before_model_callback — filter/rewrite prompts before they hit the model (block sensitive keywords, PII).
  • before_tool_callback — enforce authorization (e.g. only admin users can call a cancel_reservation tool).
  • after_model_callback — sanitize outputs (strip leaked emails, secrets) before they reach the user.
  • Least-privilege tools — scope each tool narrowly; don't give a single agent broad filesystem/database access if it only needs to read one table.
  • Human-in-the-loop — ADK's Task API and tool confirmation flows let you pause for human approval before high-risk actions execute.

MCP tools & multi-model support

ADK agents can consume Model Context Protocol (MCP) servers as tools directly — useful if you've already built or used MCP servers elsewhere. Model choice is equally flexible: swap between Gemini, Gemma, Claude (via Vertex AI), Ollama, vLLM, or any LiteLLM-supported provider without rewriting agent logic.

from google.adk.tools.mcp_tool import MCPToolset

toolset = MCPToolset(connection_params={"url": "https://my-mcp-server.example.com/mcp"})

agent = Agent(
    name="mcp_powered_agent",
    model="gemini-flash-latest",
    instruction="Use the connected MCP tools to fulfill user requests.",
    tools=[toolset],
)
Day 3 capstone: combine what you've learned — a SequentialAgent retrieval-then-answer pipeline (retriever tool → generator LlmAgent), guarded by a before_model_callback, with session summaries persisted to VertexAiMemoryBankService, deployed to Cloud Run. This maps directly onto a RAG-style "tribal knowledge" assistant pattern if you want to prototype something like that.

Day 3 checklist

  • Wired up a memory service and reviewed retrieved memories in a prompt
  • Wrote at least one ADK eval set
  • Deployed an agent to Cloud Run (or reviewed the Agent Platform CLI flow)
  • Added at least one guardrail callback
  • Understand how MCP tools and LiteLLM model swapping fit in

Bonus — Advanced Production Topics

Once Days 1–3 feel solid, these are the topics that separate a working prototype from a system you'd trust in production.

Agent2Agent (A2A) protocol

Everything in Day 2 assumed sub-agents live in the same process. Real systems often need agents built by different teams, in different repos, even different frameworks, to talk to each other over the network. That's what the open Agent2Agent (A2A) protocol is for: it standardizes how one agent discovers, calls, and exchanges results with another as a Task — a unit of work with a lifecycle (submitted → working → completed/failed) plus artifacts holding the results.

Fig 4 — A2A lets independently deployed agents (possibly on different frameworks) discover and call each other via a standard "agent card" and Task lifecycle, hosted on a runtime like Agent Platform or Cloud Run.

Expose an ADK agent as an A2A server

from google.adk.a2a import to_a2a

reservation_agent = LlmAgent(
    name="ReservationAgent",
    model="gemini-flash-latest",
    instruction="Manage restaurant reservations: create, check, cancel.",
    tools=[create_reservation, check_reservation, cancel_reservation],
)

# Wraps the agent as an A2A-compliant FastAPI app, serving an agent card for discovery
a2a_app = to_a2a(reservation_agent, port=8001)

Consume a remote A2A agent as a sub-agent

from google.adk.a2a import RemoteA2aAgent

remote_reservation_agent = RemoteA2aAgent(
    name="ReservationAgent",
    agent_card="https://reservation-agent-xyz.run.app/.well-known/agent-card.json",
)

concierge_agent = LlmAgent(
    name="ConciergeAgent",
    model="gemini-flash-latest",
    instruction="Route menu questions locally; delegate booking requests to ReservationAgent.",
    sub_agents=[remote_reservation_agent],
)
When to reach for A2A: independent deploy/scale lifecycles per agent, cross-team ownership, or mixing frameworks (an ADK orchestrator calling a LangGraph or CrewAI agent). Within a single team/repo, in-process sub-agents (Day 2) are simpler and faster.

Observability & tracing

ADK implements the OpenTelemetry (OTel) semantic conventions for GenAI and emits standard OTLP output — so traces plug into Cloud Trace, Jaeger, Grafana Tempo, or Datadog without custom glue. Each agent run is a root span; LLM calls and tool executions become nested child spans, giving you a waterfall view of exactly where time and tokens went.

# Enable Cloud Trace when running locally
adk web --otel_to_cloud

# Enable Cloud Trace on deploy
adk deploy cloud_run --trace_to_cloud --project=my-gcp-project my_agent

For deeper, queryable analytics (per-tool latency, token cost by agent, failure-reason breakdowns), pipe traces into BigQuery via the BigQuery Agent Analytics plugin — it also tracks tool provenance (was a call local, MCP, sub-agent, or A2A?) and events like agent transfers, state checkpoints, and paused/resumed long-running tools.

Production habit: set OTEL_SEMCONV_STABILITY_OPT_IN correctly and scrub PII from span attributes before shipping — traces routinely include full prompts and tool arguments, which can leak sensitive data into your observability backend if you're not deliberate about it.

Artifacts & binary data

Session state is for small text/JSON values. For files — PDFs, images, generated reports — ADK has a separate ArtifactService that stores binary blobs by key and version, referenced from state without bloating the LLM context. Artifacts are lazy-loaded: the agent only pulls the actual bytes in when a tool needs them.

from google.adk.tools import ToolContext
import google.genai.types as types

async def save_report(tool_context: ToolContext, report_bytes: bytes) -> dict:
    """Saves a generated PDF report as a versioned artifact."""
    version = await tool_context.save_artifact(
        filename="quarterly_report.pdf",
        artifact=types.Part(inline_data=types.Blob(mime_type="application/pdf", data=report_bytes)),
    )
    return {"status": "success", "version": version}

Plugin system for cross-cutting concerns

Callbacks (Day 2) are attached per-agent. When you need the same behavior — rate limiting, cost tracking, PII redaction, caching — applied consistently across every agent in a system, a Plugin is the right tool: it hooks into the same lifecycle points globally, at the Runner level, instead of being wired into each agent individually.

from google.adk.plugins import BasePlugin

class RateLimitPlugin(BasePlugin):
    async def before_model_callback(self, *, callback_context, llm_request):
        if self._over_quota(callback_context.session.user_id):
            return LlmResponse(content=types.Content(
                role="model", parts=[types.Part(text="Rate limit exceeded, try again shortly.")]))
        return None

runner = Runner(agent=root_agent, app_name="prod_app",
                 session_service=session_service, plugins=[RateLimitPlugin()])

Context engineering at scale

Long-running agents accumulate huge event histories. ADK treats context like a scarce resource by default: it filters irrelevant events, summarizes older conversational turns (event compaction), lazy-loads artifacts instead of inlining them, and tracks token usage per invocation — but for complex, long-horizon tasks you'll want to tune this explicitly.

TechniquePurpose
Event compactionSummarize old turns instead of replaying full history every call
include_contents on LlmAgentControl exactly which prior events are sent to the model
Lazy artifact loadingKeep large files out of the token stream until a tool actually needs them
State checkpointsSnapshot state at key points so long-running tasks can resume after a crash
Rule of thumb: if an agent's latency or cost grows noticeably over a long conversation, check event compaction settings and include_contents before reaching for a bigger context-window model — it's usually cheaper and faster to trim what you send.

Structured input/output schemas

For agents feeding other systems (APIs, databases, other agents) rather than a human, force the model into a strict schema instead of parsing free text. ADK supports Pydantic-based input_schema / output_schema on LlmAgent.

from pydantic import BaseModel

class TriageResult(BaseModel):
    category: str       # "billing" | "technical" | "sales"
    priority: int       # 1-5
    summary: str

triage_agent = LlmAgent(
    name="TriageAgent",
    model="gemini-flash-latest",
    instruction="Classify the incoming support ticket.",
    output_schema=TriageResult,   # response is now guaranteed valid JSON matching this shape
    output_key="triage_result",
)
Gotcha: an LlmAgent with output_schema set cannot also call tools in that same turn — the model is constrained to only emit the schema. Do tool-using work in an upstream agent, then hand off to a schema-constrained agent purely for formatting/classification.

Production readiness checklist

  • Every tool has narrow, least-privilege scope and explicit auth checks
  • At least one guardrail callback (input filtering) and one output-sanitizing callback
  • Eval sets cover both happy-path and adversarial/off-topic cases
  • Tracing enabled (--otel_to_cloud / --trace_to_cloud) with PII scrubbing on spans
  • Long-running or multi-turn tasks have state checkpoints for crash recovery
  • Rate limiting / cost controls applied globally via a Plugin, not per-agent
  • Deployment target matches scale needs (Agent Platform → Cloud Run → GKE, in order of increasing control/complexity)
  • If agents span teams or frameworks, boundaries are exposed via A2A rather than tightly coupled imports
Bonus capstone: take your Day 3 RAG-style capstone and (1) wrap it as an A2A server so another team's orchestrator could call it, (2) add OpenTelemetry tracing and inspect a run in Cloud Trace, and (3) add a RateLimitPlugin. That combination — A2A boundary, tracing, and a global plugin — is close to what a real internal-tools team would ship.

Full Agentic Project Lifecycle & GCP Deployment

How a real ADK project moves from a blank folder to a monitored, versioned service running on Google Cloud — and back again for the next iteration.

The end-to-end agentic project lifecycle

Everything you've learned in Days 1–4 slots into a repeating cycle. Treat it as a loop, not a one-way pipeline — production agents get re-tuned constantly as you observe real traffic.

Fig 5 — The lifecycle is a loop: production observability (Day 4) is the input to the next Design phase, not an afterthought bolted on at the end.
StageWhat happensKey ADK / GCP tooling
DesignDefine agent roles, decide single vs multi-agent, sketch the workflow (Sequential/Parallel/Loop or A2A boundaries)Whiteboard, ADK Studio graph view
BuildWrite agent.py, tools, callbacks; iterate fast locallyadk create, adk web, adk run

Monitor & IterateWatch traces, error rates, and cost; feed findings back into DesignCloud Trace, Cloud Monitoring, BigQuery Agent Analytics

CI/CD pipeline (GitHub Actions + Terraform)

The standard pattern: a pull request triggers evaluation and unit tests; on merge to main, the agent is built, gated on eval results, and deployed. Infrastructure (the Cloud Run service, IAM bindings, networking) is defined in Terraform so environments stay reproducible.

# .github/workflows/deploy-agent.yml
name: deploy-agent
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test-and-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r requirements.txt
      - run: pytest tests/                              # unit tests for tools/callbacks
      - run: adk eval my_agent eval_sets/*.json           # regression-test agent behavior

  deploy-dev:
    needs: test-and-eval
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.DEPLOY_SA }}
      - run: |
          adk deploy cloud_run \
            --project=$PROJECT_ID --region=$REGION \
            --service_name=my-agent-dev my_agent

  deploy-prod:
    needs: test-and-eval
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ secrets.WIF_PROVIDER }}
          service_account: ${{ secrets.DEPLOY_SA }}
      - run: |
          adk deploy cloud_run \
            --project=$PROJECT_ID --region=$REGION \
            --service_name=my-agent-prod --trace_to_cloud my_agent
Credential-less alternative: instead of granting your CI runner deploy permissions directly, use Developer Connect (GitOps) — Cloud Build watches your repo and pulls code on merge. Deployments become immutably tied to a Git commit hash, which makes "what's running in prod right now" and rollback trivial, at the cost of more upfront setup.

Minimal Terraform for the Cloud Run side, so environments are declarative and reviewable:

# main.tf
resource "google_cloud_run_v2_service" "agent" {
  name     = "my-agent-prod"
  location = var.region

  template {
    containers {

Deployment deep-dive: three real paths to production

You met these three tiers in Day 3 at a glance — here's the actual deployment code for each, plus when to pick which.

1. Vertex AI Agent Engine — fully managed, "Cloud Run for agents"

Best when you want Google to own scaling, versioning, and the built-in Playground UI, and you don't need custom networking or non-Python components.

import vertexai
from vertexai.preview import reasoning_engines
from vertexai import agent_engines
from my_agent.agent import root_agent

vertexai.init(project=PROJECT_ID, location="us-central1", staging_bucket="gs://my-agent-staging")

# Wrap the ADK agent; enable_tracing turns on Cloud Trace automatically
app = reasoning_engines.AdkApp(agent=root_agent, enable_tracing=True)

remote_app = agent_engines.create(
    agent_engine=app,
    requirements=["google-adk", "google-cloud-aiplatform[adk,agent_engines]"],
    extra_packages=["./my_agent"],
)
print(remote_app.resource_name)

Or the one-line CLI equivalent, ideal for CI scripts:

adk deploy agent_engine --project=$PROJECT_ID --region=us-central1 \
  --staging_bucket=gs://my-agent-staging my_agent

2. Cloud Run — containerized, scale-to-zero, most flexible middle ground

Best default choice for most teams: full control over the container, works with any language/framework mixed in, integrates cleanly into existing declarative CI/CD, and costs nothing when idle.

# One-command deploy (ADK builds the container for you)
adk deploy cloud_run --project=$PROJECT_ID --region=us-central1 \
  --service_name=my-agent-prod --trace_to_cloud my_agent

# Equivalent explicit path if you maintain your own Dockerfile
gcloud builds submit --tag us-central1-docker.pkg.dev/$PROJECT_ID/adk-repo/my-agent
gcloud run deploy my-agent-prod \
  --image us-central1-docker.pkg.dev/$PROJECT_ID/adk-repo/my-agent \
  --region us-central1 --allow-unauthenticated=false

3. GKE — full Kubernetes control, for advanced scaling/networking needs

Reach for this when you need custom autoscaling policies, service mesh, GPU node pools, or you're already standardized on Kubernetes for the rest of your platform.

# agent-deployment.yaml
apiVersion: apps/v1

Versioning, traffic splitting & rollback

Cloud Run keeps every deploy as an immutable revision. You control exactly how much traffic each revision gets — this is your canary and rollback mechanism in one.

# Canary: send 10% of traffic to the newly deployed revision, 90% stays on the known-good one
gcloud run services update-traffic my-agent-prod \
  --to-revisions=my-agent-prod-00042=90,my-agent-prod-00043=10

# Full promote once the canary looks healthy
gcloud run services update-traffic my-agent-prod --to-revisions=my-agent-prod-00043=100

# Instant rollback: shift 100% of traffic back to the previous known-good revision
gcloud run services update-traffic my-agent-prod --to-revisions=my-agent-prod-00042=100

Agent Engine handles versioning differently — each agent_engines.create() (or update) call produces a tracked deployment you can inspect via agent_engines.list(), with monitoring and version history visible in the Vertex AI console's Playground.

Monitoring, alerting & cost control

Connect back to the observability work from Day 4: with --trace_to_cloud (Cloud Run) or enable_tracing=True (Agent Engine) turned on, every deploy path feeds Cloud Trace and Cloud Logging automatically. On top of that, set up:

  • Cloud Monitoring alerting policies — p95 latency, 5xx error rate, and tool-failure rate thresholds that page you before users notice.
  • Token / cost dashboards — export via BigQuery Agent Analytics (Day 4) and chart spend per agent, per user, per day; catch runaway loops or prompt regressions early.
  • Budget alerts — a GCP budget alert on the project as a blunt but effective backstop against a misbehaving agent burning API quota overnight.
  • Structured logging — log tool inputs/outputs (with PII scrubbed) so a bad response can be traced back to the exact tool call that produced it.

End-to-end lifecycle checklist

  • Agent design documented: single vs multi-agent, workflow shape, A2A boundaries if any
  • Eval sets exist and run automatically on every pull request
  • Dockerfile / container build is reproducible and pinned (no floating latest base images)
  • CI/CD gates production deploys on passing tests + evals, not just a green build
  • Infrastructure defined in Terraform, not clicked together manually
  • Chosen deployment tier matches actual scale/control needs (don't default to GKE "just in case")
  • Tracing enabled from day one of deployment, not added after the first incident
  • Rollback procedure tested at least once before you need it for real
  • Cost dashboards and budget alerts configured before opening to real traffic
Full lifecycle exercise: take any agent you built on Day 1–4, add one eval set, write a two-job GitHub Actions workflow (test-and-eval → deploy to Cloud Run), deploy it with --trace_to_cloud, then practice a canary rollout and rollback using gcloud run services update-traffic. Doing this once end-to-end will teach you more than reading about it three times.
