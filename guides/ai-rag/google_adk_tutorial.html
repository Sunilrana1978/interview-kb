Google Agent Development Kit (ADK)

Building Enterprise AI Agents

An interactive code-first guide to mastering Google's open-source Agent Development Kit (ADK). Learn how to orchestrate multi-agent and multi-modal systems, manage session states, and build graph-based workflows over a 3-day revision path.

Est. Revision Time: 3 Days (4-5 hrs/day)
Target Framework: google-adk (Code-First)
Primary Language: Python 3.10+

Day 1 Framework Foundations & Getting Started

On Day 1, we cover the purpose of the Google Agent Development Kit (ADK) and explore how to declare basic agents, configure environment models, and execute them locally.

1. Core Concepts & Architecture

Unlike legacy prompt-engineering wrappers, Google ADK is a code-first, model-agnostic framework for designing, debugging, and executing robust agentic applications at enterprise scale. Its core structure models building blocks as modular components:

  • Agent: The core processing unit. An agent is defined by instructions, description, a target LLM model (e.g., Google's Gemini models), and tools it is allowed to call.
  • Workflow: A graph-based execution system. Workflows control the routing of tasks between agents, enabling conditional paths, loops, parallel fan-outs, and human confirmation checkpoints.
  • Runtime: The stateful manager that executes the workflow, preserves intermediate memory, invokes tools, and streams token results.

2. Installation & Directory Setup

Install the official package via pip. You will need a Gemini API Key stored in your environment to authenticate agent prompts:

Terminal 
# Install the core Agent Development Kit
pip install google-adk

# Expose Gemini credential
export GEMINI_API_KEY="your-api-key-from-ai-studio"

Enterprise Practice: ADK projects are structured cleanly. A typical local workspace is laid out like this:

Workspace Layout 
my_adk_project/
├── app.py          # Main application / entrypoint
├── agents/         # Sub-agents definition directory
│   ├── __init__.py # Imports agents to expose them
│   └── weather.py  # Agent specific declarations
└── tools/          # Custom utilities for tool calling

3. Defining Your First Agent

Declare an Agent by passing a model name, instruction directives, and optional tool functions:

agents/weather.py 
from google.adk.agents import Agent

# Create a localized tool function
def get_temperature(city: str) -> dict:
    """Retrieves the current weather temperature for a specified city.

    Args:
        city: City name, e.g. "Chicago"
    """
    # Simulation response
    return {"temperature": "68F", "condition": "Partly Cloudy"}

# Instantiate the agent
weather_agent = Agent(
    name="weather_agent",
    model="gemini-2.5-flash",
    description="Queries and returns weather details.",
    instruction="You are a helpful assistant. Use get_temperature to answer temperature questions.",
    tools=[get_temperature]
)

4. Executing via ADK CLI

Once defined, run and test the agent using the built-in development tools. The ADK CLI compiles the structure and manages local loops:

Terminal 
# 1. Run the agent interactively in the terminal
adk run agents/weather.py

# 2. Launch the local web debugger UI to inspect the agent graph and tools
adk web agents/

Day 2 Custom Tools & Contextual Memory

On Day 2, we dive deeper into tool declarations, injecting stateful session memory into tools, and delegating sub-agent execution.

1. Auto-Binding Python Tools

Google ADK automatically compiles function parameters, docstrings, and typing definitions into JSON schema definitions for the LLM. Write standard Python type annotations to guarantee exact parsing:

tools/math.py 
def compute_compound_interest(principal: float, rate: float, periods: int) -> float:
    """Calculates compound interest on principal sum.

    Args:
        principal: Starting capital balance.
        rate: Decimal annual interest rate, e.g. 0.05.
        periods: Total number of compound interest periods (years).
    """
    return principal * ((1 + rate) ** periods)

2. Stateful Session Tracking

ADK includes a state management service to share context. By injecting ToolContext, a tool can access and update conversation states across execution steps:

tools/shopping.py 
from google.adk.tools import ToolContext

def update_inventory_list(item: str, qty: int, ctx: ToolContext) -> str:
    """Updates inventory counts in the session storage.

    Args:
        item: Item name.
        qty: Item count to add.
        ctx: Context injected by the ADK runtime.
    """
    # Retrieve the state for the current session
    inventory = ctx.get_state("inventory", {})
    
    inventory[item] = inventory.get(item, 0) + qty
    ctx.set_state("inventory", inventory)
    
    return f"Updated {item} to {inventory[item]}. Full Inventory: {inventory}"

3. Spawning & Delegating Sub-agents

For complex business domains, break down monolithic intelligence into specialized sub-agents. The main agent coordinates and delegates questions to dedicated sub-agents when required:

agents/delegation.py 
from google.adk.agents import Agent

# Define specialized sub-agents
db_agent = Agent(
    name="db_agent",
    model="gemini-2.5-flash",
    instruction="Query SQL database tables for client data."
)

summary_agent = Agent(
    name="summary_agent",
    model="gemini-2.5-flash",
    instruction="Summarize data payloads in markdown tables."
)

# The orchestrator uses subagents as tools to coordinate execution
orchestrator = Agent(
    name="orchestrator",
    model="gemini-2.5-pro",
    instruction="Route client requests to db_agent, then send the output to summary_agent.",
    tools=[db_agent, summary_agent]
)

Day 3 Graph Workflows & Custom Agent Structs

On Day 3, we orchestrate agents deterministically using graph-based Workflows, create custom agents using BaseAgent, and overview enterprise deployment.

1. Graph-Based Workflows

For predictable pipelines, ADK provides the Workflow class. A workflow schedules execution paths using a graph composed of nodes and directed edges:

workflow.py 
from google.adk import Agent, Workflow

# Define nodes (agents)
generate_fruit = Agent(
    name="generate_fruit",
    instruction="Provide a single random fruit."
)

analyze_benefit = Agent(
    name="analyze_benefit",
    instruction="State a scientific health benefit about the provided fruit."
)

# Configure the graph execution flow
fruit_workflow = Workflow(
    name="fruit_workflow",
    edges=[
        ("START", generate_fruit),
        (generate_fruit, analyze_benefit),
        (analyze_benefit, "END")
    ]
)

2. Custom Agents with BaseAgent

For advanced pipelines requiring custom logic, branching, or conditional routing, inherit from BaseAgent and override _run_async_impl:

custom_agent.py 
from google.adk.agents import BaseAgent, Agent
from google.adk.types import AgentResponse

class RouterAgent(BaseAgent):
    def __init__(self, coder_agent: Agent, tester_agent: Agent):
        # Pass sub-agents to the parent initializer to manage their lifecycle
        super().__init__(
            name="router_agent",
            sub_agents=[coder_agent, tester_agent]
        )
        self.coder = coder_agent
        self.tester = tester_agent

    async def _run_async_impl(self, user_query: str, ctx) -> AgentResponse:
        # Custom routing orchestration logic
        code_response = await self.coder.run_async(user_query, context=ctx)
        
        # Check outputs programmatically
        if "error" in code_response.text.lower():
            return code_response
            
        test_response = await self.tester.run_async(code_response.text, context=ctx)
        return test_response

3. Lifecycle Callbacks

ADK supports interception triggers like before_agent_callback to set up databases, validate authentication, or write audit traces:

callbacks.py 
def audit_logger(agent_name: str, payload: str):
    print(f"[Audit log] Node '{agent_name}' triggered with query: '{payload}'")

agent = Agent(
    name="audited_agent",
    model="gemini-2.5-flash",
    instruction="Summarize task files.",
    before_agent_callback=audit_logger
)

4. Cloud Deployment

To deploy ADK agents in production, packages are containerized and deployed to **Google Cloud Run**, where the **Agent Runtime** manages scaling, routing, and tool credentials securely.
