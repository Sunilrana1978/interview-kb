Reference · Retrieval-Augmented Generation on Google Cloud

Where should your documents live?

Every RAG stack on Google Cloud is really answering one question: how much of the search engine do you want Google to run for you, and how much do you want to control yourself? This reference walks the two options usually compared first — Gemini API File Search and Vertex AI Search data stores — then places them against the rest of the field, including the option that sits between them.

01 Gemini API — File Search Store

A retrieval tool built directly into the Gemini API. You upload files; Google's infrastructure handles chunking, embedding, indexing, and semantic search behind one tool call.

File Search Store

Fully managed
AutomatedYou control it

File Search operates as a single-shot retrieval mechanism — one API call triggers retrieval, context injection, and answer generation atomically, and it returns grounding metadata showing which chunks were used, even though the internal ranking and reformulation logic stays opaque. You keep ownership of the database, metadata, permissions, and UX; Google owns the search engine and chat layer underneath it.

01Upload file
02Auto-chunk
03Auto-embed
04Index in store
05Query → cited answer

Good fit

  • Fast prototyping — a working RAG demo in hours, not a sprint
  • Agents that need quick, bespoke access to a document set with no infra to run
  • Small teams who'd rather spend time on UX and flows than on chunking strategy
  • Built-in per-chunk citations without extra plumbing

Real constraints

  • No control over chunking strategy, embedding model, or ranking — you inherit Google's defaults
  • Current limits: 10 stores per project, and Google recommends keeping each store under 20GB for retrieval quality; a single query can span at most 5 stores
  • ADK has no built-in wrapper for File Search yet — you write a custom tool class that injects file_search_store_names into the low-level model request
  • You cannot combine the native GoogleSearch tool and File Search in the same request to the same agent — route through a two-agent orchestrator pattern instead
ADK note — because there's no native wrapper, the common pattern is a root agent that delegates to a RAG specialist (holding the custom FileSearch tool) and a separate search specialist (holding GoogleSearch), rather than giving one agent both tools.

Minimal shape (Python)

# conceptual — wraps the low-level request since ADK has no native FileSearch tool yet
file_search_store = client.file_search_stores.create(display_name="support-kb")
client.file_search_stores.upload(store=file_search_store.name, file="handbook.pdf")

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="How do I reset a device?",
    tools=[FileSearchTool(file_search_store_names=[file_search_store.name])]
)

02 Vertex AI Agent Builder — Data Store

Part of Vertex AI Search (built on the Discovery Engine API), inside the Agent Builder / AI Applications console. Designed for enterprise-scale grounding across structured and unstructured sources.

Vertex AI Search Data Store

Managed, connector-driven
AutomatedYou control it

The Discovery Engine API powers Vertex AI Search, letting you create data stores and run semantic search over private documents — mixing unstructured files, structured records, and websites in one search app. Ingestion sources include Cloud Storage, BigQuery, website crawl, and third-party connectors, with the console guiding setup end to end: pick a source, import documents, and the data store handles chunking and indexing on its own schedule.

01Connect source
02Import / sync
03Auto-index
04Search app
05Grounded answer

Good fit

  • Multiple ingestion sources — GCS, BigQuery, websites, and connectors — feeding one search app
  • Enterprise requirements: IAM-backed access control, audit, and console-level governance
  • Native ADK integration — no custom wrapper needed
  • Blended structured + unstructured retrieval (e.g. documents alongside a BigQuery table)

Real constraints

  • Console/setup overhead is heavier than File Search — you provision a data store and a search app, not just a file upload
  • Data stores need time to ingest before they're queryable, so first-run latency is higher
  • Ranking and chunking are still Google-managed — you're trading File Search's simplicity for governance, not for retrieval control
ADK note — Google ships a built-in VertexAiSearchTool specifically to ground responses on a data store you've already created in AI Applications — you pass the search engine ID and it's usable immediately, no custom wrapper.

Minimal shape (Python)

from google.adk.tools import VertexAiSearchTool

grounding_tool = VertexAiSearchTool(
    search_engine_id="projects/YOUR_PROJECT_ID/locations/global/collections/default_collection/engines/YOUR_SEARCH_APP_ID"
)

root_agent = Agent(
    model="gemini-2.5-flash",
    name="kb_agent",
    tools=[grounding_tool]
)

03 Vertex AI RAG Engine (also branded Knowledge Engine)

Google's answer to "I want RAG plumbing I can actually tune." Sits between the two options above: managed infrastructure, but you choose the corpus, embedding, and retrieval parameters.

RAG Engine — Corpus + Retrieval Tool

Configurable, still managed infra
AutomatedYou control it

A RAG corpus is a single ingested collection — think of it as a folder of related files optimized for similarity search — and you can query one corpus for targeted retrieval or several at once for broader recall. Ingestion pulls from local files, Cloud Storage, or Google Drive, then chunks, embeds, and indexes into that corpus, and by default the vectors sit in RagManagedDb, a fully managed Spanner instance offered in a cost-effective Basic tier (fixed, small footprint) or an autoscaling Scaled tier (1–10 nodes) for production load.

01Ingest files
02Chunk
03Embed
04Index → Corpus
05Retrieve (top-k)
06Generate + cite

Good fit

  • You need tunable retrieval parameters — similarity top-k, distance threshold — without hand-rolling a vector database
  • Multiple corpora queried selectively (per topic, per project, per tenant)
  • Deployment to Vertex AI Agent Engine, where the agent's identity is auto-granted permission to query its corpus
  • Citations back to source documents are native to the retrieval response

Real constraints

  • The VertexAiRagRetrieval tool can only be used by itself within an agent instance — it doesn't combine with other tools on the same agent without a workaround
  • You still own more setup than File Search: creating the corpus, the GCS staging bucket, and wiring the corpus ID into your agent
  • Uses vertexai.preview.rag — a preview surface, so expect API movement

Minimal shape (Python)

from google.adk.tools.retrieval.vertex_ai_rag_retrieval import VertexAiRagRetrieval
from vertexai.preview import rag

retrieval_tool = VertexAiRagRetrieval(
    name="retrieve_documentation",
    description="Retrieve relevant documents from the knowledge base",
    rag_resources=[rag.RagResource(rag_corpus="projects/PROJECT/locations/us-central1/ragCorpora/CORPUS_ID")],
    similarity_top_k=10,
    vector_distance_threshold=0.6,
)

root_agent = Agent(model="gemini-2.5-flash", name="rag_agent", tools=[retrieval_tool])

04 Other paths worth knowing

If none of the three above fit, these are the usual next stops — each trading more automation for more control.

Vertex AI Vector Search (Matching Engine)

The raw approximate-nearest-neighbor index underneath Google's managed options, exposed directly. You choose the embedding model, build the chunking pipeline yourself, and query the index directly. Full control, full responsibility — this is what you reach for when RAG Engine's parameters aren't enough.

AlloyDB / Cloud SQL with pgvector

Keep vectors alongside relational data you already run in Postgres. Useful when retrieval needs to join against transactional data, or when a team's operational comfort is SQL, not a separate search product.

BigQuery vector search

Embeddings stored as a column, searched with VECTOR_SEARCH in SQL. Fits when your source data is already a BigQuery warehouse and you'd rather not move it anywhere else for retrieval.

Custom pipeline (LlamaIndex / LangChain)

Full ownership of chunking, embedding model, reranker, and vector store, orchestrated yourself and called as an ADK custom tool. Same ceiling as Vector Search, but with a framework doing the wiring — the default choice when a team already standardized on one of these libraries.

06 Automation ranking, end to end

Ranked by how much of the pipeline — ingestion, indexing, agent wiring, and governance — is handled for you out of the box.

1

Vertex AI Search data store

Most automation overall: connector-driven ingestion, console-managed sync, native ADK tool, and IAM governance baked in — the closest thing to a turnkey enterprise RAG stack.

2

Gemini File Search Store

Fastest to a working prototype — no infra to provision — but automation stops at the file boundary: no connectors, no native ADK tool, and you write the wrapper yourself.

3

Vertex AI RAG Engine

Managed indexing and a native retrieval tool, but you assemble the corpus and staging resources yourself — automation of the backend, manual assembly of the pipeline.

4

Vector Search / custom

Least automated by design — every stage from loader to reranker is something your team owns and maintains.

07 Decision guide

  • Just testing an idea, one document set, no infra appetite → Gemini File Search Store.
  • Multiple source types (files + BigQuery + websites) and you need IAM-backed access control → Vertex AI Search data store.
  • You need retrieval tuning (top-k, distance thresholds, multiple corpora) but don't want to run your own vector DB → Vertex AI RAG Engine.
  • You need a specific embedding model, custom reranker, or already standardized on LlamaIndex/LangChain → Vertex AI Vector Search or a fully custom pipeline.
  • Vectors need to live next to relational or warehouse data you already query → AlloyDB/Cloud SQL pgvector, or BigQuery vector search.
Compiled reference — Gemini API File Search, Vertex AI Agent Builder / Discovery Engine, Vertex AI RAG Engine (Knowledge Engine), and adjacent GCP retrieval options. Verify current quotas and pricing against Google's docs before committing to an architecture, as these limits move quickly.
