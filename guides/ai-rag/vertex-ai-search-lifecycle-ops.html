Part A — Implementation Lifecycle

1. Planning & Design

Define the retrieval boundary. Vertex AI Search organizes content into data stores (the ingested corpus) and search apps / engines (the queryable surface, which can span one or more data stores). Decide this before provisioning:

  • One data store per domain/tenant vs. one shared data store with metadata-based filtering
  • Which source types you need: unstructured documents (GCS), structured records (BigQuery), website content (crawl), or third-party connectors
  • Whether you need a pure "search" app or a "chat/answer" app (grounded generation on top of search)

Data classification. For each content source, capture:

Attribute Why it matters
Sensitivity tier Drives ACL/IAM design in Section 3
Owner / steward Who approves ingestion and re-ingestion
Refresh cadence Determines sync schedule (Section 4)
Format PDF, HTML, structured JSON, BigQuery table — affects parser config
Expected volume Data stores accept large corpora, but plan schema and quota against your actual document count and size

Capacity and quota planning. Check current per-project quotas for data stores, search apps, and daily import volume before committing an architecture — these are managed by Google and do change over time, so confirm against the Discovery Engine documentation for your project's tier rather than assuming a fixed number.

Naming convention. Fix a convention early since it's expensive to rename later:

ds-<domain>-<env>            # data store id
se-<domain>-<env>            # search engine/app id

2. Prerequisites & Environment Setup

Enable APIs:

gcloud services enable discoveryengine.googleapis.com \
  --project=YOUR_PROJECT_ID

IAM roles — assign at the narrowest scope that works:

Role Purpose Assign to
roles/discoveryengine.admin Full data store / engine lifecycle management Platform/infra team, CI service account
roles/discoveryengine.editor Create/update content, no deletion of engines Data ingestion service account
roles/discoveryengine.viewer Query-only access Application/agent runtime service account

Service accounts. Separate the identity that ingests data from the identity that queries it — this is the single most important control for least-privilege in this architecture:

gcloud iam service-accounts create sa-search-ingest --project=YOUR_PROJECT_ID
gcloud iam service-accounts create sa-search-query  --project=YOUR_PROJECT_ID

gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
  --member="serviceAccount:sa-search-ingest@YOUR_PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/discoveryengine.editor"

gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
  --member="serviceAccount:sa-search-query@YOUR_PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/discoveryengine.viewer"

Source bucket / dataset access. Grant the ingest service account roles/storage.objectViewer on the source GCS bucket, or roles/bigquery.dataViewer on the source BigQuery dataset — the ingest identity needs to read the source, not just write to Discovery Engine.

3. Provisioning the Data Store

Console path (fastest for first pass): AI Applications → Data Stores → Create data store → choose source (Cloud Storage / BigQuery / Website / Third-party connector) → select unstructured/structured/site-search mode.

Terraform (repeatable, CI-driven):

resource "google_discovery_engine_data_store" "kb" {
  data_store_id             = "ds-support-prod"
  location                  = "global"
  display_name              = "Support KB — Production"
  industry_vertical         = "GENERIC"
  content_config            = "CONTENT_REQUIRED"
  solution_types             = ["SOLUTION_TYPE_SEARCH"]
  project                    = var.project_id
}

resource "google_discovery_engine_search_engine" "app" {
  engine_id     = "se-support-prod"
  collection_id = "default_collection"
  location      = google_discovery_engine_data_store.kb.location
  display_name  = "Support Search App — Production"
  project       = var.project_id
  data_store_ids = [google_discovery_engine_data_store.kb.data_store_id]

  search_engine_config {
    search_add_ons = ["SEARCH_ADD_ON_LLM"]
  }
}

Keep separate .tfvars per environment (dev, staging, prod) and separate state — data stores are not the place to share resources across environments, since ingestion mistakes in dev should never be able to touch prod indices.

Post-create validation: confirm the data store shows ACTIVE state before pointing any ingestion job at it — a data store still provisioning will silently reject or queue import calls.

4. Data Ingestion Pipeline

Import job pattern (GCS source):

gcloud alpha discovery-engine documents import \
  --data-store=ds-support-prod \
  --location=global \
  --gcs-source-uris=gs://your-bucket/docs/* \
  --reconciliation-mode=INCREMENTAL

INCREMENTAL adds/updates without wiping existing documents; use FULL only for scheduled full refreshes where you intend to replace the entire corpus.

Metadata and ACL mapping. If documents carry per-document access restrictions, map them into the document's structured metadata at import time (e.g., allowed_groups: ["team-a"]) — Vertex AI Search supports filtering search results by such metadata at query time, but you are responsible for populating it correctly during ingestion. This is the most common source of "wrong document surfaced" incidents in production.

Sync scheduling. For recurring sources (a docs wiki, a ticket system connector), use Cloud Scheduler → Cloud Run/Cloud Function → import API, on a cadence matched to how often the source actually changes — not more often, since every import run consumes quota and triggers reindexing.

Ingestion pipeline diagram:

Source (GCS / BigQuery / connector)
   → Extract & validate (format, ACL metadata)
   → Import API (incremental or full)
   → Discovery Engine indexing (managed, async)
   → Data store ACTIVE / queryable

5. Search App / Engine Configuration

  • Enable the generative answer / search add-on if you want grounded natural-language answers rather than a raw result list.
  • Configure facets and filters on structured metadata fields you populated during ingestion (domain, sensitivity tier, document type).
  • Set result ranking controls (boost/bury rules) if certain sources should be preferred — e.g., an official policy doc over a Slack export.
  • If serving multiple domains from one engine, plan your filter expression now; retrofitting domain isolation after launch means re-tagging the whole corpus.

6. Agent Integration (ADK)

Vertex AI Search has a native ADK tool — no custom wrapper required:

from google.adk.agents import Agent
from google.adk.tools import VertexAiSearchTool

grounding_tool = VertexAiSearchTool(
    search_engine_id="projects/YOUR_PROJECT_ID/locations/global/collections/default_collection/engines/se-support-prod"
)

root_agent = Agent(
    model="gemini-2.5-flash",
    name="kb_agent",
    instruction="Answer using the search tool. Cite the source document for every claim.",
    tools=[grounding_tool],
)
  • Test the agent against held-out questions whose correct source document you already know, before wiring it into any orchestration layer.
  • If you run multiple domain-scoped agents, one VertexAiSearchTool per engine (or per filter expression against a shared engine) keeps blast radius contained.

7. Evaluation & Testing

Test type What it checks Suggested cadence
Retrieval precision/recall Does the right document surface for a labeled question set? Every ingestion pipeline change
Answer groundedness Does the generated answer only assert what the retrieved chunks support? Every model/prompt change
ACL/filter correctness Does a restricted-access query correctly exclude out-of-scope documents? Every metadata schema change
Load test Query latency and error rate under expected concurrent load Before each production promotion
Regression suite Fixed question set, expected answer set, run in CI Every deploy

Build the labeled question set from real stakeholder questions, not synthetic ones — synthetic questions tend to match document language too closely and overstate retrieval quality.

8. CI/CD & Promotion Across Environments

Pipeline shape:

PR merged → CI builds/validates Terraform → terraform plan (dev)
  → apply (dev) → run eval suite → promote config to staging
  → apply (staging) → run eval suite + load test → manual approval
  → apply (prod) → smoke test → monitor
  • Promote configuration (Terraform, ingestion job definitions, ADK agent code) through environments — never promote a data store itself; each environment gets its own data store created from the same Terraform module.
  • Gate production promotion on the eval suite passing above a fixed threshold, not just "no errors" — a silent retrieval quality regression is the more common failure mode than an outright error.
  • Store the search engine ID and data store ID as environment-specific Terraform outputs, consumed by the agent's deployment config — never hardcode them in agent source.

9. Security & Compliance Sign-off

Before go-live, confirm:

  • [ ] Ingest and query service accounts are separate, each with least-privilege roles
  • [ ] ACL metadata is populated and filter expressions tested against at least one restricted-access scenario
  • [ ] Audit logging is enabled on the Discovery Engine API (Cloud Audit Logs → Data Access logs)
  • [ ] Source bucket/dataset access is reviewed — the ingest SA should not have write access to production source data
  • [ ] A documented data retention and deletion process exists for the data store (see Section 15)

Part B — Post-Deployment Operations

10. Monitoring & Observability

Key signals to dashboard (Cloud Monitoring):

Metric Why
Query latency (p50/p95/p99) User-facing performance; p99 spikes often indicate ingestion contention
Query error rate Distinguish quota errors from malformed queries from backend faults
Import job success/failure rate Silent ingestion failures leave a stale corpus with no visible symptom until users notice
Document count per data store (trend) Sudden drops indicate a bad FULL reconciliation; sudden spikes indicate unintended duplicate ingestion
Zero-result query rate Rising trend suggests corpus gaps or a broken filter expression

Logging. Enable Cloud Audit Logs (Data Access) on discoveryengine.googleapis.com so every query and import call is attributable to a service account and timestamp — essential for the incident types in Section 12.

Alerting thresholds (starting points, tune to your traffic): - Import job failure → page on-call immediately (stale-data risk compounds silently) - Query error rate > 2% over 5 minutes → alert - Zero-result rate > 15% over 1 hour → alert (usually a filter or corpus regression, not a traffic issue)

11. Data Freshness & Reindexing

  • Maintain a freshness SLA per source (e.g., "wiki content reflected within 4 hours of edit") and monitor actual import-to-queryable latency against it.
  • Run a scheduled full reconciliation (e.g., weekly) even if incremental syncs are working, to catch drift from deleted-but-not-retracted documents.
  • Log every import job's reconciliation-mode, source URI pattern, and document count delta — this log is your primary debugging tool when someone reports "that document isn't showing up."

12. Incident Response & Runbooks

Runbook: Stale or missing document reported 1. Confirm the document exists in the source system and check its last-modified timestamp. 2. Query Cloud Audit Logs for the most recent import job covering that source path. 3. If the import succeeded but the document is absent, check ACL/metadata filters — the most common cause is a mis-tagged allowed_groups field silently excluding it from the querying identity's results. 4. If the import failed, check the import job's error detail for format/parsing issues, then re-run in INCREMENTAL mode after fixing the source file.

Runbook: Sudden spike in zero-result queries 1. Check whether a recent search-app config change (filter expression, facet config) shipped around the same time. 2. Check document count trend — a large unexplained drop indicates a bad FULL reconciliation overwrote the corpus. 3. Roll back the most recent Terraform apply for the search engine config if the timing correlates.

Runbook: Query latency degradation 1. Check concurrent query volume against known quota/capacity limits. 2. Check whether a recent ingestion burst is competing with query serving (large imports can affect index freshness operations concurrently). 3. Escalate to Google Cloud support with the project ID and time window if no application-side cause is found — this is managed infrastructure, and sustained latency issues outside your control are a support-case, not a config fix.

Escalation path: on-call engineer → platform team lead → Google Cloud support (for anything confirmed as backend-side, since Discovery Engine internals aren't directly inspectable).

13. Scaling & Capacity Management

  • Vertex AI Search scales ingestion and query serving as managed infrastructure — your primary capacity planning task is quota management, not server sizing.
  • Track daily import volume and query volume against project quotas monthly; request quota increases proactively before a known traffic event (a product launch, a new domain onboarding), not reactively.
  • If you're running many domain-scoped data stores, periodically review whether some should consolidate into a shared store with filter-based isolation — many small stores multiply the operational surface (each needs its own monitoring, ingestion job, and access review) without a retrieval-quality benefit.

14. Cost Management

  • Cost drivers are primarily indexing volume (documents stored/updated) and query volume — track both per data store, not just at the project level, so you can attribute cost to the domain/team that owns each store.
  • Set a budget alert per project tied to Discovery Engine API spend.
  • Review low-value data stores (low query volume relative to ingestion cost) quarterly — a domain that's rarely queried but re-ingests on a tight schedule is a common silent cost sink.

15. Backup, Versioning & Rollback

  • The data store itself is not a point-in-time-recoverable resource in the way a database is — your real backup is the source system (GCS bucket, BigQuery table, wiki) plus your ingestion job definitions, since a data store can always be rebuilt from source via a FULL reconciliation.
  • Version your Terraform configuration and ingestion job definitions in source control — this is your actual "rollback" mechanism for the data store and search app configuration.
  • For search-app config changes (ranking rules, facets), keep the previous Terraform state's plan output archived so you can diff and revert precisely.
  • Document a data retention and deletion procedure: how to fully purge a document (and confirm it's gone from the index, not just the source) when required for compliance.

16. Access Review & Governance Cadence

Activity Cadence
Review IAM bindings on ingest/query service accounts Quarterly
Review ACL metadata accuracy against a sample of restricted documents Quarterly
Review data store inventory (are all stores still owned/needed?) Semi-annually
Audit log review for anomalous query patterns Monthly, or continuous via alerting

17. Change Management / Release Process

  1. All data store, search engine, and ACL schema changes go through Terraform PR review — no console-only changes in staging/prod.
  2. Every PR that touches ingestion or filter logic must include an eval suite run against the affected domain before merge.
  3. Production changes require a documented rollback plan (usually: revert the Terraform PR and re-apply) before approval.
  4. Maintain a changelog per data store — what changed, when, why, who approved — since "why does this document rank differently now" is a recurring support question.

18. Troubleshooting Reference

Symptom Likely cause First check
Document missing from results ACL/metadata mismatch, or import not yet complete Audit log for last import job on that source path
Wrong/outdated answer generated Stale corpus (sync lag) or generative add-on not grounded Check document last-indexed timestamp vs. source edit time
Sudden latency spike Concurrent large import job, or quota near-exhaustion Cloud Monitoring — import job status + quota dashboard
Zero results for a previously-working query Filter expression or facet config regression Diff recent Terraform applies on the search engine resource
403 on query Query service account IAM role missing/removed Confirm roles/discoveryengine.viewer binding still present

19. Appendix — CLI / Terraform Cheatsheet

# Enable API
gcloud services enable discoveryengine.googleapis.com

# List data stores
gcloud alpha discovery-engine data-stores list --location=global

# Trigger an incremental import
gcloud alpha discovery-engine documents import \
  --data-store=ds-support-prod --location=global \
  --gcs-source-uris=gs://your-bucket/docs/* \
  --reconciliation-mode=INCREMENTAL

# Check IAM bindings
gcloud projects get-iam-policy YOUR_PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:sa-search-query@*"

terraform plan  -var-file=envs/prod.tfvars
terraform apply -var-file=envs/prod.tfvars

Confirm exact command surfaces (gcloud alpha vs. stable, resource field names) against current Google Cloud documentation before running in production — the Discovery Engine API surface has moved fast and alpha/preview flags are common.
