Resilience Pattern
Circuit Breaker Implementation

Yes — Circuit Breakers are essential in this architecture. Since services call each other synchronously (Cart → Course, Cart → Payment, Subscription → Payment), a failure in one service can cascade and take down the entire platform. Circuit Breakers prevent this by detecting failures and stopping calls to unhealthy services.

🟢 CLOSED
Normal operation. All requests pass through. Failure count monitored. Threshold: <5 failures in 60s.
🔴 OPEN
Circuit tripped. All requests immediately fail-fast. No calls made to downstream. Fallback logic executes.
🟡 HALF-OPEN
After timeout (30s), allows 1 probe request. If it succeeds → CLOSED. If it fails → back to OPEN.

pybreaker or tenacity

Distributed Transactions
Saga Pattern Implementation

In a microservices architecture, you can't use a traditional database transaction across multiple services. The Saga Pattern solves this by breaking a distributed transaction into a sequence of local transactions — each publishing an event or message. If any step fails, compensating transactions are executed to rollback previous steps and maintain data consistency.

Failure Path — Compensating Transactions (Rollback)

If any step fails, the Saga Orchestrator triggers compensating transactions in reverse order to undo completed steps. Each service must implement a rollback action.

Step That Failed Compensating Action Service Responsible Event Published
Payment fails Release cart lock, notify user Cart Service payment.failed
Subscription activation fails Refund payment via Stripe Payment Service payment.refunded
Enrollment grant fails Cancel subscription + refund payment Subscription + Payment enrollment.failed
Any step times out Saga marks step as failed, triggers rollback chain Saga Orchestrator saga.rollback
Saga State Machine — saga_state Table
STARTED
Saga created, first step executing
IN_PROGRESS
Steps executing sequentially
COMPLETED
All steps succeeded
COMPENSATING
Rollback in progress
FAILED
Rollback complete, user notified
SAGA_STATE TABLE SCHEMA
saga_id UUID PRIMARY KEY  ·  user_id UUID  ·  order_id UUID  ·  current_step VARCHAR  ·  status ENUM  ·  payload JSONB  ·  created_at TIMESTAMP  ·  updated_at TIMESTAMP
🎯
Idempotency
Each saga step must be idempotent — if replayed due to failure, it should not double-charge or double-enroll. Use unique idempotency keys per saga step.
⏱️
Timeouts
Each step has a timeout. If a service doesn't respond in time, the orchestrator treats it as failure and begins compensating transactions automatically.
📋
Audit Trail
Every saga state transition is logged. This gives you a full audit trail of every purchase attempt, what succeeded, what failed, and what was rolled back.
