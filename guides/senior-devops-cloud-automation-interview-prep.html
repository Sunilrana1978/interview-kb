01

Go Programming

Must-have

The single non-negotiable skill. Be ready to write idiomatic Go live, reason about concurrency, and explain why Go fits infra/CLI tooling.

G1Why is Go a good fit for building internal infrastructure tools and CLIs? What would push you toward Go over Python here?{chev}
Talking points
  • Single static binary, no runtime/interpreter to ship — ideal for distributing tools across heterogeneous warehouse/edge hosts.
  • Fast startup and low memory; cross-compilation (GOOS/GOARCH) for ARM edge devices from one build host.
  • First-class concurrency for tools that fan out across many nodes/clusters.
  • Strong stdlib for networking, HTTP, and CLI; the cloud-native ecosystem (K8s, Terraform, ArgoCD, Docker) is Go, so SDKs/clients are native.
  • Honest trade-off: Python wins for quick glue/data work; Go wins when the tool is long-lived, performance-sensitive, or shipped as a binary.
G2Explain goroutines vs OS threads, and how channels coordinate them. When would you use a channel vs a mutex?{chev}
Talking points
  • Goroutines are lightweight, multiplexed onto OS threads by the runtime scheduler (M:N), starting at ~2KB stacks that grow — so thousands are cheap.
  • Channels pass ownership of data ("share memory by communicating"); good for pipelines, fan-out/fan-in, signaling completion.
  • Mutex protects shared state read/written in place — simpler for a counter or a cache than routing it through a channel.
  • Rule of thumb: channels for transferring work/ownership; mutex (sync.Mutex/RWMutex) for guarding shared state.
  • Mention sync.WaitGroup, and that buffered vs unbuffered channels change blocking semantics.
Follow-up: "How do you avoid a goroutine leak?" → ensure every goroutine has a termination path; use context cancellation and bounded channels.
G3How do you use context.Context? Walk through cancellation and timeouts in a tool that calls multiple cloud APIs.{chev}
Talking points
  • context carries deadlines, cancellation signals, and request-scoped values down a call tree.
  • Create with context.WithTimeout/WithCancel; pass as the first arg to every blocking call; always defer cancel().
  • On cancel/timeout the ctx.Done() channel closes — downstream SDK calls (AWS SDK v2 takes a ctx) abort instead of hanging.
  • Don't stuff business data in context values; use it for cancellation and tracing IDs.
G4How does Go's error handling work, and how do you design errors in a library others will consume? Discuss wrapping and sentinel errors.{chev}
Talking points
  • Errors are values, returned explicitly — no exceptions. Handle at the right layer; add context as it bubbles up.
  • Wrap with fmt.Errorf("doing X: %w", err) to preserve the chain; unwrap with errors.Is / errors.As.
  • Sentinel errors (var ErrNotFound = errors.New(...)) or typed errors let callers branch on conditions.
  • panic only for truly unrecoverable programmer errors; recover at goroutine boundaries in long-running daemons so one bad request doesn't kill the process.
G5Live-coding style: write a worker pool that processes N jobs across a fixed number of concurrent workers and collects results.{chev}
Structure they want to see
  • A jobs channel and a results channel; spin up a fixed pool of worker goroutines ranging over jobs.
  • sync.WaitGroup to know when all workers finish, then close results in a separate goroutine.
  • Pass context so the pool can be cancelled mid-run; handle per-job errors without killing the pool.
  • Talk about bounding concurrency to protect downstream (API rate limits, node capacity) — directly relevant to fanning out across a warehouse fleet.
Watch for: deadlocks from unclosed channels, and the classic loop-variable capture bug in goroutines (pre-1.22).
G6How do you test Go code? Cover table-driven tests, interfaces for mocking, and the race detector.{chev}
Talking points
  • Table-driven tests with subtests (t.Run) for clear case coverage.
  • Depend on small interfaces so you can inject fakes for cloud clients; avoid heavyweight mock frameworks where a hand-written fake is clearer.
  • go test -race in CI to catch data races; -cover for coverage; benchmarks via testing.B.
  • Mention httptest for HTTP handlers and golden files for CLI output.
G7How do you profile and reduce memory/CPU in a Go service? What about GC tuning?{chev}
Talking points
  • pprof (CPU, heap, goroutine, block profiles); go tool trace for scheduler/latency analysis.
  • Reduce allocations: reuse buffers (sync.Pool), preallocate slices/maps, avoid unnecessary interface boxing and string→[]byte churn.
  • GOGC trades CPU for memory; GOMEMLIMIT (1.19+) gives a soft cap — useful in memory-constrained edge containers.
  • Measure first; don't optimize on a hunch.
02

Kubernetes & Docker

Core

You'll run self-hosted clusters, not just managed ones. Expect cluster-internals and "it's broken at 2am" troubleshooting questions.

K1Walk me through what happens, end to end, when you kubectl apply a Deployment until pods are running.{chev}
Talking points
  • Request hits the API server → auth/authz/admission → persisted to etcd.
  • Deployment controller creates a ReplicaSet; ReplicaSet controller creates Pods (desired state).
  • Scheduler binds each unscheduled Pod to a node via filtering + scoring (resources, affinity, taints).
  • Kubelet on the node pulls images, asks the CRI runtime (containerd) to start containers, runs probes, reports status.
  • kube-proxy / CNI wire up networking; Service + endpoints make it reachable.
K2A pod is stuck in CrashLoopBackOff. Another is Pending. How do you debug each?{chev}
Talking points
  • CrashLoop: kubectl logs --previous, check exit code, describe for OOMKilled, verify config/secrets/env, failing liveness probe, bad command. Reproduce locally if possible.
  • Pending: kubectl describe pod → events. Usually unschedulable: insufficient CPU/mem, no node matches affinity/taints, PVC unbound, or image pull issues on a node.
  • Mention cluster-level: cluster-autoscaler can't add nodes, or quota exhausted.
K3Requests vs limits — how do they affect scheduling, QoS class, and eviction? What's a sane default policy?{chev}
Talking points
  • Requests drive scheduling and reservations; limits cap usage (CPU throttled, memory OOMKilled).
  • QoS: Guaranteed (requests=limits) > Burstable > BestEffort — order in which pods get evicted under node pressure.
  • Sane policy: always set memory request=limit to avoid OOM surprises; set CPU request, be cautious with hard CPU limits (throttling); use LimitRange/ResourceQuota per namespace.
  • For latency-sensitive robotics workloads, avoid aggressive CPU throttling and consider guaranteed QoS.
K4How does pod networking work? Explain the CNI's role, ClusterIP vs NodePort vs LoadBalancer, and how a Service routes traffic.{chev}
Talking points
  • Every pod gets a routable IP; CNI plugin (Calico, Cilium, etc.) handles pod-to-pod connectivity and NetworkPolicy enforcement.
  • Service is a stable virtual IP + DNS name; endpoints track healthy pods; kube-proxy (iptables/IPVS) or eBPF programs the routing.
  • ClusterIP = internal only; NodePort = port on every node; LoadBalancer = provisions an external LB (ELB/NLB on AWS).
  • Ingress / Gateway API for L7 HTTP routing in front of ClusterIP services.
  • On self-hosted/on-prem, LoadBalancer needs something like MetalLB since there's no cloud LB.
K5What is a CRD and the operator pattern? When would you build a custom operator instead of using Helm?{chev}
Talking points
  • CRD extends the K8s API with new resource types; an operator is a controller that reconciles those resources toward desired state (control loop).
  • Helm packages/templates manifests — great for install/upgrade, but it's one-shot, not a continuous reconciler.
  • Build an operator when you need ongoing operational logic: handling stateful upgrades, failover, backups, or domain workflows (e.g., managing a fleet of robotics controllers as CRs).
  • Tie-in: operators are typically written in Go (controller-runtime / Kubebuilder) — connects to the Go must-have.
K6How do you build a small, secure container image? Cover multi-stage builds, distroless/scratch, non-root, and image scanning.{chev}
Talking points
  • Multi-stage build: compile in a builder stage, copy only the static Go binary into a minimal final stage (scratch or distroless).
  • Run as non-root, read-only root filesystem, drop capabilities; pin base image digests.
  • Scan in CI (Trivy/Grabber); keep images small to speed pulls — matters for bandwidth-limited edge sites.
  • Mention layer ordering for cache efficiency and not baking secrets into layers.
K7How do you run and maintain a self-hosted Kubernetes cluster? What's harder than a managed cluster like EKS?{chev}
Talking points
  • You own the control plane: etcd backups/restore, certificate rotation, version upgrades (kubeadm/kubespray or a lighter distro like k3s for edge).
  • You own load balancing (MetalLB), storage (CSI for local/NFS/Ceph), and networking yourself.
  • Capacity, node lifecycle, and OS patching are on you — no managed node groups.
  • For warehouse edge: lightweight distros (k3s/RKE2), constrained hardware, and surviving network partitions to the cloud.
K8How do you do zero-downtime deploys in K8s? Compare rolling, blue/green, and canary, and how readiness probes fit in.{chev}
Talking points
  • Rolling (default): maxUnavailable/maxSurge control pace; readiness probes gate traffic so only healthy pods receive it.
  • Blue/green: full second environment, flip traffic, instant rollback — costs double capacity.
  • Canary: shift a % of traffic, watch metrics, promote — best paired with Argo Rollouts + a metrics provider.
  • PodDisruptionBudgets protect availability during voluntary disruptions (node drains/upgrades).
🔥 Common Production Issues — Kubernetes & Docker
P1Issue: Pods stuck in ImagePullBackOff across the cluster after a routine deploy.{chev}
Root causes & fixes
  • Wrong image tag or image deleted: a tag like latest or a feature-branch tag was pushed over or removed. Pin to immutable SHA digests in production manifests.
  • ECR auth token expired: the imagePullSecret or node IAM role lost permissions. Check ecr:GetAuthorizationToken on the node role; credential helpers (ECR Credential Helper) auto-refresh tokens.
  • VPC endpoint / network broken: the CNI or a security group rule change cut egress to ECR. Verify the endpoint exists and SG rules allow 443 outbound.
  • Private registry with no imagePullSecret: missing or expired Secret in the target namespace. Verify with kubectl get secret; use a service account–bound pull secret.
  • Rate limiting (Docker Hub): anonymous pull limits hit. Move to ECR mirrors or authenticated pulls.
Quick triage: kubectl describe pod <name> → Events section shows the exact pull error and registry response code.
P2Issue: Node goes NotReady intermittently; pods on it get evicted and reschedule endlessly.{chev}
Root causes & fixes
  • Kubelet stops sending heartbeats: high load, OOM, or a stuck systemd unit. SSH to the node: systemctl status kubelet, journalctl -u kubelet -n 100.
  • Disk pressure: /var/lib/docker or /var/lib/containerd full from image/log bloat — triggers eviction. Add a logrotate policy, set container log limits, prune dangling images via a CronJob.
  • Memory pressure / OOMKill: the kernel kills the kubelet or containerd. Set --eviction-hard thresholds to proactively shed pods before the kernel OOMKills critical daemons.
  • Network plugin crash: CNI pod (Calico/Cilium) on the node crashed — network handshake to the control plane times out. Check CNI pod status on that node.
  • Cloud instance health issue: AWS underlying hardware degraded; replace the instance and let the autoscaler heal.
Prevention: node-problem-detector daemonset surfaces kernel and system-level issues as NodeConditions before kubelet gives up.
P3Issue: Service is up, pod is running, but requests return 503 or connections are refused intermittently.{chev}
Root causes & fixes
  • Pod not yet ready, already in endpoint list: readiness probe passes but the app isn't truly accepting requests (JVM warm-up, DB connection). Tune the probe's initialDelaySeconds and failureThreshold.
  • Terminating pod still in endpoints: during a rolling deploy, traffic hits a pod already in Terminating state. Add a preStop sleep (e.g., 5s) to give kube-proxy time to drain the endpoint before the process exits.
  • kube-proxy lag: iptables/IPVS rules haven't propagated yet. Usually transient but can be prolonged on a busy node with many services.
  • NetworkPolicy blocking: a new policy accidentally denies ingress. kubectl describe networkpolicy and test with a debug pod using curl.
  • Resource exhaustion: the container is up but CPU-throttled and unable to serve in time; check throttle metrics in cAdvisor/Prometheus.
P4Issue: Cluster autoscaler is not scaling up even though pods are Pending.{chev}
Root causes & fixes
  • PodAffinity/anti-affinity rules or taints: the pending pod can only land on a specific node type that the autoscaler can't provision in the configured node group. Review and relax affinity rules or add a matching node group.
  • Max node group size reached: the ASG or Karpenter limit is hit. Raise the cap if justified, or review over-large resource requests.
  • Cluster autoscaler IAM perms missing: it can't call the EC2/ASG APIs. Check its service-account permissions.
  • Pods requesting resources no node can satisfy: a single pod requesting 96 vCPU won't fit any instance type in the node group — change the resource request or add a larger instance type.
  • Scale-up cooldown / stabilization window: autoscaler is waiting before acting. Check the --scale-down-delay-after-add and --scan-interval flags in its logs.
Quick check: kubectl logs -n kube-system deployment/cluster-autoscaler — it logs the exact reason it decided not to scale.
P5Issue: OOMKilled containers — pod keeps restarting, memory limit looks fine on paper.{chev}
Root causes & fixes
  • Limit set too low for actual workload: the app has a memory spike on startup or under certain inputs. Profile with heap dumps; raise the limit and align JVM/Go runtime heap settings to it.
  • Memory leak: RSS grows unbounded over time. Inspect cgroup memory stats via kubectl top pod + time-series in Prometheus; the leak must be fixed in the app.
  • JVM/language runtime ignoring container limits: JVM pre-1.10 didn't respect cgroup limits and allocated based on host RAM, causing OOM. Use a modern JVM and set -XX:MaxRAMPercentage.
  • Init containers or sidecars counted against the pod budget: the main container's limit is fine but total pod memory exceeds the node. Account for all containers when sizing.
  • tmpfs / emptyDir volumes consuming memory: in-memory volumes count against the container's working set. Watch for log or cache dirs mounted as medium: Memory.
Diagnosis: kubectl describe pod shows OOMKilled with exit code 137; dmesg on the node shows the kernel OOM event and killed process.
P6Issue: Docker image builds are slow in CI and layer cache is never reused.{chev}
Root causes & fixes
  • COPY source code before dependency install: any code change invalidates the dependency layer. Restructure: COPY go.mod go.sum . → RUN go mod download → COPY . . → RUN go build.
  • No cache mount in CI runner: ephemeral runners don't preserve the Docker layer cache between runs. Use BuildKit cache mounts (--mount=type=cache) or push/pull a cache image from a registry.
  • Using --no-cache in the pipeline: sometimes set defensively. Remove it; let layer hashes do their job.
  • Base image not pinned to a digest: rebuilds always pull latest and Docker can't confirm the cache is valid. Pin to a digest or a specific version tag.
  • Large context directory: .dockerignore missing — multi-GB node_modules, dist, or binary artifacts are sent to the daemon on every build. Add a thorough .dockerignore.
🎯 Top 5 Most-Asked Interview Questions — Kubernetes & Docker
#1What is the difference between a Deployment, StatefulSet, and DaemonSet? When do you use each?{chev}
Model answer
  • Deployment: manages identical, stateless pods; any pod can die and be replaced by an identical one. Use for web/API services, Go microservices, workers. Rolling updates are the default.
  • StatefulSet: pods get stable network identities and ordered start/stop. PVC per pod persists across rescheduling. Use for databases, Kafka, ZooKeeper, anything that needs stable storage or peer identity.
  • DaemonSet: exactly one pod per (matching) node. Use for node-level agents: log collectors (Fluentd/Promtail), monitoring (node-exporter), CNI plugins, security agents.
  • The interviewer wants to hear: it comes down to identity, storage, and topology needs — not just "stateful vs stateless."
#2How does Kubernetes handle service discovery? How does a pod know where to find another service?{chev}
Model answer
  • Every Service gets a DNS name: <service>.<namespace>.svc.cluster.local. CoreDNS resolves it to the ClusterIP.
  • ClusterIP is a virtual IP; kube-proxy (iptables/IPVS) or eBPF maps it to one of the healthy pod IPs (from the Endpoints/EndpointSlice object).
  • Pods inherit the cluster DNS resolver automatically via /etc/resolv.conf injected by the kubelet.
  • For headless services (clusterIP: None), DNS returns pod IPs directly — useful for StatefulSets where a client needs to address a specific replica.
  • Environment variable injection is legacy/secondary; DNS is the standard and preferred mechanism.
#3What is a Kubernetes Namespace and how do you use it for multi-tenancy and resource isolation?{chev}
Model answer
  • Namespaces are a logical partition of the cluster — scoping for names, RBAC, NetworkPolicy, and ResourceQuotas.
  • Use one namespace per team/environment/application for isolation. RBAC binds roles at the namespace level so teams can't touch each other's resources.
  • ResourceQuotas cap CPU/memory/object count per namespace; LimitRanges set defaults and floors for pod requests/limits.
  • Soft multi-tenancy: namespaces + RBAC + NetworkPolicy is sufficient for trusted teams. For hostile multi-tenancy (e.g., per-customer), you need separate clusters or a dedicated isolation layer.
  • Cluster-scoped resources (Nodes, PersistentVolumes, ClusterRoles) are not namespaced.
#4Explain liveness vs readiness vs startup probes. What happens when each fails?{chev}
Model answer
  • Readiness: "is this container ready to serve traffic?" Failure removes the pod from the Service endpoints — traffic stops, container keeps running. The most impactful probe to get right.
  • Liveness: "is this container still alive?" Failure triggers a container restart. Use for detecting deadlocks. Don't make it too aggressive — a misconfigured liveness probe is a common cause of cascading restarts under load.
  • Startup: gates liveness/readiness during slow-start apps (JVM, ML model loading). Gives the container time to initialize before the stricter probes kick in.
  • Common mistake: making the readiness probe check a downstream dependency — if the DB is down, all pods become not-ready and the app goes completely dark. Check only local health.
#5What is the difference between a Docker ENTRYPOINT and CMD? How does K8s command/args map onto them?{chev}
Model answer
  • ENTRYPOINT: the fixed executable that always runs; can't be overridden by docker run arguments (only by --entrypoint).
  • CMD: default arguments to the entrypoint (or the full command if no entrypoint). Easily overridden at runtime.
  • Best practice for a Go binary: ENTRYPOINT ["/app/server"] and use CMD for default flags that operators can override.
  • In K8s: spec.containers[].command overrides ENTRYPOINT; spec.containers[].args overrides CMD. Commonly confused source of "my container runs fine in Docker but not in K8s."
  • Shell vs exec form matters: exec form (["cmd","arg"]) runs PID 1 directly and receives signals correctly; shell form wraps in /bin/sh -c and signals may not reach the app.

AWS Cloud

Core

Your strong suit. Expect questions on EKS, networking, IAM for workloads, and connecting cloud to on-prem warehouses.

A1How do you give a pod in EKS scoped access to AWS services without static credentials?{chev}
Talking points
  • IRSA (IAM Roles for Service Accounts): the cluster OIDC provider lets a K8s service account assume a scoped IAM role; the SDK picks up temporary creds automatically.
  • Newer: EKS Pod Identity simplifies the association without per-cluster OIDC plumbing.
  • Principle of least privilege per workload; no node-wide instance-profile sharing, no long-lived keys in secrets.
A2Design VPC networking for an EKS cluster that also needs private connectivity to AWS services and to an on-prem warehouse.{chev}
Talking points
  • Multi-AZ VPC, private subnets for nodes, public subnets only for ingress/NAT; nodes egress via NAT Gateway.
  • VPC endpoints (Gateway for S3/DynamoDB, Interface/PrivateLink for ECR, STS, etc.) to keep traffic off the public internet and cut NAT cost.
  • On-prem link: Site-to-Site VPN for quick/backup, Direct Connect for stable low-latency; Transit Gateway to hub multiple VPCs/accounts.
  • Watch CIDR overlap with on-prem ranges; plan IP space for the VPC CNI (pods consume VPC IPs).
A3When would you reach for AWS Outposts, Snowball Edge, or IoT Greengrass for warehouse/edge workloads?{chev}
Talking points
  • Outposts: AWS-managed rack on-site for low-latency, data-residency, or local processing needs while staying in the AWS control plane.
  • Greengrass: run Lambda/containers and ML at the edge on constrained devices, with local messaging and intermittent-connectivity sync to the cloud.
  • Snowball Edge: bulk data transfer / edge compute where bandwidth is the bottleneck.
  • Be honest: many edge cases are better served by a lightweight self-hosted K8s on local hardware + cloud control plane — match the tool to latency/connectivity constraints.
A4Secrets Manager vs SSM Parameter Store — how do you decide, and how do you surface secrets to pods?{chev}
Talking points
  • Secrets Manager: built-in rotation, cross-account sharing, higher per-secret cost — use for DB creds/API keys needing rotation.
  • Parameter Store: cheap/free standard tier for config and lightweight secrets (SecureString), good for plain config.
  • Into pods: External Secrets Operator or Secrets Store CSI driver, scoped via IRSA — keeps secrets out of Git for GitOps.
A5EKS node group is healthy but pods can't pull images and can't reach S3. Where do you look?{chev}
Talking points
  • Egress path: NAT Gateway / route tables for private subnets; security groups and NACLs.
  • ECR: missing VPC endpoint or IAM perms on the node role / IRSA; check ecr:GetAuthorizationToken.
  • S3: Gateway endpoint route entry present? Bucket policy / VPC endpoint policy blocking?
  • DNS resolution (CoreDNS) and the VPC CNI IP exhaustion as secondary suspects.
A6How do you think about cost on a multi-cluster AWS platform without hurting reliability? (FinOps angle){chev}
Talking points
  • Right-size requests/limits (over-requesting wastes nodes); use cluster-autoscaler/Karpenter to match capacity to demand.
  • Spot for fault-tolerant/batch workloads, on-demand/reserved for critical control-plane-adjacent services.
  • Cut NAT and inter-AZ data costs with VPC endpoints and topology-aware routing; tag for cost allocation; dashboards before mandates.
  • Frame it as: visibility → allocation → optimization, with reliability SLOs as a guardrail.

CI/CD & ArgoCD GitOps

Core

The deployment model is explicitly GitOps via ArgoCD, across cloud and edge. Know the pull-based model cold.

C1What is GitOps, and how is ArgoCD's pull model different from a traditional push-based CI/CD pipeline?{chev}
Talking points
  • GitOps: Git is the single source of truth for desired state; a controller continuously reconciles the cluster to match.
  • Push: CI has cluster credentials and runs kubectl apply — secrets leave the cluster, drift is invisible.
  • Pull: ArgoCD runs inside the cluster, watches the repo, and applies changes — no external creds, automatic drift detection and (optional) self-heal.
  • Audit trail = Git history; rollback = git revert.
  • Pull model is a big win for many edge clusters: each cluster pulls its own config rather than CI needing reachability into every warehouse.
C2Describe ArgoCD's architecture: Application, sync, health, the app-of-apps pattern, and how you'd manage 50 edge clusters.{chev}
Talking points
  • An Application maps a Git source (path/branch/Helm/Kustomize) to a destination cluster+namespace; ArgoCD diffs live vs desired.
  • Sync = apply desired; sync waves/hooks order resources; health checks roll up resource status.
  • App-of-apps: a root Application that manages child Applications — bootstraps a whole environment from one entry point.
  • For fleet scale: ApplicationSets with generators (cluster/Git generators) to template one definition across many clusters; per-cluster overlays via Kustomize.
C3How do you handle secrets in a GitOps repo? You can't commit plaintext.{chev}
Talking points
  • Never commit raw secrets. Options: External Secrets Operator pulling from AWS Secrets Manager, or Sealed Secrets (encrypted, only the cluster controller can decrypt), or SOPS-encrypted files.
  • Git holds only references or ciphertext; the actual secret lives in a secret store, scoped by IRSA.
  • Rotation handled in the store, not by re-committing.
C4How would you do progressive delivery (canary/automated rollback) with ArgoCD?{chev}
Talking points
  • Argo Rollouts replaces the Deployment with a Rollout CR supporting canary/blue-green strategies and traffic shaping.
  • Analysis templates query a metrics provider (Prometheus/CloudWatch); promote on healthy metrics, auto-rollback on breach.
  • Pair with progressive sync across cluster tiers (dev → staging → a few warehouses → fleet) to limit blast radius.
C5Design the full path from a code merge to a Go service running on edge clusters. What's in CI vs CD?{chev}
Talking points
  • CI: on merge → test (go test -race), lint, build multi-arch image, scan, push to ECR, then bump the image tag in the config repo (PR or automated commit).
  • CD: ArgoCD detects the config-repo change and syncs to target clusters; ApplicationSet fans it across the fleet with environment overlays.
  • Separate app repo from config repo so deploys are config changes, not rebuilds.
  • Edge nuance: clusters pull when reachable, so a temporarily offline warehouse converges automatically when it reconnects.
C6An ArgoCD app shows OutOfSync even though no one changed Git. What's happening and how do you respond?{chev}
Talking points
  • Drift — something mutated the live state directly (a hotfix, an HPA changing replicas, a mutating webhook, or default fields).
  • Diff the live vs desired; decide: ignore expected diffs (ignoreDifferences for HPA-managed replicas), or enable self-heal to snap it back.
  • Cultural point: direct kubectl edits in a GitOps world are an anti-pattern — fix it in Git.

Terraform & Ansible

Core

Provisioning (Terraform) vs configuration (Ansible). Be crisp on state, drift, and when each tool is the right one.

T1How do you manage Terraform state safely for a team across multiple environments and accounts?{chev}
Talking points
  • Remote backend (S3 + DynamoDB lock, or S3 native locking) — never local state for teams.
  • Isolate state per environment/account (separate state files or workspaces) to limit blast radius; encrypt at rest.
  • Reusable modules with versioned inputs; plan in CI on PR, gated apply on merge.
  • Mention drift: terraform plan as detection; avoid out-of-band console changes.
T2Terraform vs Ansible — what does each own, and where do they overlap? Give a concrete warehouse example.{chev}
Talking points
  • Terraform: declarative provisioning of cloud/infra resources (VPC, EKS, on-prem VMs via providers), tracks state.
  • Ansible: agentless configuration management — install packages, push OS config, deploy onto existing hosts; procedural and idempotent.
  • Example: Terraform stands up the warehouse edge nodes/VPC link; Ansible configures the OS, joins them to the k3s cluster, and pushes node-level settings.
  • Overlap exists, but use each for its strength rather than forcing one to do both.
T3What does idempotency mean in Ansible, and how do you keep playbooks idempotent and safe to re-run?{chev}
Talking points
  • Re-running yields the same end state with no unintended changes; modules report changed only when they actually change something.
  • Prefer declarative modules over raw shell/command; when you must shell out, add creates/changed_when guards.
  • Use handlers for restarts, --check (dry run) and --diff to preview, and roles for structure.
T4How do you structure Terraform to avoid a monolithic state and reduce blast radius as the platform grows?{chev}
Talking points
  • Split by lifecycle/ownership: networking, cluster, platform add-ons, apps each in their own state.
  • Pass values between layers via remote state outputs or a data source / parameter store — avoid one giant apply.
  • Versioned modules + a tool like Terragrunt or a CI orchestration layer to keep DRY across many accounts.
  • Trade-off: more states = more coordination but far smaller blast radius.
🔥 Common Production Issues — Terraform & Ansible
P7Issue: terraform apply destroys and re-creates a critical production resource (e.g., RDS instance, EKS cluster).{chev}
Root causes & fixes
  • Force-replace triggered by an immutable argument change: changing engine_version, availability_zone, or a tag that forces replacement. Always run terraform plan in CI and block merge if any plan shows -/+ on production resources.
  • Moved or renamed resource in code: Terraform sees the old address as "delete" and the new as "create." Use the moved block (1.1+) or terraform state mv to tell Terraform it's the same resource.
  • Module version bump changed a resource's schema: a new module version renamed or restructured resources under the hood. Pin module versions in source and review changelogs before upgrading.
  • State drift — someone manually modified the resource: the real state diverged; Terraform wants to make it match code. Use terraform import or terraform state rm + re-import to reconcile, then fix the root cause (no console edits).
  • Mitigation: add lifecycle { prevent_destroy = true } to critical stateful resources as a safety net; it forces a conscious override.
Golden rule: treat any -/+ or - on a prod resource in the plan as a blocking review item, not a normal approval.
P8Issue: Terraform state lock is stuck; no one can run plan or apply.{chev}
Root causes & fixes
  • A previous apply was killed mid-run: the DynamoDB lock item was never released. Identify who owns the lock from terraform force-unlock output or directly in DynamoDB (LockID item), then run terraform force-unlock <ID>.
  • CI pipeline crashed or was cancelled: same root cause. Add a pipeline post-step that calls force-unlock on failure, or use a lock timeout.
  • Concurrent runs: two PRs merged at the same time triggered simultaneous applies. Serialize applies with a CI queue or use Terraform Cloud/Atlantis which manages this natively.
  • Prevention: S3 native state locking (AWS 2024) removes the DynamoDB dependency entirely; consider migrating.
P9Issue: terraform plan shows a large diff of "no-op" changes on every run — nothing was changed in code.{chev}
Root causes & fixes
  • Provider version not pinned: a new provider version changed how it reads/normalizes an attribute. Pin providers in required_providers with exact versions; commit the lock file (.terraform.lock.hcl).
  • Ordering-sensitive attributes: lists like security group rules or IAM policy statements are order-sensitive in state but the API returns them in a different order. Use sets where possible, or ignore_changes for stable-but-noisy attributes.
  • Default values injected by the API: AWS fills in defaults (e.g., tags, description) that weren't in your config. Bring them into the config explicitly or use ignore_changes.
  • Timestamp or computed fields: some resources always have a "last modified" or generated ARN that Terraform re-reads. Add them to lifecycle.ignore_changes.
P10Issue: Ansible playbook partially fails midway; re-running it leaves infrastructure in an inconsistent state.{chev}
Root causes & fixes
  • Non-idempotent tasks: a shell task that appends a line without a guard has already run on 3 of 10 hosts. Audit all raw shell tasks; add creates, removes, or changed_when: false guards, or replace with a declarative module.
  • Missing error handling: a task failing on one host aborts the whole play. Use ignore_errors: true cautiously, block/rescue/always for structured error handling, and any_errors_fatal: false to isolate failures per host.
  • Partial package installs: apt/yum transaction interrupted. Declarative package modules handle this gracefully on re-run; avoid wrapping package installs in raw shell commands.
  • Re-run strategy: use --limit failed_hosts to re-run only on broken hosts; Ansible's retry file tracks them automatically after a failed run.
  • Prevention: always test playbooks with --check --diff in CI against a staging group before production; use molecule for role-level testing.
P11Issue: Ansible runs take 45+ minutes on a 200-node fleet; changes to one host group still affect everything.{chev}
Root causes & fixes
  • Default serial = all hosts: Ansible fans out via SSH but with a small default fork count (5). Raise -f / forks in ansible.cfg for faster parallel execution.
  • Monolithic playbook — no tagging: running the full play when only the app config changed. Tag tasks (tags: app_config) and run with --tags to target only what changed.
  • SSH connection overhead: each task opens a new connection by default. Enable pipelining in ansible.cfg (pipelining = True) and use ControlMaster/ControlPersist to reuse SSH connections.
  • Fact gathering on all hosts: gather_facts: true on 200 hosts adds minutes. Disable globally and enable only where facts are needed, or cache facts with Redis/memcached (fact_caching).
  • Architecture fix: restructure into focused roles run via CI only when the relevant part of the repo changes; use pull-based config (AWX/AAP schedules or triggered webhooks) instead of a serial push.
🎯 Top 5 Most-Asked Interview Questions — Terraform & Ansible
#1What is Terraform state, why is it needed, and what are the risks if it gets out of sync?{chev}
Model answer
  • State is Terraform's record of what it created and the mapping between config and real-world resource IDs. Without it, Terraform can't compute what to change or delete.
  • Out-of-sync state is called drift — it happens when someone changes a resource outside Terraform (console, CLI, another tool). The next plan may try to revert or re-create things.
  • Detection: terraform plan shows drift as unexpected changes. Fix with terraform import (add untracked resources) or terraform refresh (update state from reality) plus fixing the config.
  • Mitigation: prevent out-of-band changes via SCPs/deny policies; treat the plan output as a required code-review artifact.
  • State contains secrets (DB passwords, etc.) in plaintext — encrypt S3 bucket with SSE-KMS; restrict IAM access tightly.
#2What are Terraform modules? How do you design a good module vs a bad one?{chev}
Model answer
  • A module is a directory of .tf files with defined inputs and outputs — the unit of reuse and encapsulation in Terraform.
  • Good module: single, well-scoped responsibility (e.g., "EKS node group"), opinionated defaults with escape-hatch variables, stable interface, versioned in a registry, well-documented outputs.
  • Bad module: a wrapper that exposes every single argument of the underlying resources (essentially no abstraction); or the opposite — a mega-module that provisions an entire environment as one unit (too much blast radius, slow to plan).
  • Treat modules like APIs: semver them, don't break consumers, and add a changelog. Use terraform-docs to auto-generate documentation.
  • Pin module sources to a version tag, not a branch, in production.
#3How does Terraform handle dependency ordering? When do you use depends_on explicitly?{chev}
Model answer
  • Terraform builds an implicit dependency graph from resource references — if resource B uses an attribute of resource A (resource_a.id), Terraform creates A first automatically.
  • Use depends_on only for hidden dependencies Terraform can't infer from the config — e.g., a Lambda that reads from an S3 bucket via an IAM role that isn't referenced by ARN in the Lambda resource itself.
  • Overuse of depends_on forces sequential execution and slows applies; let the reference graph do the work wherever possible.
  • At module level, depends_on on a module affects all resources inside it — blunt instrument; prefer scoping it to specific resources.
#4What are Ansible roles and why should you use them instead of one big playbook?{chev}
Model answer
  • A role is a structured, self-contained unit of automation: tasks, handlers, defaults, vars, templates, and files — all in a predictable directory layout that Ansible auto-loads.
  • Benefits: reusable across projects and playbooks; independently testable with Molecule; separates concerns so the web role doesn't know about the DB role; easier to version and share via Ansible Galaxy.
  • A flat playbook with 500 tasks becomes unmaintainable and impossible to test in isolation. Roles enforce boundaries.
  • Compose roles into playbooks for different host groups; pass behavior via role variables rather than duplicating tasks.
  • Mention include_role vs import_role: import is static (parsed at playbook load, supports tags); include is dynamic (runtime, can use variables in the role name).
#5How do you manage secrets in Terraform and Ansible? What are the anti-patterns?{chev}
Model answer
  • Terraform: never put secrets in *.tf or terraform.tfvars in Git. Pull them at plan time from Secrets Manager / SSM via data sources, or inject via environment variables (TF_VAR_). Mark outputs sensitive = true to suppress log output.
  • Ansible: use ansible-vault to encrypt vars files at rest (committed as ciphertext); or pull from HashiCorp Vault / AWS Secrets Manager via a lookup plugin at runtime. Never define secrets in group_vars plaintext.
  • Anti-patterns: hardcoded passwords in playbooks/templates; secrets in CI environment variables visible in build logs; no_log: false on tasks that print secrets; state files with DB passwords in an unencrypted bucket.
  • Audit tip: git-secrets or detect-secrets as a pre-commit hook catches accidental commits before they reach the repo.

Linux & Networking

Core

"Strong Linux systems and networking fundamentals" — expect first-principles troubleshooting, not memorized flags.

L1A Linux host is "slow." Walk me through how you triage CPU, memory, disk, and network.{chev}
Talking points
  • Start broad: top/htop, load average vs core count, vmstat.
  • CPU: user vs sys vs iowait — high iowait points at disk; iostat/iotop.
  • Memory: free, check swap, OOM killer in dmesg.
  • Network: ss -s, iftop, retransmits; DNS latency.
  • Narrow to the process: strace/perf if needed. Mention the USE method (Utilization, Saturation, Errors).
L2Explain Linux namespaces and cgroups, and how they make containers work.{chev}
Talking points
  • Namespaces isolate what a process sees: PID, network, mount, UTS, IPC, user — each container gets its own view.
  • cgroups limit what a process can use: CPU, memory, I/O.
  • A container = a process with its own namespaces + cgroup limits + a layered filesystem; no separate kernel (unlike a VM).
  • This is why K8s requests/limits map onto cgroups, and memory limits trigger OOMKill.
L3"I can't reach service X from this host." How do you debug it layer by layer?{chev}
Talking points
  • DNS first: does the name resolve (dig/nslookup)?
  • L3/L4: ping (if ICMP allowed), traceroute, then curl -v / nc to the port.
  • Firewalls/SGs/NACLs/NetworkPolicy, routing tables, MTU mismatches.
  • TLS handshake / cert issues at L7. Confirm the service is actually listening (ss -tlnp).
L4How does systemd manage services, and how would you make a Go daemon resilient on an edge node?{chev}
Talking points
  • Unit file with Restart=on-failure, resource limits, After=/Requires= ordering, journald logging.
  • Health via a watchdog or readiness check; graceful shutdown on SIGTERM (handle the signal in Go).
  • On constrained edge hardware, set memory/CPU accounting so a runaway process can't take down the node.
L5Subnetting/CIDR quick check: a warehouse needs ~500 hosts. What prefix, and how do you avoid overlap with the cloud VPC?{chev}
Talking points
  • 500 hosts → need 512 addresses → /23 (510 usable).
  • Allocate non-overlapping RFC1918 ranges per site; document an IP plan so VPN/Direct Connect routes don't collide.
  • Leave headroom for the K8s pod/service CIDRs, which must also not overlap cross-site.

Observability

Core

You'll own monitoring, logging, and alerting across distributed systems. Talk in SLO terms, not just dashboards.

O1Metrics, logs, and traces — what is each best for, and how do they work together in an incident?{chev}
Talking points
  • Metrics: cheap, aggregate, alert on trends/thresholds — your "what's wrong and how bad."
  • Traces: request path across services — your "where the latency/error is."
  • Logs: detailed, high-cardinality context — your "why."
  • Flow: an alert fires (metric) → trace narrows to a service/span → logs explain the failure. Correlate via trace/request IDs.
  • Mention OpenTelemetry as the vendor-neutral instrumentation layer feeding Prometheus/CloudWatch/Datadog.
O2Define SLI, SLO, and error budget. How do they change how you alert?{chev}
Talking points
  • SLI = a measured signal (e.g., success rate, p99 latency). SLO = the target for it. Error budget = 1 − SLO, the allowed unreliability.
  • Alert on symptoms that burn the budget (user-facing), not on every CPU blip — reduces noise.
  • Burn-rate alerts: page fast on rapid budget burn, ticket on slow burn.
  • Budget also governs release pace: out of budget → freeze risky deploys.
O3Your on-call is drowning in alerts. How do you fix alert fatigue?{chev}
Talking points
  • Every page must be actionable and tied to user impact; delete or downgrade the rest to dashboards/tickets.
  • Symptom-based alerting over cause-based; group/dedupe related alerts; add good runbooks.
  • Review alert quality regularly (pages per shift, % actionable); treat noisy alerts as bugs.
O4How would you instrument and monitor a distributed pipeline that crosses async boundaries (queues/events)?{chev}
Talking points
  • Propagate trace context through the message (e.g., inject/extract headers) so a trace survives the queue hop.
  • Track queue depth, consumer lag, processing latency, and DLQ counts as first-class metrics.
  • Watch for the "fire and forget" blind spot — correlate producer and consumer with a shared ID.
  • OpenTelemetry context propagation across SNS/SQS/Lambda-style boundaries is the standard approach.
O5What would you monitor that's specific to robotics/warehouse edge that you wouldn't for a typical web service?{chev}
Talking points
  • Connectivity/heartbeat per site, time-since-last-sync, and behavior during cloud disconnects.
  • Device/robot health: command-to-actuation latency, missed cycles, sensor staleness, firmware/version skew across the fleet.
  • Local resource pressure on constrained edge hardware; clock skew (critical for real-time coordination).
  • Buffer observability locally so you don't lose telemetry during a network partition; ship on reconnect.

Edge, Hybrid & Robotics

Preferred · differentiator

This is what makes the role unusual. Even without robotics history, show you understand the constraints: latency, partitions, mission-critical safety.

E1A warehouse loses its link to the cloud for 30 minutes. What should keep working, and how do you design for that?{chev}
Talking points
  • Local autonomy: the on-prem cluster/control plane must keep robots and core flows running without the cloud (design for partition tolerance).
  • Buffer telemetry/events locally and reconcile on reconnect (eventual consistency, idempotent replay).
  • Cache config/images locally (local registry mirror) so nothing depends on a live pull.
  • GitOps pull model helps: each site converges on its own when the link returns.
  • Be explicit about what is safe to be stale vs what must be locally authoritative.
E2How do you manage software updates and rollouts across many physically distributed edge sites safely?{chev}
Talking points
  • Staged/ring rollouts: canary one or two sites, bake, then expand — never all sites at once.
  • Automated health gates + fast rollback; immutable, versioned artifacts so any site can pin/revert deterministically.
  • Bandwidth-aware delivery (small images, local mirrors); handle sites that are offline during the rollout window.
  • Maintenance windows that respect warehouse operating hours / safety.
E3What does "mission-critical / real-time" change about how you build and operate infrastructure?{chev}
Talking points
  • Latency determinism matters more than throughput; avoid noisy-neighbor and CPU-throttle surprises (guaranteed QoS, careful limits).
  • Failure must be safe: graceful degradation and defined fail-safe states for physical equipment, not just retries.
  • Higher bar on testing, change control, and rollback — a bad deploy can stop a physical line or risk safety.
  • Redundancy and clear blast-radius boundaries per site.
  • Honesty: I'd lean on the robotics/controls engineers for the hard real-time guarantees; my job is the platform reliability around them.
E4Would you run full Kubernetes at the edge? How do you choose between k3s, RKE2, full K8s, or no K8s at all?{chev}
Talking points
  • Match to hardware and ops capacity: k3s/RKE2 for small footprint and simpler ops at remote sites; full K8s where the site is large and well-resourced.
  • Sometimes the answer is no orchestrator — systemd + a managed binary is simpler and more robust for a tiny fixed workload.
  • Consistency vs simplicity trade-off: a uniform platform across sites lowers cognitive load, but don't over-engineer a closet with three nodes.
E5You have no direct robotics experience. Why should we trust you in this domain?{chev}
How to frame it (honest + confident)
  • Map your real experience to the constraints: distributed systems, hybrid/on-prem, low-latency, mission-critical reliability, fleet operations.
  • Show you understand why robotics is harder (physical safety, real-time, partitions) and that you respect the boundary with controls engineers.
  • Point to where you've learned a new domain fast before; lead with a concrete prior system that had similar properties.
  • Don't overclaim robotics expertise — claim platform expertise and curiosity.

Reliability & Scale

Core

"Improve system reliability, scalability, and deployment workflows" is a core duty. Expect incident and design questions.

R1Walk me through how you handle a production incident from page to postmortem.{chev}
Talking points
  • Stabilize first: assess impact, declare severity, assign roles (incident commander, comms, ops).
  • Mitigate before root-cause — roll back, fail over, shed load — restore service, then investigate.
  • Communicate to stakeholders on a cadence; keep a timeline.
  • Blameless postmortem with action items; fix the system and the process, not the person.
R2A region/AZ goes down. How did you design the platform to survive it, and what's the recovery plan?{chev}
Talking points
  • Multi-AZ by default (nodes/data spread); decide multi-region by RTO/RPO and cost.
  • Stateless services scale anywhere; stateful needs replication/backups and a tested restore.
  • Know your hard dependencies (e.g., a single-region control service) and remove single points of failure.
  • Edge angle: warehouses keep running locally during a cloud-region event — design the cloud as the brain, not the heartbeat.
  • DR is only real if you've tested the failover.
R3How do you make deployments safe by default so a bad change can't take down the platform?{chev}
Talking points
  • Progressive rollout + automated health checks + one-click (or automatic) rollback.
  • Limit blast radius: stage by environment and by site ring; feature flags to decouple deploy from release.
  • Immutable, versioned artifacts; reproducible builds; everything in Git for instant revert.
  • Pre-prod parity and good test gates in CI before anything reaches a warehouse.
R4How do you approach capacity planning and autoscaling for spiky, distributed workloads?{chev}
Talking points
  • HPA on the right signal (custom/queue metrics, not just CPU); cluster-autoscaler/Karpenter for node capacity.
  • Headroom for spikes; understand scale-up latency (node provisioning can be slower than the spike).
  • Load-test to find the knee; set sane min replicas so cold starts don't bite.
  • Edge sites are often fixed-capacity — there you optimize and prioritize rather than scale out.
