01 VPC Fundamentals

A Virtual Private Cloud (VPC) is your private, isolated network on AWS. Understanding its building blocks is foundational to every architecture conversation.

CIDR Planning — What is a CIDR?

CIDR (Classless Inter-Domain Routing) notation expresses an IP address range as base_IP / prefix_length. The prefix length (the number after /) defines how many bits are fixed (the network portion) — all remaining bits are free to be assigned to hosts. The more bits you fix, the smaller the range.

Formula: Total IPs = 2^(32 - prefix). AWS reserves 5 IPs per subnet (network address, VPC router, DNS, future use, broadcast), so usable IPs = Total − 5.

RFC 1918 Private Ranges

Only use these for AWS VPCs. Never use public IP space as your VPC CIDR.

  • 10.0.0.0/8 — 16.7M IPs. Avoid using the full /8; carve /16 blocks per VPC (10.0.0.0/16, 10.1.0.0/16, etc.)
  • 172.16.0.0/12 — 1M IPs. Range 172.16–172.31.x.x. Useful for mid-size orgs.
  • 192.168.0.0/16 — 65K IPs. Fine for small isolated VPCs; often conflicts with home/office routers — avoid for hybrid setups.
Planning Rules for Enterprise
  • Assign a /16 per VPC — gives 65K IPs and room for 256 /24 subnets
  • Never overlap CIDRs with on-prem or other VPCs you'll peer/TGW with — this cannot be changed after creation
  • Add secondary CIDRs later if you run out (up to 5 per VPC), but plan upfront
  • Reserve CIDR blocks centrally (in IPAM or a spreadsheet) before provisioning
  • Use AWS VPC IP Address Manager (IPAM) for org-wide visibility and allocation
  • Leave gaps between VPC CIDRs for future growth (e.g. 10.0.0.0/16, 10.2.0.0/16, skip 10.1)

VPC 3-Tier Multi-AZ Architecture — NAT Gateway per AZ (HA)
Why one NAT Gateway per AZ? NAT Gateways are AZ-scoped resources. If you put a single NAT GW in AZ-a and AZ-b or AZ-c goes down, instances in those AZs lose egress. Worse, if AZ-a itself fails, everything loses internet egress. Deploying one per AZ means each AZ is self-sufficient — and avoids cross-AZ data transfer costs ($0.01/GB each way). The rule: each private subnet's route table must point 0.0.0.0/0 to the NAT GW in the same AZ.
Route Tables — How AWS Routing Works

Every subnet is associated with exactly one route table. When a packet leaves an instance, AWS looks up the destination IP in the route table and applies the most specific matching route (longest prefix match). Routes are evaluated by specificity — a /32 beats a /24 beats 0.0.0.0/0.

AWS always adds a local route for the VPC CIDR automatically — this cannot be deleted and takes priority over all other routes for intra-VPC traffic.

Public Subnet Route Table

Attached to public subnets (10.0.1.0/24, 10.0.2.0/24, 10.0.3.0/24). The 0.0.0.0/0 → IGW route makes these subnets "public".

DestinationTargetStatusTriggered byExample request
10.0.0.0/16 local Active Any intra-VPC traffic ALB → App server (10.0.11.25)
0.0.0.0/0 igw-xxxxxx Active Any traffic not matched above User browser → ALB public IP
Note: The subnet also needs Auto-assign public IPv4 enabled AND a security group that allows inbound 443/80. The route table alone isn't enough — both conditions must be true for internet traffic to reach an instance.
Private App Subnet Route Table (per AZ)

Each AZ has its own route table pointing 0.0.0.0/0 to the NAT GW in the same AZ. This avoids cross-AZ traffic if one AZ's NAT GW fails.

Private Data Subnet Route Table

Data tier has NO internet route at all. It can only receive traffic from within the VPC. S3/Secrets Manager are accessed via VPC Gateway/Interface Endpoints — no NAT needed.

DestinationTargetStatusTriggered byExample request
10.0.0.0/16 local Active App tier connecting to DB App (10.0.11.25) → RDS (10.0.21.14) on port 5432
pl-xxxxxxxx (S3) vpce-xxxxxx Active S3 traffic via Gateway Endpoint RDS export → S3 bucket (no NAT cost)
No 0.0.0.0/0 route here. If a misconfigured service in the data tier tries to call the internet, it will fail with "no route to host." This is intentional and is the correct security posture. Any AWS service access (Secrets Manager, SSM, KMS) should be configured as Interface VPC Endpoints in the data subnet.

Routing Priority (longest prefix wins)
  • /32 (host route) — highest priority
  • /24, /20, /16 — more specific wins
  • 0.0.0.0/0 — default route, lowest priority
  • Local VPC route always beats 0.0.0.0/0
  • Propagated routes from VGW/TGW added automatically via BGP
Security Groups vs NACLs
  • SGs: stateful, instance-level
  • NACLs: stateless, subnet-level
  • SG deny is implicit; NACL has explicit allow/deny
  • NACLs evaluated by rule number order
VPC Flow Logs
  • Capture IP traffic metadata (not payload)
  • Published to CloudWatch Logs or S3
  • Used for security forensics, bandwidth analysis
  • Enable at VPC, subnet, or ENI level
02 VPC-to-VPC Connectivity

Three main options for connecting VPCs: Peering, Transit Gateway, and PrivateLink. Choosing the right one depends on the traffic pattern, scale, and transitive routing needs.

Transit Gateway Hub-and-Spoke Architecture (Multi-Account)
VPC Peering — When to use it ▼
VPC Peering is a one-to-one, non-transitive connection between two VPCs. Traffic stays on AWS backbone. Use it when you have a small number of VPCs that need direct communication and CIDRs don't overlap. Avoid for 5+ VPCs — the number of connections grows as n(n-1)/2. Does not support transitive routing (A↔B, B↔C does NOT mean A↔C). Does not require a gateway; uses route table entries. Supports cross-account and cross-region (inter-region peering at slightly higher cost).
Transit Gateway — Enterprise scale ▼
Transit Gateway is a managed hub that supports up to 5,000 VPC attachments. It supports transitive routing, route table isolation (for segmentation), VPN attachment, and Direct Connect Gateway attachment. Each attachment gets associated with a TGW route table. Use route table segmentation to isolate production from dev — they share the TGW but can't route to each other. RAM (Resource Access Manager) allows sharing TGW across accounts in an AWS Organization. Bandwidth per attachment: up to 50 Gbps burst.
PrivateLink — Exposing services without network access ▼
AWS PrivateLink exposes a service (behind an NLB) to consumers via an Interface VPC Endpoint, without requiring VPC peering, route tables, or public internet. The consumer only gets access to the specific service, not the entire VPC. Overlapping CIDRs are fine — no routing is involved. Ideal for SaaS vendors, partner integrations, cross-account shared services. The service provider creates an endpoint service; consumers create an Interface Endpoint in their VPC. DNS resolves the endpoint to a private IP in the consumer's VPC.

03 Hybrid Connectivity

Connecting on-premises networks to AWS via Direct Connect or VPN is a critical pattern in enterprise architectures. Understanding the reliability, latency, and cost trade-offs is essential.

Hybrid Connectivity Architecture — Direct Connect + VPN Failover
Direct Connect — Key Facts
  • Dedicated 1G/10G/100G physical connection
  • Hosted connections: 50Mbps–10Gbps via partners
  • Uses BGP for route advertisement
  • Private VIF — connects to VGW/TGW in your VPC
  • Public VIF — reaches AWS public endpoints (S3, DynamoDB)
  • Transit VIF — connects to DX Gateway → TGW
  • No encryption in transit — use MACsec or IPSec tunnel over DX
  • Lead time: weeks to months for provisioning
Site-to-Site VPN
  • IPSec over internet — quick to provision (minutes)
  • Up to 1.25 Gbps per tunnel (2 tunnels per connection)
  • Terminated at Virtual Private Gateway (VGW) or TGW
  • Supports Static routes or BGP (dynamic routing)
  • Accelerated VPN: routes through AWS Global Accelerator edge
  • ECMP: multiple VPN connections to TGW for higher throughput
  • Customer Gateway device must support IKEv1/IKEv2
Critical interview point: Direct Connect does NOT provide encryption by default. For compliance (HIPAA, PCI), you must either use MACsec (Layer 2 encryption on dedicated connections) or run an IPSec tunnel over the DX connection. Always clarify this when asked about securing hybrid connectivity.

04 DNS & Route 53

Route 53 is far more than a DNS service — it's a traffic management platform. Understanding routing policies and the Resolver is critical for hybrid architectures.

Routing Policies
  • Simple — Single record, multiple values returned randomly
  • Weighted — Split traffic by % (A/B testing, gradual cutover)
  • Latency-based — Routes to lowest-latency region
  • Failover — Active/passive with health check
  • Geolocation — Routes by user's geographic location
  • Geoproximity — Routes by geographic proximity with bias
  • Multi-value — Returns up to 8 healthy records
  • IP-based — Routes by client IP/CIDR (newest policy)
Route 53 Resolver (Hybrid DNS)
  • Inbound Resolver Endpoint — DNS queries from on-prem → AWS private hosted zones
  • Outbound Resolver Endpoint — DNS queries from AWS → on-prem DNS servers
  • Resolver Rules forward specific domains to specific DNS servers
  • Share rules across accounts via RAM
  • DNS Firewall: block/allow domains at VPC level
  • DNSSEC supported for public hosted zones
Route 53 Resolver — Hybrid DNS Resolution Flow
Health Checks: Route 53 health checks monitor endpoints every 10 or 30 seconds. Calculated health checks combine results from multiple checks. Combine with Failover routing policy for automatic DNS-level failover between regions (typical RTO: 60–90 seconds for DNS propagation).
05 CloudFront & CDN

CloudFront is AWS's globally distributed CDN with 450+ PoPs. Beyond caching static assets, it's a security perimeter, a real-time compute platform (Lambda@Edge / CloudFront Functions), and a DDoS shield.

Origins & Behaviors
  • S3 Origin — Use OAC (Origin Access Control) to keep bucket private; never public
  • ALB/API GW Origin — Add custom header to only accept requests from CloudFront IPs
  • Cache Behaviors — Path-based routing (/*.jpg → S3, /api/* → ALB)
  • TTL Control — Override via Cache-Control, Expires headers or behavior settings
  • Compress — Enable Gzip/Brotli compression at edge
Security Features
  • AWS WAF — Attach to distribution for L7 filtering
  • AWS Shield Advanced — DDoS protection with response team
  • Geo-restriction — Allowlist/blocklist by country
  • Field-Level Encryption — Encrypt PII at the edge
  • Signed URLs/Cookies — Time-limited access to private content
  • HTTPS only — Redirect HTTP, enforce TLS 1.2+
Lambda@Edge vs CloudFront Functions — Differences ▼
CloudFront Functions: Ultra-lightweight JS, executes in <1ms, runs at ALL 450+ PoPs. Only for Viewer Request/Response events. Max 2MB code size, no network access, no file system. Use for: URL rewrites, header manipulation, A/B testing redirects, query string normalization.

Lambda@Edge: Full Node.js/Python Lambda, runs at regional edge caches (~13 locations). Supports all 4 events (Viewer Request, Origin Request, Origin Response, Viewer Response). Can make network calls, access DynamoDB. Max 5–10 second timeout. Use for: authentication/authorization, SSR, dynamic origin selection, personalization based on cookies.
06 Architecture Patterns

Common networking patterns that come up in whiteboard interviews at the senior architect level.

Egress VPC — Centralized Internet Outbound via TGW
Hub-and-Spoke vs Full Mesh — When to choose ▼
Hub-and-Spoke (TGW): Best for 5+ VPCs, cross-account connectivity, and centralized services. Transitive routing is handled automatically. Route table isolation lets you control which spokes can communicate. Cost: $0.05/hr per attachment + data processing fees.

Full Mesh (VPC Peering): Use when you have 2–4 VPCs with low-latency, high-bandwidth requirements (no TGW throughput overhead). Connections = n(n-1)/2 so 10 VPCs = 45 peering connections. No data processing charge (unlike TGW). Cannot be transitive.
Centralized VPC Endpoints Pattern ▼
Instead of creating Interface VPC Endpoints in every spoke VPC (costly), create them once in a shared services VPC and share via TGW + private DNS. Requires: 1) Interface endpoints in shared VPC, 2) Private hosted zones associated with spoke VPCs, 3) DNS resolution enabled on TGW. Supports services like S3, DynamoDB, SSM, Secrets Manager, ECR. Saves $0.01/hr per endpoint per AZ across many accounts.

Security is layered across VPC design, access controls, traffic inspection, and monitoring. An architect should be able to defend each layer.

Security Group Best Practices
  • Never use 0.0.0.0/0 in inbound rules except for public-facing ALBs on 443/80
  • Reference SG IDs instead of CIDR ranges for dynamic EC2/ECS environments
  • Separate SGs for web tier, app tier, data tier
  • Audit with AWS Config rule restricted-ssh and restricted-common-ports
  • Use prefix lists for shared CIDR management across accounts
Network Firewall vs WAF vs NACL
  • Network Firewall — L3-L7, deployed in subnet, stateful IPS/IDS, domain filtering
  • WAF — L7 HTTP/S only, deployed at ALB/CloudFront/API GW, rule groups
  • NACL — Stateless, subnet boundary, numbered rules, allow + deny

08 Interview Questions & Model Answers

Senior-level questions you should be ready to answer with architecture-depth. Practice narrating these out loud.

Architecture Design
Design a network architecture to connect 40 VPCs across 5 AWS accounts with centralized egress and security inspection.
Answer framework: Use Transit Gateway as the hub, with one TGW per region shared via RAM across all accounts. Spoke VPCs have no IGW — default route (0.0.0.0/0) points to TGW. Create an Egress VPC with AWS Network Firewall → NAT Gateway → IGW for outbound inspection. Use TGW route table segmentation to isolate prod from dev/test (they can't communicate). For inbound traffic, use a centralized Ingress VPC with ALB/NLB → TGW → spoke VPCs. Share VPC Interface Endpoints from a Shared Services VPC via private DNS. Enable VPC Flow Logs to S3/CW for forensics.
Hybrid Connectivity
Your on-premises datacenter needs sub-10ms latency to AWS services and the connection must never go down. What do you architect?
Answer framework: Use Direct Connect Maximum Resiliency model — two dedicated connections at two different DX locations, each attached to the same DX Gateway. Both connections run BGP with active/active (equal metric) for load sharing, or active/passive using BGP LOCAL_PREF. Attach DX Gateway to Transit Gateway with Transit VIF for multi-VPC access. As a cold failover path, configure Site-to-Site VPN attached to the same TGW — BGP will prefer DX (shorter AS path). For encryption: enable MACsec on the dedicated connections if the DX provider supports it, otherwise run IPSec tunnel over DX.
DNS & Networking
EC2 instances in AWS cannot resolve on-premises hostnames (corp.internal). How do you fix this without rebuilding DNS?
Answer framework: Deploy a Route 53 Resolver Outbound Endpoint in the VPC (2 ENIs, one per AZ for HA). Create a Forwarding Rule for domain corp.internal pointing to on-premises DNS server IPs (reachable via DX/VPN). Associate the rule with the VPC. Now any DNS query for corp.internal from EC2 is forwarded to on-prem DNS. For the reverse (on-prem queries to aws.internal), deploy an Inbound Endpoint and configure the on-prem DNS to forward aws.internal queries to the Inbound Endpoint IPs. Share rules across accounts using RAM.
Security
How do you prevent data exfiltration from EC2 instances in a private subnet to the internet?
Answer framework (layered): 1) No direct internet route — private subnets have no IGW. 2) NAT Gateway with restrictive egress SGs (only 443 outbound). 3) Route traffic through Network Firewall (via Egress VPC + TGW) — configure domain-based allow list to block unknown destinations. 4) VPC Endpoints for S3/DynamoDB/SSM to eliminate NAT for AWS services entirely. 5) S3 Bucket Policies with aws:sourceVpc condition to prevent direct S3 writes from outside VPC. 6) GuardDuty for anomaly detection on unusual DNS queries or connection patterns.
Cost Optimization
Your monthly AWS bill shows $40K in data transfer charges. How do you diagnose and reduce this?
Answer framework: 1) Use Cost Explorer grouped by Usage Type to find top transfer categories (inter-AZ, inter-region, NAT GW, CloudFront → origin). 2) Inter-AZ costs — ensure services (RDS, ElastiCache, EKS nodes) are accessed within same AZ. 3) NAT Gateway costs — replace with VPC Endpoints for S3 (free), SSM, ECR. 4) CloudFront — use origin shield to reduce direct origin hits. 5) Direct Connect — if data transfer is consistently high, DX egress is 80% cheaper than internet egress. 6) Enable VPC Flow Logs → Athena to identify top-talkers by IP pair.
