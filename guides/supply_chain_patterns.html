Order Management

4 patterns
Domain Pattern
Order Orchestration
Process Manager EIP
Software Solutions
Manhattan OMS Blue Yonder OMS Salesforce OMS MuleSoft / Apigee
OMS is the central orchestrator: it sequences ATP check → WMS pick/pack → carrier booking → ERP fulfillment record in order. MuleSoft handles the API fan-out and error recovery between each step.
Used At
Nike DTC & wholesale
Columbia multi-channel
Blue Yonder OMS deployments
Domain Pattern
Order Splitting
Splitter EIP
Software Solutions
OMS Rule Engine EDI 850 Vendor REST API
OMS applies sourcing rules (proximity, stock level, SLA cost) and splits one customer order into separate fulfillment sub-orders routed to DC, store, or drop-ship vendor — all running in parallel.
Used At
Nike multi-DC network
Columbia ship-from-store
Drop-ship vendor programs
Domain Pattern
ATP — Available to Promise
Request-Reply EIP
Software Solutions
SAP GATP Blue Yonder OMS ATP Custom ATP microservice
Synchronous Request-Reply pattern. The storefront calls an ATP service which queries real-time inventory across all nodes and returns a promise date before the order is confirmed. Target SLA: <200ms p99.
Used At
Checkout stock validation
Pre-order promising
B2B allocation confirmation
Domain Pattern
Returns / Reverse Logistics
Saga EIP
Software Solutions
Blue Yonder WMS SAP ERP Kafka event chain
A Saga with compensating transactions: RMA creation → carrier label → DC receipt → quality grading → restock or liquidation. Each step publishes a domain event; failures trigger reverse compensations upstream.
Used At
Store returns processing
eComm label generation
DC restocking workflows

Inventory Management

4 patterns
Domain Pattern
Unified Inventory Visibility
Aggregator EIP
Software Solutions
Apache Kafka Redis cache Manhattan Active Omni Fluent Commerce IMS SAP CAR
Aggregator EIP. Each inventory node publishes stock-on-hand change events to Kafka. A streaming aggregator (Kafka Streams + Redis) merges them into a single real-time unified view consumed by OMS and ATP services.
Used At
Nike omni-channel
Columbia BOPIS
Any multi-node retailer
Domain Pattern
Inventory Reservation
Claim Check EIP
Software Solutions
OMS / IMS module Redis atomic counter Kafka saga
Two-phase reservation: soft hold at cart add (TTL-limited, auto-released on abandon via Redis expiry), hard hold at order confirm. Implemented as an atomic Redis DECR or SELECT FOR UPDATE to prevent overselling during flash sales.
Used At
Flash sale prevention
Cart checkout flow
Warehouse pick allocation
Domain Pattern
CDC for Inventory Sync
Event Sourcing EIP
Software Solutions
Debezium CDC Kafka Connect SAP HANA CDC
Change Data Capture reads the WAL of the ERP/WMS database and streams every inventory mutation as a Kafka event — zero impact on source system performance, sub-second latency to downstream consumers.
Used At
ERP → IMS real-time sync
WMS → analytics pipeline
Inventory audit trail
Domain Pattern
Safety Stock & Reorder Trigger
Event-Driven Consumer
Software Solutions
Blue Yonder Luminate SAP ERP MRP o9 Solutions
When on-hand crosses the reorder point, an automated PO or transfer order fires. BY Luminate uses ML-computed dynamic safety stock adjusting for demand variability, lead time uncertainty, and service level targets.
Used At
DC replenishment
Store-to-store transfers
Seasonal safety stock

Content Enricher EIP
Software Solutions
Kafka Streams Blue Yonder Luminate SAP IBP
POS sale events stream via Kafka → demand sensing engine enriches with weather, events, promos → adjusts short-horizon (0–4 week) forecast → auto-triggers replenishment PO or warehouse transfer order.

Domain Pattern
Exception-Driven Alerting
Message Filter EIP
Software Solutions
Kafka + Apache Flink PagerDuty / Slack API Blue Yonder Control Tower
Flink stream processing evaluates shipment and inventory events against SLA rules. When a rule fires (delivery T+2 overdue, stock below safety), the Message Filter routes an alert to ops — normal events pass through silently.
Used At
Carrier delay escalation
Out-of-stock alerts
ASN discrepancy flags
Domain Pattern
Digital Twin
Event Sourcing EIP
Software Solutions
Kafka + Event Store Blue Yonder Luminate Azure Digital Twins
Every physical state change is appended to an immutable event log. The digital twin rebuilds by replaying the log — enabling what-if simulation (port closure, demand spike) on a shadow copy before any live change is committed.
Used At
DC simulation
Carrier network modeling
Disruption response

Supplier & EDI Integration

4 patterns
Domain Pattern
EDI Integration (X12 / EDIFACT)
Message Translator EIP
Software Solutions
IBM Sterling B2B SPS Commerce Boomi / MuleSoft OpenText B2B
Message Translator EIP: EDI gateway translates internal canonical XML/JSON from SAP/ERP into X12 or EDIFACT. Sterling/SPS handles envelope wrapping, 997 ACK, and retry. Key transactions: 850 PO, 856 ASN, 810 Invoice.
Used At
Nike factory orders
Columbia wholesale EDI
3PL 940/945 transactions
Domain Pattern
Vendor-Managed Inventory (VMI)
Pub-Sub (inverted)
Software Solutions
EDI 852 / 855 SAP SNC Supplier Portal API
Publish-Subscribe inversion: the retailer publishes inventory and POS signals (EDI 852) to the supplier's planning system. The supplier self-generates replenishment POs — reducing buyer workload and improving fill rates.
Used At
Key material suppliers
Packaging VMI programs
Store-in-store partnerships
Domain Pattern
Drop-Ship Integration
Content-Based Router EIP
Software Solutions
EDI 850 / REST OMS routing engine SPS Commerce
Router + Translator pattern: OMS applies drop-ship rules and routes qualifying orders to the vendor via EDI 850 or REST. Vendor returns 856 ASN + tracking; OMS closes the loop. No DC touch — ships direct to customer.
Used At
Nike marketplace extended
Columbia vendor assortment
Long-tail SKU fulfillment

SAP Ariba Coupa Custom REST / GraphQL
Request-Reply API pattern: suppliers consume a portal API to view POs, confirm capacity, upload ASNs, and submit invoices — replacing email and full EDI for SMB suppliers who cannot support Sterling or SPS implementations.
Used At

Capacity confirmation
Invoice submission portal

Resilience & Exception Handling

5 patterns
Domain Pattern
Saga / Compensating Transactions
Saga EIP + Event Sourcing
Software Solutions
Apache Kafka Netflix Conductor Temporal.io AWS Step Functions
Each step in order fulfillment publishes a domain event. On failure, the saga orchestrator (Temporal/Conductor) triggers compensating transactions in reverse: cancel pick → reverse payment → release inventory. No distributed 2PC needed.
Used At
Order fulfillment rollback
Return refund chain
Allocation failure recovery
Domain Pattern
Circuit Breaker
Circuit Breaker Pattern
Software Solutions
Resilience4j Istio Service Mesh AWS App Mesh
When a downstream system (carrier tracking API, WMS) fails repeatedly past a threshold, the circuit opens. Calls are short-circuited with a cached/fallback response — preventing cascade failures across the entire order pipeline.
Used At
Carrier API timeouts
3PL system outages
ERP maintenance windows
Domain Pattern
Idempotent Receiver
Idempotent Consumer EIP
Software Solutions
Kafka consumer groups ERP idempotency key EDI 997 ACK
Each message carries a unique ID (PO number, ASN ID, shipment tracking). The receiver checks a deduplication store (Redis or DB) before processing — guaranteeing exactly-once business semantics despite at-least-once Kafka delivery.
Used At
Duplicate ASN processing
Double PO sends
Payment retry storms
Domain Pattern
Transactional Outbox
Outbox Pattern
Software Solutions
Debezium + Kafka SAP Event Mesh
Instead of dual-writes (DB + broker), the app writes only to a transactional outbox table in the same DB transaction. Debezium CDC tails the outbox and publishes to Kafka — guaranteeing zero lost events even if the broker is temporarily down.
Used At
ERP → WMS order sync
Guaranteed event delivery
Audit trail requirements
Domain Pattern
Dead Letter + Alerting
Dead Letter Channel EIP
Software Solutions
IBM MQ DLQ Kafka Dead Letter Topic PagerDuty / OpsGenie
Failed messages (EDI parse errors, schema mismatches, downstream rejects) route to a Dead Letter Topic. An alerting consumer monitors the DLT and pages the integration ops team with the failed payload and structured error context for triage.
Used At
EDI translation failures
WMS receipt errors
Carrier API rejects
