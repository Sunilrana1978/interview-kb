Building Autonomous Agentic Systems

Welcome to the basic-to-advanced study guide for the Google Antigravity SDK. This interactive tutorial is designed to take you from a complete beginner to writing production-grade, multi-agent workflows in 3 days.

Est. Revision Time: 3 Days (4-6 hrs/day)
Prerequisites: Python 3.10+, Asyncio
Default Model: gemini-3.5-flash

Day 1 Fundamentals & Core Pillars

On your first day, we focus on establishing a solid mental model of the SDK's architecture and starting your first chat interactions.

1. The Three Pillars of Antigravity

The Google Antigravity SDK architecture is built on three core pillars that separate configuration, lifecycle orchestration, and protocol transport:

  • Agent: The primary developer-facing interface. You initialize it using a configuration (e.g., LocalAgentConfig) and it automatically configures tools, loads policies, and provisions required connectors.
  • Conversation: Represents a single, continuous interactive thread. It holds the historical log (or trajectory) of what the agent has said and done, handles context window compaction when limits are reached, and hosts the active chat session.
  • Connection: The physical or logical channel used to send payloads to the Gemini API or a local simulation runtime. By separating Conversation from Connection, the SDK allows developers to switch between running agents locally or hosting them in the cloud without modifying application logic.

2. Installation & Setup

To configure your local environment, install the Google Antigravity package and verify your Gemini credentials:

Terminal 
# 1. Install the SDK
pip install google-antigravity

# 2. Set the API key (Get one from https://aistudio.google.com/app/api-keys)
export GEMINI_API_KEY="your-api-key-here"
ℹ️
API Key Provisioning

If you don't have an API key, get one at Google AI Studio. You can also pass the API key directly in Python code via LocalAgentConfig(api_key="..."), though using environment variables is the recommended practice.

3. Your First Agent: Hello World & Streaming

Because the SDK relies on asynchronous operations, always execute agent flows inside an async/await context. Below are examples for basic chat, token-by-token streaming, and thought-by-thought reasoning:

hello_world.py 
import asyncio
from google.antigravity import Agent, LocalAgentConfig

async def main():
    # Initialize using default local configuration (uses gemini-3.5-flash)
    config = LocalAgentConfig()
    
    async with Agent(config) as agent:
        # 1. Basic Turn-Based Chat
        response = await agent.chat("Hello! What is your default model?")
        print("Agent response:", await response.text())
        
        # 2. Streaming Response
        stream_response = await agent.chat("Write a 1-sentence joke.")
        print("Streaming: ", end="")
        async for token in stream_response:
            print(token, end="", flush=True)
        print()

        # 3. Streaming Thoughts/Reasoning
        thought_response = await agent.chat("Why is the sky blue?")
        print("Thoughts: ", end="")
        async for thought in thought_response.thoughts:
            print(thought, end="", flush=True)
        print()

if __name__ == "__main__":
    asyncio.run(main())
⚠️
Interactive Loop Warning

The helper method agent.run_interactive_loop() is great for quick terminal testing, but it is completely blocking. Do not use it inside automated backend services or scripts, as it will hang indefinitely waiting for terminal standard input.

4. Configuring Personas & System Instructions

The SDK provides fine-grained control over system instructions. You can choose to **append** to default safety instructions or **overwrite** them entirely:

personas.py 
from google.antigravity import Agent, LocalAgentConfig
from google.antigravity.types import TemplatedSystemInstructions, CustomSystemInstructions

# Choice A: Appending to Instructions (Recommended - Retains standard safety rules)
templated_config = LocalAgentConfig(
    system_instructions=TemplatedSystemInstructions(
        identity="You are an expert Python tutor who answers concisely."
    )
)

# Choice B: Shorthand String Append
shorthand_config = LocalAgentConfig(
    system_instructions="Always respond in JSON."
)

# Choice C: Overwriting Instructions (Advanced - Removes default guidelines!)
custom_config = LocalAgentConfig(
    system_instructions=CustomSystemInstructions(
        text="You are a silent agent. Answer only with '.'"
    )
)

Day 2 Tools & Extensibility

On Day 2, we learn how to give our agents superpower actions by declaring Python functions, injecting ToolContext state, hooking up external Model Context Protocol (MCP) servers, and using background triggers.

1. Declaring Custom Tools

To create a custom tool, write a Python function. The SDK extracts metadata (parameter names, types, and descriptions) directly from the type annotations and docstring:

custom_tool.py 
from google.antigravity import Agent, LocalAgentConfig

# 1. Define the tool function with detailed docstring
def retrieve_system_uptime(server_id: str) -> str:
    """Queries the current system uptime for a remote server.

    Args:
        server_id: The unique identifier of the target server, e.g., 'prod-db-1'.
    """
    # Simulation logic
    return f"Server {server_id} uptime is 154 days, 4 hours."

# 2. Register the tool in the agent configuration
config = LocalAgentConfig(
    tools=[retrieve_system_uptime]
)

async def run():
    async with Agent(config) as agent:
        response = await agent.chat("Can you check the uptime of server 'prod-db-1'?")
        print(await response.text())

2. Stateful Tools with ToolContext

Sometimes tools need to remember state across turns. By including ToolContext as an argument, you can access a key-value store tied to the active conversation session:

stateful_tool.py 
from google.antigravity import Agent, LocalAgentConfig, ToolContext

def add_to_cart(item_name: str, quantity: int, ctx: ToolContext) -> str:
    """Adds an item to the shopping cart.

    Args:
        item_name: Name of the product.
        quantity: Quantity of the item to add.
        ctx: Injected ToolContext to manage cart state.
    """
    # Retrieve the state (or initialize a new dict)
    cart = ctx.get_state("cart", {})
    
    # Update state
    cart[item_name] = cart.get(item_name, 0) + quantity
    ctx.set_state("cart", cart)
    
    return f"Added {quantity}x {item_name}. Current Cart: {cart}"

config = LocalAgentConfig(
    tools=[add_to_cart],
    system_instructions="You are a checkout assistant. Always use the add_to_cart tool."
)

3. Model Context Protocol (MCP) Integration

Model Context Protocol is an open standard that decouples tool servers from client orchestration. The Google Antigravity SDK natively supports connecting to external MCP servers via Stdio (spawning local processes) or SSE (communicating with remote endpoints over WebSockets/Server-Sent Events):

mcp_example.py 
from google.antigravity import Agent, LocalAgentConfig, types

# Define external MCP servers
mcp_servers = [
    # 1. Stdio Server (managed locally by launching a command)
    types.McpStdioServer(
        command="python3",
        args=["/path/to/weather_mcp_server.py"]
    ),
    
    # 2. SSE Server (connecting to a remote server on the network)
    types.McpSseServer(
        url="https://api.internal.tools/mcp/sse",
        headers={"Authorization": "Bearer sse-secret-token"}
    )
]

config = LocalAgentConfig(
    mcp_servers=mcp_servers
)

4. Proactive Triggers & Periodic Checks

Triggers allow your agent to be proactive, reacting to background events such as periodic timers or directory modifications, and injecting messages into the agent thread:

triggers.py 
import asyncio
import logging
from google.antigravity import Agent, LocalAgentConfig
from google.antigravity.triggers import every, on_file_change, TriggerContext

# A. Periodic trigger callback
async def health_check(ctx: TriggerContext):
    # Pushes a status update to the active agent conversation
    await ctx.send("[System Trigger]: Active database connection count is high.")

timer_trigger = every(interval_seconds=300, callback=health_check)

# B. File Watcher trigger callback
async def config_watch(ctx: TriggerContext, changes):
    for change in changes:
        logging.info(f"File {change.path} was {change.kind}")
    await ctx.send("[System Trigger]: configuration file modified.")

# Watches directory for file edits (requires 'watchfiles' module)
file_trigger = on_file_change("/tmp/configs", config_watch)

config = LocalAgentConfig(
    triggers=[timer_trigger, file_trigger]
)

Day 3 Advanced Architectures & Production

On Day 3, we dive into production orchestration: routing tasks to nested subagents, intercepting execution with hooks, specifying safety predicates, debugging, tracking token costs, and persisting conversations across restarts.

1. Multi-Agent Delegation (Subagents)

To prevent context bloat and separation-of-concerns issues, you can configure an agent to delegate tasks to subagents. Enable this capability in your config:

subagents.py 
from google.antigravity import Agent, LocalAgentConfig, types

config = LocalAgentConfig(
    capabilities=types.CapabilitiesConfig(
        enable_subagents=True  # Exposes subagent tools to the primary agent
    )
)

async def run():
    async with Agent(config) as agent:
        # The agent will dynamically spawn and coordinate with subagents to execute this task
        response = await agent.chat("Create a subagent to write a python test script.")
        print(await response.text())

2. Lifecycle Hooks & Error Recovery

Hooks intercept events along the agent's turn timeline. You can observe events or dynamically recover from errors:

hooks_example.py 
from typing import Any, Optional
from google.antigravity.hooks import hooks, HookContext
from google.antigravity import LocalAgentConfig

# Hook 1: Pre-turn prompt audit
@hooks.pre_turn
async def pre_turn_hook(data: str):
    print(f"[HOOK] Processing incoming prompt: {data}")
    return types.HookResult(allow=True)

# Hook 2: Custom fallback error recovery handler
class SafeFallbackHook(hooks.OnToolErrorHook):
    async def run(self, context: HookContext, data: Any) -> Optional[str]:
        if isinstance(data, ValueError):
            # Feed a graceful message back to the LLM instead of crashing
            return "[Notice: Operation failed. Please retry with a valid positive integer.]"
        return None  # Let other exceptions raise normally

config = LocalAgentConfig(
    hooks=[pre_turn_hook, SafeFallbackHook()]
)

3. Safety Policies & Execution Guardrails

By default, the SDK uses the confirm_run_command() policy: denying shell execution via run_command while allowing other actions. You can customize policies using rules and predicates:

safety_policies.py 
from google.antigravity.hooks import policy
from google.antigravity import LocalAgentConfig

# Define specific safety rules
policies = [
    # 1. Deny execution of command lines containing 'rm -rf'
    policy.deny(
        "run_command",
        when=lambda args: "rm -rf" in args.get("CommandLine", ""),
        name="block_dangerous_deletes"
    ),
    
    # 2. Block tool execution outside of explicit project folders
    policy.workspace_only(workspaces=["/Users/sunilkumar/project/src"]),
    
    # 3. Allow all other actions
    policy.allow_all()
]

config = LocalAgentConfig(policies=policies)
☣️
Policy Evaluation Order

Policies resolve strictly in order: Specific Deny ➔ Specific Ask ➔ Specific Allow ➔ Wildcard Deny ➔ Wildcard Ask ➔ Wildcard Allow. First match short-circuits. If an argument check predicate crashes, the policy fails closed and triggers a default denial.

4. Observability: Token Costs & Storage Layouts

Keep track of usage metadata across conversations. When using reasoning models, ensure you monitor thoughts_token_count as reasoning chains can consume significant token amounts:

observability.py 
from google.antigravity import Agent, LocalAgentConfig
import logging

# A. Setup SDK-wide logging outputs
logging.basicConfig(level=logging.INFO)
logging.getLogger("google.antigravity").setLevel(logging.INFO)

# B. Track token counts post-interaction
async def track_tokens():
    config = LocalAgentConfig(
        app_data_dir="/Users/sunilkumar/project/brain" # MUST be an absolute path
    )
    async with Agent(config) as agent:
        response = await agent.chat("Solve this math problem: 15482 * 9482")
        
        usage = agent.conversation.total_usage
        print(f"Prompt Tokens: {usage.prompt_token_count}")
        print(f"Reasoning Tokens: {usage.thoughts_token_count}")
        print(f"Answer Tokens: {usage.candidates_token_count}")
        print(f"Cumulative Total: {usage.total_token_count}")

5. Persistent Sessions

By defining a save_dir, the SDK automatically pickles or saves the conversation trajectory on disk. Retrieve the conversation_id to seamlessly pick up right where the session left off:

persistence.py 
from google.antigravity import Agent, LocalAgentConfig

# --- Session 1: Start and Save ---
config1 = LocalAgentConfig(save_dir="/Users/sunilkumar/project/saves")
async with Agent(config1) as agent:
    await agent.chat("Secret codeword is Antigravity")
    saved_id = agent.conversation_id

# --- Session 2: Load and Resume ---
config2 = LocalAgentConfig(
    conversation_id=saved_id,
    save_dir="/Users/sunilkumar/project/saves"
)
async with Agent(config2) as agent:
    response = await agent.chat("What was the secret codeword?")
    print(await response.text())  # Outputs: "Secret codeword is Antigravity"
