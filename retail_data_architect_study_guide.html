Two gaps to close before the interview
1. Erwin hands-on practice: Download the trial version and reverse-engineer a sample schema. Interviewers will probe logical vs. physical model creation, IDEF1X notation, and how you document JSON structures. Even 2–3 hours of hands-on time will significantly improve your confidence.

2. AlloyDB vs Cloud SQL distinction: Most candidates conflate these. Knowing that AlloyDB has a built-in columnar engine for HTAP (hybrid transactional + analytical) workloads, and is ~4× faster than Cloud SQL on writes, is a genuine differentiator.

Data modeling — OLTP, OLAP & Erwin
OLTP design principles
3NF normalization to eliminate redundancy. Strong primary/foreign key enforcement. Row-level locking for concurrent writes. Narrow, focused tables. Indexes on FK and high-cardinality filter columns. Short, fast transactions. ACID compliance is non-negotiable. Avoid SELECT * in queries; use parameterized queries.
OLAP design principles
Star schema (fact + dimensions) or snowflake. Denormalized for read speed. Columnar storage (BigQuery, AlloyDB columnar). Partition by date dimension. Cluster by frequently-filtered dims. Pre-aggregate rollup tables. Slowly Changing Dimensions (SCD Type 1/2/4). Surrogate keys on all dims.
Erwin data modeling — key workflow
Logical model: Entity-relationship, platform-agnostic, business semantics (Products, Orders, Customers). Use IDEF1X notation. Identify all entities, attributes, cardinality, and relationships. This is your communication tool with business stakeholders.

Physical model: Forward-engineer to target DB (Cloud SQL/AlloyDB/BigQuery). Assign data types, constraints, indexes, partitions, tablespaces. One physical model per target platform.

Reverse engineering: Connect Erwin to live DB2 schema → auto-generate the logical model → annotate with business meaning → design future-state physical model as a separate target.

JSON / complex types in Erwin: Model JSON columns as BLOB/TEXT or CLOB logically; document the expected JSON schema structure in the entity's data dictionary notes. In BigQuery physical model, map to STRUCT/ARRAY types.
JSON & complex data structures in retail
Product attributes (color, size, material, care instructions) are semi-structured and vary by product category — ideal for JSONB. Pattern: use typed columns for universally-queried attributes (sku_id, color_code, size_code) and a attributes JSONB overflow column for category-specific extras.

In AlloyDB/PostgreSQL: index with GIN index on attributes for fast JSONB path searches. Use GENERATED ALWAYS AS columns to extract frequently-queried JSON paths as real columns.

In BigQuery: use STRUCT<color STRING, size STRING, material STRING> for known schemas; JSON type for truly flexible attributes. Always enforce JSON Schema validation at the application layer — fully schemaless leads to data quality issues at scale.

SCD (Slowly Changing Dimensions) — retail pricing example
SCD Type 1: Overwrite — use for non-critical corrections (spelling fix on product name).
SCD Type 2: Add new row with effective_date + expiry_date + is_current flag — use for price history, customer address, product category reassignment.
SCD Type 4: Mini-dimension — extract fast-changing attributes (price, promo flag) to a separate table; join at query time. Used when SCD2 creates too many rows.

Retail pricing table pattern (SCD2):
DIM_PRICE (price_key, product_key, price_type, price_amount, currency, effective_date, expiry_date, is_current)

This enables point-in-time queries: "What was the price of SKU-1234 on Black Friday 2023?" — critical for receipt validation and return processing.

Retail workflows and their data dependencies
Merchandise planning: Assortment plans → OTB (Open to Buy) → inventory targets. Data needed: sales history, statistical forecast, margin targets, supplier lead times.

Omnichannel order routing (DOM): Distributed Order Management routes to store/DC/vendor-direct based on real-time ATP, SLA, and cost. Requires sub-second inventory visibility across all nodes.

Markdown/promotion management: Price ladders with effective-dated tables. Compliance reporting requires full price history (many states mandate "original price" accuracy). Promo rules engine needs to know overlapping promos and priority.

Supply chain (inbound): PO → ASN (856 EDI) → GR → putaway → bin location. Triggers replenishment when stock falls below reorder point. Costing flows from factory PO through landed cost to retail margin.

Returns/reverse logistics: RMA → inspection → disposition (restock / refurbish / liquidate / destroy). Inventory adjustment creates OLTP write + OLAP event.

Available-to-Promise (ATP) — key concept for interviews
ATP = On-hand stock − Reserved stock − Safety stock

Calculating real-time ATP at SKU × Location level is one of the hardest data problems in retail. Requires:
— Sub-second write throughput for stock reservation on order placement
— Distributed locking or optimistic concurrency to prevent oversell
— Event-sourced inventory log to reconstruct any point-in-time balance
— CQRS: write model in AlloyDB (strong consistency), read model in Redis (materialized ATP cache) + BigQuery (historical analysis)

Domain Driven Design — key concepts for this role
Bounded contextExplicit boundary where a specific domain model applies (Order, Inventory, Catalog, Pricing)
Ubiquitous languageShared vocabulary between developers and business within a context. "Product" means something different in Catalog vs. OMS vs. PLM.
AggregateCluster of entities with a single root (Order + OrderLines + OrderPayment); the consistency boundary for writes
Context mapDiagram showing how bounded contexts relate: shared kernel, customer-supplier, conformist, anti-corruption layer (ACL)
Anti-corruption layerTranslation adapter between legacy DB2 model and new domain model; prevents legacy schema from "leaking" into new design
In data modelingSchema per bounded context; avoid cross-context foreign keys; use domain events (not shared tables) for integration
Retail DDD bounded contexts — example mapping
Catalog (Product display, search) Inventory (ATP, stock levels) Order Management (OMS) Pricing (base, promo, member) Customer (identity, loyalty) Fulfillment (pick/pack/ship) PLM (design, BOM, costing) Finance (GL, AP, AR)

Each context owns its "Product" entity with only the attributes it cares about. Integration happens via domain events (ProductCreated, StockChanged, OrderPlaced), not shared tables. This is the key architectural shift from the legacy DB2 monolith.

Legacy modernization & re-architecture
Reverse engineering legacy data models — 5-step approach
Step 1 — Discovery: Use Erwin reverse-engineer against live DB2. Capture all tables, columns, FKs, indexes, stored procs, views, triggers. Export to report for stakeholder review.

Step 2 — Current-state analysis: Identify data quality issues (nulls where NOT NULL expected), missing constraints, redundant data, and implicit business rules buried in stored procs or app code. Document technical debt.

Step 3 — Domain mapping: Group tables into bounded contexts. Identify aggregates (which tables form a consistency unit?). Surface implicit domain events hidden as status columns and timestamp audit fields.

Step 4 — Future-state design: Create target logical + physical models per bounded context in Erwin. Define canonical data model for cross-context integration. Document every significant decision in Architecture Decision Records (ADRs).

Step 5 — Migration strategy: Choose approach per context: dual-write (safest), CDC via Datastream (low risk, near real-time), or big-bang (risky, only for non-critical systems). Validate with shadow-mode reads comparing old vs. new query results.
Migration patterns — know these
Strangler figGradually replace legacy features one by one; route traffic to new service as each feature completes. Zero downtime migration.
Anti-corruption layer (ACL)Translation adapter between old and new model. Prevents legacy "leaking" into new design during transition period.
Event sourcingCapture all state changes as immutable events. Replay to rebuild any projection. Enables full audit trail.
CQRSSeparate read model (BigQuery / Redis) from write model (AlloyDB). Optimizes each independently.
CDC / Datastream (GCP)Capture DB2 change data → GCP Pub/Sub → Dataflow → BigQuery. Near real-time sync without impacting source DB.
Schema migrationFlyway or Liquibase for versioned, auditable, team-coordinated schema changes. Never ad-hoc DDL in prod.
Dual-writeApplication writes to both old and new DB simultaneously during transition. Compare results. Cut over when confident.
Web application data model performance — key techniques
Read-heavy APIs (PDP, search)Materialized views in AlloyDB; read replicas; Memorystore (Redis) cache; CDN for static product data
Write-heavy (cart, inventory)Optimistic locking; connection pooling via PgBouncer; AlloyDB write scaling; idempotent writes with client-generated IDs
JSON payloads (product attrs)JSONB with GIN index; partial indexes on common JSON paths; generated columns for hot paths
Pagination (large catalogs)Keyset pagination (WHERE id > last_id) instead of OFFSET; OFFSET degrades at large page numbers
Connection managementAlloyDB Auth Proxy; max_connections tuning; separate pools for OLTP reads vs. writes
Reporting queriesOffload to BigQuery or AlloyDB columnar reads; never run aggregation queries on primary OLTP DB
