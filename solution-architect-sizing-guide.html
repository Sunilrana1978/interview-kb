01 · Foundations

Core Sizing Principles

Before touching a sizing spreadsheet, a Solution Architect must ground the exercise in three universal laws that govern all distributed systems.

⚖️

Little's Law

The average number of items in a system equals the arrival rate multiplied by the average time spent. Foundational for queue-depth and concurrency calculations.

📐

Amdahl's Law

Theoretical speedup is limited by the sequential portion of the workload. Guides parallelism decisions — there are diminishing returns beyond a certain scale.

🌊

Universal Scalability Law

Extends Amdahl's to account for coherency penalties (crosstalk between nodes). Critical for predicting performance degradation beyond the linear scaling regime.

// Little's Law — N = λ × W
N (concurrent requests) = Arrival Rate (λ) × Average Response Time (W)

// For 10K RPS @ 50ms avg latency:
N = 10,000 req/s × 0.050 s = 500 concurrent requests in-flight

Step 3 — Kafka Partition Sizing

Partitions are the unit of parallelism. Under-partitioning is the #1 cause of EDA bottlenecks.

// Rule of thumb: 1 partition ≈ 10–30 MB/s throughput
Min partitions = Design throughput ÷ Per-partition throughput
Min partitions = 25,000 msg/s ÷ 1,000 msg/s/partition = 25 partitions

// Round to power of 2 for even key distribution
Recommended = 32 partitions (with 3 brokers → ~11 partitions/broker)

// Consumer group sizing: consumers ≤ partitions
Consumer pods = 32 (one per partition for max parallelism)

Step 4 — Consumer Sizing

// Each consumer must process its share
Per-consumer rate = 10,000 ÷ 32 = 312.5 msg/s
Per-consumer with safety = 312.5 × 2.5 = ~781 msg/s per pod

// Processing time budget (p99 target = 200ms end-to-end)
Consumer processing time budget ≤ 80ms (40% of budget)
Broker propagation budget ≤ 20ms
Downstream write budget ≤ 50ms
Network + serialization ≤ 50ms

04 · Latency Engineering

p99 Latency Budget Decomposition

A p99 latency target means 99 out of every 100 requests must complete within the budget. Achieving this requires explicitly budgeting every network hop, compute stage, and I/O operation — and designing for tail latency, not averages.

p99 vs p50 — Why It Matters At 10,000 RPS, a p99 of 200ms means 100 requests per second are slow. At 10K RPS with a 1-second SLA breach window, 1% tail latency is 86,400 slow requests per day. Size for the tail, not the median.

Tail Latency Mitigation Techniques

🔁

Hedged Requests

Send a duplicate request to a second replica after a short delay (e.g., p90 latency). Return whichever responds first. Reduces p99 at the cost of ~2× downstream load.

🎛️

Backpressure + Rate Limiting

Use token bucket or leaky bucket algorithms at the ingress to prevent consumer overload and queue depth blow-out during traffic spikes.

🏃

Connection Pooling

Pre-warm DB and broker connections. At 10K RPS, TCP handshake overhead per request is fatal. PgBouncer (transaction mode) dramatically reduces DB latency tail.

📦

Micro-batching

Kafka consumers can batch 100–1000 messages per poll cycle, amortizing deserialization and DB round-trip costs. Trade slight latency for 10–50× throughput gain.

06 · Estimation Methods in Software Development

The Full Estimation Toolkit

Size estimation extends beyond infrastructure. A complete Solution Architect must command all major effort, cost, and capacity estimation techniques across the SDLC.

🎯

Story Points & Planning Poker

Effort Sizing

Relative complexity scoring (1, 2, 3, 5, 8, 13, 21) using Fibonacci sequence. Teams vote simultaneously to avoid anchoring bias. Best for sprint-level planning in Agile. Velocity (story points/sprint) is tracked to forecast delivery timelines.

📊

Function Point Analysis (FPA)

Functional Sizing

IFPUG / COSMIC standard. Counts External Inputs, Outputs, Inquiries, Internal Logical Files, and External Interface Files. Each is weighted and summed into Unadjusted Function Points, then multiplied by a Value Adjustment Factor. Language-agnostic and contractually defensible.

📏

COCOMO II

Algorithmic Estimation

Constructive Cost Model. Effort = A × Size^B × ∏EM. Uses KSLOC as input with 22 cost drivers (reliability, complexity, team capability). Highly calibrated for large, historically-informed projects. Available in Basic, Intermediate, and Detailed modes.

🎲

Three-Point (PERT) Estimation

Probabilistic

Uses Optimistic (O), Most Likely (M), and Pessimistic (P) estimates. PERT formula: E = (O + 4M + P) / 6. Standard deviation = (P-O)/6. Forces explicit uncertainty modeling. Ideal for risk-heavy or novel work.

🔢

T-Shirt Sizing

Lightweight

XS / S / M / L / XL / XXL buckets used in PI planning and portfolio backlogs. Deliberately imprecise — optimizes for speed over accuracy. Translates to rough time ranges (S = 1–3 sprints; L = 1–2 quarters). Calibrate buckets to team-specific historical data.

🧠

Expert Judgment / Delphi

Qualitative

Iterative anonymous expert polling until consensus converges. Eliminates social bias and HiPPO (Highest Paid Person's Opinion) effects. Best when historical data is scarce and the domain is highly specialized. Combine with quantitative methods for calibration.

📈

Analogous (Parametric) Estimation

Historical

Scale estimates from a reference project using adjustment factors. Requires a maintained project data repository. Formula: Estimate = Ref × (New Size / Ref Size)^α. Accuracy improves dramatically with calibrated organizational data.

⚡

Back-of-Envelope / Fermi

Quick Sanity Check

Order-of-magnitude reasoning used in system design interviews and early architecture discussions. Start with knowns (RPS, data size), derive unknowns via chained approximations. Accuracy within 1 order of magnitude is sufficient for architectural decisions. Essential skill for SA interviews.

07 · SA Interview Readiness

Sizing & Estimation Checklist

Use this checklist when approaching any system design or capacity planning problem in interviews or production planning sessions.

Requirements Gathering

  • Clarify peak vs. average RPS (never assume they're equal)
  • Establish latency targets per percentile (p50, p95, p99)
  • Define consistency requirements (strong vs. eventual)
  • Identify data retention and compliance constraints
  • Determine geographic distribution (single vs. multi-region)
  • Understand read:write ratio (drives storage & caching strategy)

Capacity Calculations

  • Apply 2–3× safety multiplier on all peak figures
  • Calculate storage growth rate (monthly, annual)
  • Compute bandwidth at every tier (inbound + replication)
  • Size connection pools to prevent thread exhaustion
  • Partition count must ≥ max desired consumer parallelism
  • Validate with Little's Law (N = λ × W) at each stage

Latency Engineering

  • Decompose end-to-end budget into per-hop allocations
  • Design for p99 tail latency, not median
  • Account for GC pauses (JVM consumers; use G1GC or ZGC)
  • Enable async I/O wherever possible (non-blocking consumers)
  • Pre-warm connections; avoid cold-start penalties under load
  • Use hedged requests for critical paths (read operations)

Resilience & Observability

  • Plan for broker failure: RF=3, min.insync.replicas=2
  • Define DLQ (Dead Letter Queue) strategy for poison pills
  • Set up consumer lag alerting (KEDA / CloudWatch alarms)
  • Establish SLO dashboards: error rate, latency, throughput
  • Run chaos engineering drills before production traffic
  • Document runbooks for consumer lag > threshold events
Interview Pro Tip In SA interviews, always show your arithmetic. State assumptions first ("I'll assume 1KB avg event size, 80/20 read-write split"), then derive bottom-up. Interviewers reward structured reasoning over memorized answers. Use Back-of-Envelope + Little's Law as your default starting point, then refine.
