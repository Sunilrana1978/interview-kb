02   Database Design

Per-Service Schema

Each service owns its own PostgreSQL database. The saga_id UUID is the correlation key across all services.

📦 orders order-db
PK order_idUUID
FK customer_idUUID
IDX saga_idUUID
statusENUM
total_amountDECIMAL(10,2)
itemsJSONB
created_atTIMESTAMPTZ
updated_atTIMESTAMPTZ
status: PENDING → CONFIRMED | CANCELLED | COMPLETED
🏭 inventory_reservations inv-db
PK reservation_idUUID
FK order_idUUID
IDX saga_idUUID
product_idUUID
quantityINTEGER
warehouse_idUUID
statusENUM
reserved_atTIMESTAMPTZ
status: RESERVED → CONFIRMED | RELEASED (compensation)
💳 payment_transactions pay-db
PK transaction_idUUID
FK order_idUUID
IDX saga_idUUID
amountDECIMAL(10,2)
currencyCHAR(3)
statusENUM
gateway_refVARCHAR(255)
processed_atTIMESTAMPTZ
status: AUTHORIZED → CAPTURED | REFUNDED (compensation) | FAILED
🚚 shipments ship-db
PK shipment_idUUID
FK order_idUUID
IDX saga_idUUID
carrierVARCHAR(100)
tracking_numberVARCHAR(255)
statusENUM
destinationJSONB
scheduled_atTIMESTAMPTZ
status: SCHEDULED → DISPATCHED → DELIVERED | RETURNED (compensation)
📋 saga_orchestration_log orchestrator-db · central saga state
PK log_idUUID
IDX saga_idUUID — correlation key
saga_typeVARCHAR(100)
current_stepVARCHAR(100)
statusENUM(IN_PROGRESS, COMPLETED, COMPENSATING, FAILED)
payloadJSONB — original request
compensationsJSONB — undo log stack
created_at / updated_atTIMESTAMPTZ
⚑ The compensations JSONB column acts as an undo-log stack. Each completed step appends its inverse operation so the orchestrator can roll back in reverse order on failure.

04   Kafka Event Catalog

Topics & Events

All events carry saga_id and order_id as mandatory headers. Partitioned by order_id to guarantee ordering per order.

Topic Event Type Publisher Consumer Type Key Payload Fields
order.commands ORDER_CREATE_REQUESTED API Gateway Order Service COMMAND customer_id, items[], total_amount
order.events ORDER_CREATED Order Service Orchestrator EVENT order_id, saga_id, status=PENDING
inventory.commands INVENTORY_RESERVE_REQUESTED Orchestrator Inventory Svc COMMAND order_id, saga_id, items[], warehouse_id
inventory.events INVENTORY_RESERVED Inventory Svc Orchestrator EVENT reservation_id, saga_id, status=RESERVED
payment.commands PAYMENT_PROCESS_REQUESTED Orchestrator Payment Svc COMMAND order_id, saga_id, amount, currency, payment_method
payment.events PAYMENT_AUTHORIZED Payment Svc Orchestrator EVENT transaction_id, saga_id, gateway_ref, status=AUTHORIZED
payment.events PAYMENT_FAILED Payment Svc Orchestrator FAILURE saga_id, reason, error_code → triggers compensation
shipment.commands SHIPMENT_SCHEDULE_REQUESTED Orchestrator Shipping Svc COMMAND order_id, saga_id, destination, items[], carrier
shipment.events SHIPMENT_SCHEDULED Shipping Svc Orchestrator EVENT shipment_id, saga_id, tracking_number, status=SCHEDULED
inventory.commands INVENTORY_RELEASE_REQUESTED Orchestrator Inventory Svc COMPENSATE saga_id, reservation_id → SET status=RELEASED
order.commands ORDER_CANCEL_REQUESTED Orchestrator Order Service COMPENSATE saga_id, order_id, reason → SET status=CANCELLED
saga.dead-letter COMPENSATION_FAILED Orchestrator Ops / Alerting DLQ saga_id, failed_step, retry_count, original_payload

Topic Config

partitions: 12 (by order_id hash)
replication: 3
retention: 7 days
delivery: at-least-once
acks: all (strongest guarantee)

Idempotency

Each consumer checks saga_id in its own DB before processing. Duplicate messages → no-op. Uses INSERT … ON CONFLICT DO NOTHING with saga_id as unique index. Safe for Kafka retries.

Dead Letter Queue

Failed compensations after 3 retries go to saga.dead-letter. Ops team is alerted. Manual intervention or automated saga recovery process reconciles state.
