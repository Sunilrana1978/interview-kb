1
Account Setup & Structure
Create databases, schemas, and virtual warehouses
▼
Create Data Lake Database
Set up the raw landing zone for all incoming data — this is your Data Lake inside Snowflake.
CREATE DATABASE DATA_LAKE;

CREATE SCHEMA DATA_LAKE.RAW;       -- immutable landing zone
CREATE SCHEMA DATA_LAKE.STAGING;   -- light cleansing layer
Create Data Warehouse Database
A separate database for structured, modeled data with staging, core, and mart layers.
CREATE DATABASE DATA_WAREHOUSE;

CREATE SCHEMA DATA_WAREHOUSE.STAGING;   -- typed & validated
CREATE SCHEMA DATA_WAREHOUSE.CORE;      -- dims + facts
CREATE SCHEMA DATA_WAREHOUSE.MARTS;     -- business-ready views
Create Virtual Warehouses (Compute)
Separate compute clusters for loading vs. analytics to avoid resource contention.
CREATE WAREHOUSE LOAD_WH
  WAREHOUSE_SIZE = 'MEDIUM'
  AUTO_SUSPEND   = 60
  AUTO_RESUME    = TRUE
  COMMENT        = 'ETL loading warehouse';

CREATE WAREHOUSE ANALYTICS_WH
  WAREHOUSE_SIZE = 'SMALL'
  AUTO_SUSPEND   = 300
  AUTO_RESUME    = TRUE
  COMMENT        = 'BI and analyst queries';
2
External Stage & Storage Integration
Connect Snowflake to cloud storage (S3 / ADLS / GCS)
▼
Create Storage Integration
Delegate cloud storage access to Snowflake securely (done once by ACCOUNTADMIN).
CREATE STORAGE INTEGRATION s3_integration
  TYPE                      = EXTERNAL_STAGE
  STORAGE_PROVIDER          = 'S3'
  ENABLED                   = TRUE
  STORAGE_ALLOWED_LOCATIONS = ('s3://your-bucket/');

-- Get the IAM values to configure your S3 bucket policy:
DESC INTEGRATION s3_integration;
Create External Stage
Point Snowflake at the cloud bucket where your raw files land.
CREATE STAGE DATA_LAKE.RAW.s3_raw_stage
  STORAGE_INTEGRATION = s3_integration
  URL         = 's3://your-bucket/raw-data/'
  FILE_FORMAT = (TYPE = 'PARQUET');  -- or JSON, CSV, AVRO

-- List files in the stage
LIST @DATA_LAKE.RAW.s3_raw_stage;
3
Raw Landing Tables (Data Lake)
Schema-on-read ingestion using VARIANT type
▼
Create Raw Table
Use the VARIANT column to store semi-structured data (JSON, XML, Avro) without defining a schema upfront.
CREATE TABLE DATA_LAKE.RAW.orders_raw (
  raw_data  VARIANT,
  file_name VARCHAR,
  loaded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
);
Bulk Load with COPY INTO
Load files from stage into the raw table in one command.
COPY INTO DATA_LAKE.RAW.orders_raw (raw_data, file_name)
FROM (
  SELECT $1, METADATA$FILENAME
  FROM @DATA_LAKE.RAW.s3_raw_stage/orders/
)
FILE_FORMAT = (TYPE = 'JSON')
ON_ERROR    = 'CONTINUE';
Setup Snowpipe (Continuous Ingestion)
Automate ingestion whenever new files arrive in the bucket using event notifications.
CREATE PIPE DATA_LAKE.RAW.orders_pipe
  AUTO_INGEST = TRUE
AS
  COPY INTO DATA_LAKE.RAW.orders_raw
  FROM @DATA_LAKE.RAW.s3_raw_stage/orders/;

-- Get SQS ARN to configure S3 event notifications:
SHOW PIPES LIKE 'orders_pipe';
4
Lake Staging Layer
Flatten and lightly cleanse VARIANT data
▼
Flatten JSON / VARIANT Data
Extract individual fields from VARIANT using colon notation. Create a view or table for reuse.
CREATE OR REPLACE TABLE DATA_LAKE.STAGING.stg_orders_flat AS
SELECT
  raw_data:order_id::VARCHAR     AS order_id,
  raw_data:customer_id::VARCHAR  AS customer_id,
  raw_data:order_date::DATE      AS order_date,
  raw_data:amount::FLOAT         AS order_amount,
  raw_data:status::VARCHAR       AS order_status,
  file_name,
  loaded_at
FROM DATA_LAKE.RAW.orders_raw
WHERE raw_data:order_id IS NOT NULL;   -- basic quality filter
5
DW Staging Layer
Typed, validated, and business-rule-applied data
▼
Create DW Staging Table
Pull from the lake staging layer, enforce strict types, and apply business cleansing logic.
CREATE OR REPLACE TABLE DATA_WAREHOUSE.STAGING.stg_orders AS
SELECT
  order_id,
  customer_id,
  TRY_TO_DATE(order_date)          AS order_date,
  COALESCE(order_amount, 0)        AS order_amount,
  UPPER(TRIM(order_status))        AS order_status,
  CURRENT_TIMESTAMP()             AS dw_loaded_at
FROM DATA_LAKE.STAGING.stg_orders_flat
WHERE order_amount > 0            -- data quality rule
  AND order_date >= '2020-01-01';
6
Core Layer — Dimension Tables
Build slowly changing dimensions (SCD Type 2)
▼
Create Dimension Tables
Dimensions describe the "who, what, where" context of your facts. Use surrogate keys for joins.
CREATE TABLE DATA_WAREHOUSE.CORE.dim_customer (
  customer_sk   INT AUTOINCREMENT PRIMARY KEY,  -- surrogate key
  customer_id   VARCHAR NOT NULL,                -- natural key
  customer_name VARCHAR,
  email         VARCHAR,
  country       VARCHAR,
  segment       VARCHAR,
  -- SCD Type 2 tracking columns
  valid_from    DATE    DEFAULT CURRENT_DATE(),
  valid_to      DATE    DEFAULT '9999-12-31',
  is_current    BOOLEAN DEFAULT TRUE
);

CREATE TABLE DATA_WAREHOUSE.CORE.dim_date (
  date_sk    INT PRIMARY KEY,
  full_date  DATE,
  year       INT,
  quarter    INT,
  month      INT,
  month_name VARCHAR,
  week       INT,
  day_name   VARCHAR,
  is_weekend BOOLEAN
);
7
Core Layer — Fact Tables
Build the central fact table with foreign keys to dimensions
▼
Create Fact Table
Facts store measurable events (sales, clicks, orders). They reference dimension surrogate keys.
CREATE TABLE DATA_WAREHOUSE.CORE.fact_sales (
  sales_sk     INT AUTOINCREMENT PRIMARY KEY,
  order_id     VARCHAR,
  customer_sk  INT REFERENCES dim_customer(customer_sk),
  product_sk   INT REFERENCES dim_product(product_sk),
  date_sk      INT REFERENCES dim_date(date_sk),
  order_amount FLOAT,
  quantity     INT,
  discount_pct FLOAT,
  net_amount   FLOAT,
  loaded_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP()
);

-- Load fact from staging
INSERT INTO DATA_WAREHOUSE.CORE.fact_sales
SELECT
  s.order_id,
  c.customer_sk,
  p.product_sk,
  d.date_sk,
  s.order_amount,
  s.quantity,
  s.discount_pct,
  s.order_amount * (1 - s.discount_pct) AS net_amount
FROM DATA_WAREHOUSE.STAGING.stg_orders s
JOIN DATA_WAREHOUSE.CORE.dim_customer c ON s.customer_id = c.customer_id AND c.is_current
JOIN DATA_WAREHOUSE.CORE.dim_date     d ON s.order_date  = d.full_date;
8
Data Marts & BI Layer
Business-ready aggregated views for analysts
▼
Create Data Mart Views
Expose clean, pre-joined, aggregated views for specific business domains like Sales, Finance, HR.
CREATE OR REPLACE VIEW DATA_WAREHOUSE.MARTS.sales_summary AS
SELECT
  d.year,
  d.month_name,
  c.country,
  c.segment,
  SUM(f.net_amount)     AS total_revenue,
  COUNT(f.order_id)     AS total_orders,
  AVG(f.net_amount)     AS avg_order_value,
  SUM(f.quantity)       AS units_sold
FROM   DATA_WAREHOUSE.CORE.fact_sales     f
JOIN   DATA_WAREHOUSE.CORE.dim_customer   c ON f.customer_sk = c.customer_sk
JOIN   DATA_WAREHOUSE.CORE.dim_date       d ON f.date_sk    = d.date_sk
GROUP BY 1,2,3,4;

Security, Roles & Governance
RBAC, data masking, row-level security
▼
Role-Based Access Control
CREATE ROLE DATA_ENGINEER;
CREATE ROLE DATA_ANALYST;
CREATE ROLE BUSINESS_USER;

GRANT USAGE ON DATABASE DATA_LAKE      TO ROLE DATA_ENGINEER;
GRANT USAGE ON DATABASE DATA_WAREHOUSE TO ROLE DATA_ENGINEER;
GRANT SELECT ON ALL TABLES IN SCHEMA DATA_WAREHOUSE.CORE
  TO ROLE DATA_ANALYST;
GRANT SELECT ON ALL VIEWS IN SCHEMA DATA_WAREHOUSE.MARTS
  TO ROLE BUSINESS_USER;
Dynamic Data Masking
-- Mask email for non-engineer roles
CREATE MASKING POLICY mask_email
  AS (val STRING) RETURNS STRING ->
    CASE WHEN CURRENT_ROLE() = 'DATA_ENGINEER'
    THEN val
    ELSE REGEXP_REPLACE(val, '.+@', '***@')
  END;

ALTER TABLE dim_customer
  MODIFY COLUMN email
  SET MASKING POLICY mask_email;
🗄️
Data Lake vs Data Warehouse
Data Lake stores raw, unstructured data at any scale with schema-on-read. Data Warehouse stores structured, modeled data optimized for querying with schema-on-write.
📦
VARIANT Type
Snowflake's native semi-structured type. Stores JSON, XML, Avro as-is. Query sub-fields with colon notation: raw_data:field::VARCHAR.
⭐
Star Schema
Central fact table surrounded by dimension tables. Optimized for analytical queries. One join per dimension makes queries fast and intuitive.
🔄
Snowpipe
Continuous, event-driven data ingestion. Triggers automatically when new files land in cloud storage. Serverless — no warehouse needed for loading.
📅
SCD Type 2
Slowly Changing Dimension tracking. Preserves historical values by adding new rows with valid_from, valid_to, and is_current columns for each change.
🏭
Virtual Warehouses
Compute clusters independent from storage. Scale up/down instantly, auto-suspend when idle, and auto-resume on query — pay only for what you use.
🔐
RBAC Security
Role-Based Access Control. Grant privileges to roles, assign roles to users. Combine with Dynamic Data Masking and Row Access Policies for fine-grained control.
📊
Data Mart
Subset of the data warehouse tailored for a specific business domain (Sales, Finance, HR). Typically implemented as views or materialized tables for performance.
🛠️
dbt Integration
dbt (data build tool) is the industry standard for transformations inside Snowflake. Write SQL models, auto-generate lineage docs, and run tests on every deploy.
