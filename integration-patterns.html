Section 01 — Synchronous Integration

API Communication Styles

Choosing the right API protocol shapes latency, client flexibility, and operational complexity. Each style optimises for different constraints.

1.1 REST (Representational State Transfer)

REST remains the dominant style for public-facing APIs. Resources are identified by URLs, HTTP verbs convey intent, and statelessness enables horizontal scaling. It is a constraint set, not a protocol, making it implementation-flexible.

CLIENT API SERVER DATABASE │ │ │ │── GET /orders/42 ──────────────►│ │ │ │── SELECT * FROM orders ────►│ │ │◄── {id:42, status:"paid"} ──│ │◄── 200 OK {order} ─────────────│ │ │ │ │ │── POST /orders ────────────────►│ │ │ {product_id:7, qty:2} │── INSERT INTO orders... ───►│ │ │◄── new id=99 ───────────────│ │◄── 201 Created {id:99} ────────│ │
REST Resource Controller — Node.js / Express
// Resource-oriented routing with proper HTTP semantics
const express = require('express');
const router  = express.Router();

// GET  /orders/:id  — retrieve a single resource
router.get('/orders/:id', async (req, res) => {
  const order = await OrderService.findById(req.params.id);
  if (!order) return res.status(404).json({ error: 'Not found' });
  res.json(order);                           // 200 OK implicit
});

// POST /orders — create; returns 201 + Location header
router.post('/orders', async (req, res) => {
  const order = await OrderService.create(req.body);
  res.status(201)
     .set('Location', `/orders/${order.id}`)
     .json(order);
});

// PATCH /orders/:id — partial update (prefer over PUT for partials)
router.patch('/orders/:id', async (req, res) => {
  const updated = await OrderService.patch(req.params.id, req.body);
  res.json(updated);
});

// DELETE — idempotent; 204 No Content on success
router.delete('/orders/:id', async (req, res) => {
  await OrderService.delete(req.params.id);
  res.status(204).send();
});
Principal Architect Insight
REST is mature but watch for over-fetching (too many fields) and under-fetching (N+1 request problems). HATEOAS (Hypermedia As The Engine Of Application State) is often ignored in practice but critical for truly evolvable APIs. Always version your API (URI versioning /v1/ vs. header versioning).

1.2 GraphQL

GraphQL inverts the data-fetching model: the client specifies exactly what it needs in a typed query language. A single endpoint replaces dozens of REST endpoints. Ideal for complex, aggregated frontends and mobile clients where bandwidth is a concern.

Mobile Client GraphQL Gateway Services (REST/gRPC) │ │ │ │ │ │── query { │ │ │ │ │ user(id:1) { │ │ │ │ │ name, email │── fetch user ───────────►│ │ │ │ orders(last:3){│ │ │ │ │ total │── fetch orders(userId:1)─┘ │ │ │ } │ │ │ │ } │── fetch product details ─────┘ │ │ } ───────────────►│ │ │ │◄─ assemble single response ──────┘ │◄── {data:{user:…}} │
GraphQL Schema + Resolver — Apollo Server
// Schema — type system is the contract
const typeDefs = gql`
  type Order {
    id:       ID!
    total:    Float!
    status:   String!
    items:    [OrderItem!]!
  }
  type OrderItem {
    product:  Product!

// Resolvers — fetch data from wherever
const resolvers = {
  Query: {
    order: (_, { id }, { dataSources }) =>
      dataSources.orderAPI.getOrder(id),
    myOrders: (_, { last }, { dataSources, user }) =>
      dataSources.orderAPI.getUserOrders(user.id, last),

  Order: {
    // DataLoader batches & deduplicates N+1 item fetches

      loaders.orderItems.load(order.id),
  },
};

1.3 gRPC (Google Remote Procedure Call)

gRPC uses Protocol Buffers for binary serialization over HTTP/2, achieving 5–10× lower latency than REST/JSON. It generates typed client/server stubs from a .proto schema. The gold standard for internal microservice communication.

  string status      = 3;
  int64  created_at  = 4;
}

message OrderRequest { string order_id = 1; }

Section 02 — Asynchronous Messaging

Messaging & Async Integration

Async messaging decouples producers from consumers in time and space, enabling resilience, load levelling, and fan-out at scale.

2.1 Message Queue (Point-to-Point)

A queue delivers each message to exactly one consumer. This guarantees processing and enables competitive consumers (multiple workers pulling from the same queue) for horizontal scaling. Use for task dispatch, order processing, and any workflow requiring guaranteed exactly-once delivery.

Producer Queue (RabbitMQ / SQS / Azure SB) Consumers (Workers) │ ┌─────────────────────────────┐ │ │ │── msg A ────────►│ [msg E][msg D][msg C][msg B][msg A] │ │ │ │── msg B ────────►│ │────msg A──►│ │ │── msg C ────────►│ FIFO, at-least-once │────msg B──────────►│ │ └─────────────────────────────┘ │ │ Dead-Letter Queue on failure ack() ack()
SQS Producer + Consumer — AWS SDK v3
// Producer: send message to queue
import { SQSClient, SendMessageCommand } from "@aws-sdk/client-sqs";
const sqs = new SQSClient({ region: "us-east-1" });

await sqs.send(new SendMessageCommand({
  QueueUrl:     process.env.ORDER_QUEUE_URL,
  MessageBody:  JSON.stringify({ orderId: "42", action: "process" }),
  MessageGroupId: "orders",           // FIFO queue grouping
  MessageDeduplicationId: orderId,      // Idempotency key
}));

// Consumer: long-poll, process, delete (or nack → DLQ)
const consume = async () => {
  const { Messages } = await sqs.send(new ReceiveMessageCommand({
    QueueUrl: process.env.ORDER_QUEUE_URL,
    WaitTimeSeconds: 20,    // long-poll saves cost
    MaxNumberOfMessages: 10,
  }));

  await Promise.all((Messages || []).map(async msg => {
    try {
      const body = JSON.parse(msg.Body);
      await OrderService.process(body.orderId);
      await sqs.send(new DeleteMessageCommand({
        QueueUrl:      process.env.ORDER_QUEUE_URL,
        ReceiptHandle: msg.ReceiptHandle,  // ack
      }));
    } catch (err) {
      console.error('Processing failed; visibility timeout → DLQ', err);
      // Do NOT delete → message re-appears; after maxReceiveCount → DLQ
    }
  }));
};

2.2 Publish / Subscribe (Fan-Out)

A topic routes each message to all subscribed consumers. No subscriber misses a message. Ideal for event notifications, cache invalidation, and decoupling between bounded contexts.

Kafka Producer + Consumer — KafkaJS
import { Kafka } from 'kafkajs';
const kafka = new Kafka({ brokers: ['kafka:9092'] });

// --- PRODUCER ---
const producer = kafka.producer();
await producer.connect();
await producer.send({
  topic: 'order.placed',
  messages: [{
    key:   orderId,              // same key → same partition → ordering
    value: JSON.stringify(orderEvent),
    headers: { source: 'order-service', version: '1' },
  }],
});

// --- CONSUMER (Inventory Service) ---
const consumer = kafka.consumer({ groupId: 'inventory-grp' });
await consumer.subscribe({ topic: 'order.placed', fromBeginning: false });
await consumer.run({
  eachMessage: async ({ message }) => {
    const event = JSON.parse(message.value.toString());
    await InventoryService.reserve(event.items);  // idempotent!
  },
});

2.3 Webhook Pattern

Webhooks are "reverse APIs" — instead of polling, the provider calls your endpoint when events occur. Common in third-party integrations (Stripe, GitHub, Twilio). The challenge is reliability: ensure idempotent handling and validate signatures.

Webhook Receiver — Signature Validation + Idempotency
import crypto from 'crypto';

router.post('/webhooks/stripe', express.raw({ type: '*/*' }), async (req, res) => {
  // 1. Validate HMAC signature (prevents spoofed events)
  const sig       = req.headers['stripe-signature'];
  const expected  = crypto
    .createHmac('sha256', process.env.STRIPE_WEBHOOK_SECRET)
    .update(req.body)
    .digest('hex');
  if (sig !== `sha256=${expected}`)
    return res.status(401).end();

  const event = JSON.parse(req.body);

  // 2. Idempotency check — Stripe may retry on timeout
  const processed = await redis.get(`wh:${event.id}`);
  if (processed) return res.status(200).end(); // already handled

  // 3. Respond fast; process async
  res.status(200).end();
  await queue.enqueue('stripe-events', event); // hand off to queue
  await redis.setex(`wh:${event.id}`, 86400, '1');
});
Section 03 — Microservice Architecture Patterns

Microservice Patterns

Microservices introduce distributed systems challenges. These patterns solve the coordination, communication, and consistency problems that arise at scale.

🚪

API Gateway

Single entry point handling routing, auth, rate limiting, SSL termination, request aggregation.

  • Centralized cross-cutting concerns
  • Client simplicity
  • Potential bottleneck
  • Latency hop
📱

Backend for Frontend (BFF)

Dedicated API layer per client type (mobile, web, TV) optimizing payload and calls for each.

  • Client-tailored APIs
  • Team autonomy
  • Code duplication
  • More services to maintain
🔍

Service Discovery

Dynamic registry (Consul, Kubernetes DNS) where services find each other without hard-coded IPs.

  • Dynamic scaling
  • No config drift
  • Registry is a critical dep
🔀

Sidecar / Service Mesh

Proxy (Envoy/Istio) injected alongside each service, handling mTLS, retries, tracing transparently.

  • Uniform observability
  • Zero-trust networking
  • Resource overhead
  • Operational complexity
🌱

Strangler Fig

Incrementally migrate a monolith by routing traffic for new features to microservices behind a facade.

  • Low-risk migration
  • Incremental delivery
  • Two systems to run
  • Long transition
📊

Sidecar Aggregator

Aggregator service (or Gateway aggregation) composes responses from multiple downstream services.

  • Fewer client round-trips
  • Parallel fetching
  • Coupling risk
  • Latency adds up

Section 04 — Event-Driven Architecture

Event-Driven Architecture, CQRS & Saga

Event-driven systems react to state changes as first-class domain events, enabling loose coupling, auditability, and eventual consistency at scale.

4.1 CQRS — Command Query Responsibility Segregation

CQRS separates write operations (Commands) from read operations (Queries) into distinct models. Commands mutate state; Queries return data. The read model is updated asynchronously from the write model, allowing each to be optimised independently (e.g. denormalized read stores, OLAP-style projections).

WRITE SIDE (Commands) READ SIDE (Queries) Client──►Command Bus──►Command Handler Client──►Query Handler──►Read Store │ ▲ ▼ │ Write Store──event──►Event Bus──►Projection──updates──┘ (normalised) (async) (materialises view) Commands: PlaceOrder, CancelOrder Queries: GetOrderHistory, OrderSummary Write Store: PostgreSQL (normalised) Read Store: Elasticsearch / Redis / MongoDB
CQRS Command + Query Handlers — TypeScript
// ─── COMMAND SIDE ─────────────────────────────────────────
class PlaceOrderHandler {
  async execute(cmd: PlaceOrderCommand): Promise<string> {
    // Validate, apply business rules, persist to write model
    const order = Order.create(cmd.userId, cmd.items);
    await this.orderRepo.save(order);

    // Emit domain event to update read models
    await this.eventBus.publish(new OrderPlacedEvent({
      orderId: order.id,
      userId:  cmd.userId,
      total:   order.total,
      items:   order.items,
    }));
    return order.id;
  }
}

// ─── QUERY SIDE ───────────────────────────────────────────
class OrderHistoryQueryHandler {
  async execute(userId: string): Promise<OrderSummary[]> {
    // Reads from denormalised, query-optimised store
    return this.readStore.findByUser(userId);
  }
}

// ─── PROJECTION (keeps read model fresh) ──────────────────
class OrderProjection {
  @EventHandler(OrderPlacedEvent)
  async on(event: OrderPlacedEvent) {
    await this.readStore.upsert({
      id:     event.orderId,
      userId: event.userId,
      total:  event.total,
      status: 'placed',
      date:   new Date().toISOString(),
    });
  }
}

4.2 Saga Pattern — Distributed Transactions

When a business transaction spans multiple services, traditional ACID transactions are impossible. The Saga pattern breaks the transaction into a sequence of local transactions, each publishing an event. On failure, compensating transactions undo prior steps.

Two implementations: Choreography (services react to events, no central coordinator — simpler but harder to trace) and Orchestration (a Saga Orchestrator drives the flow — clearer but single coordinator is a dependency).

ORCHESTRATION SAGA — Order Fulfillment Saga Orchestrator │ ├──1. PlaceOrder ─────────────► Order Service ✓ order created │◄── OrderCreated ───────────── │ ├──2. ReserveInventory ───────► Inventory Service ✓ reserved │◄── InventoryReserved ──────── │ ├──3. ChargePayment ─────────► Payment Service ✗ FAILS │◄── PaymentFailed ──────────── │ ├──4a. ReleaseInventory ──────► Inventory Service ← COMPENSATE └──4b. CancelOrder ──────────► Order Service ← COMPENSATE
Saga Orchestrator — State Machine Pattern
class OrderFulfillmentSaga {
  private state: SagaState = 'STARTED';

  async execute(order: Order) {
    try {
      await this.step('CREATE_ORDER',
        () => orderSvc.create(order),
        () => orderSvc.cancel(order.id)       // compensate
      );
      await this.step('RESERVE_INVENTORY',
        () => inventorySvc.reserve(order.items),
        () => inventorySvc.release(order.items) // compensate
      );
      await this.step('CHARGE_PAYMENT',
        () => paymentSvc.charge(order),
        () => paymentSvc.refund(order.id)      // compensate
      );
      this.state = 'COMPLETED';
    } catch (err) {
      await this.rollback();   // run compensations in reverse
      this.state = 'FAILED';
      throw err;
    }
  }
}

4.3 Event Sourcing

Instead of storing current state, Event Sourcing persists every state change as an immutable event in an append-only log. Current state is derived by replaying events. This provides a complete audit trail, enables temporal queries, and allows projecting new read models from history.

Event Store + Aggregate Rehydration
// Events are the source of truth — immutable facts
const orderEvents = [
  { type: 'OrderCreated',   data: { id: '42', userId: 'u1', items: [...] } },
  { type: 'ItemAdded',      data: { productId: 'p99', qty: 2 } },
  { type: 'DiscountApplied', data: { code: 'SAVE10', amount: 10.00 } },
  { type: 'OrderConfirmed', data: { confirmedAt: '2024-01-15T10:00:00Z' } },
];

// Rehydrate aggregate by folding events
class OrderAggregate {
  apply(event: DomainEvent) {
    switch (event.type) {
      case 'OrderCreated':
        this.id     = event.data.id;
        this.status = 'draft';
        this.items  = [];
        break;
      case 'ItemAdded':
        this.items.push(event.data);
        break;
      case 'DiscountApplied':
        this.discount = event.data.amount;
        break;
      case 'OrderConfirmed':
        this.status = 'confirmed';
        break;
    }
  }

  static rehydrate(events: DomainEvent[]): OrderAggregate {
    const agg = new OrderAggregate();
    events.forEach(e => agg.apply(e));
    return agg;
  }
}
Section 05 — Data Integration Patterns

Data Patterns & Consistency

5.1 Outbox Pattern — Guaranteed Event Delivery

The Outbox Pattern solves the "dual write" problem: atomically save data AND publish an event in the same database transaction using a local outbox table. A relay process (polling or CDC) then reads and publishes outstanding outbox records.

WITHOUT OUTBOX WITH OUTBOX ┌──────────────────────────────┐ ┌───────────────────────────────────────┐ │ BEGIN TX │ │ BEGIN TX │ │ INSERT INTO orders ... │ │ INSERT INTO orders ... │ │ COMMIT │ │ INSERT INTO outbox │ │ ↓ │ │ (type, payload, status='pending')│ │ publish(event) ← can FAIL │ │ COMMIT ← atomic! │ └──────────────────────────────┘ └───────────────────────────────────────┘ Event lost if crash between │ commit and publish Relay (CDC/poll) reads outbox │ publish to Kafka/SQS │ mark outbox row 'published'
Outbox Pattern — Single Transaction (PostgreSQL)
async function placeOrder(order: Order, client: PoolClient) {
  await client.query('BEGIN');
  try {
    // 1. Write to business table
    const result = await client.query(
      'INSERT INTO orders (user_id, total, status) VALUES ($1,$2,$3) RETURNING id',
      [order.userId, order.total, 'placed']
    );
    const orderId = result.rows[0].id;

    // 2. Write event to outbox in SAME transaction
    await client.query(
      `INSERT INTO outbox (aggregate_id, event_type, payload, status)
       VALUES ($1, $2, $3, 'pending')`,
      [orderId, 'order.placed', JSON.stringify({ orderId, order })]
    );

    await client.query('COMMIT');  // atomic — both or neither
    return orderId;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  }
}

// Relay process (runs separately, polls or uses CDC like Debezium)
async function relayOutbox() {
  const rows = await db.query(
    `SELECT * FROM outbox WHERE status='pending' LIMIT 100 FOR UPDATE SKIP LOCKED`
  );
  for (const row of rows) {
    await kafka.send(row.event_type, row.payload);
    await db.query(`UPDATE outbox SET status='published' WHERE id=$1`, [row.id]);
  }
}

5.2 Change Data Capture (CDC)

CDC reads the database transaction log (WAL in PostgreSQL, binlog in MySQL) to capture every insert, update, and delete as a stream of events — without application code changes. Tools: Debezium, AWS DMS, Maxwell's Daemon.

Architect Tip — CDC vs Outbox
Use the Outbox Pattern when you control the application code and want explicit business events with domain meaning. Use CDC for legacy systems you can't modify, or when you need to capture every row-level change (data replication, analytics pipelines). Debezium with Kafka is the dominant CDC stack.

5.3 Data Mesh Principles

Data Mesh is an organizational and architectural pattern that treats data as a product, owned by decentralized domain teams. Each domain exposes its data through a standardized "data product" interface rather than funneling everything through a central data team.

Section 06 — Resilience & Reliability Patterns

Resilience Patterns

Distributed systems fail. These patterns contain failures, prevent cascades, and maintain service during degraded conditions.

⚡

Circuit Breaker

Wraps calls to external services. Trips OPEN after N failures, returning fast errors without calling the downstream. Automatically half-opens after a timeout to probe recovery.

  • Prevents cascade failures
  • Fast failure response
  • False trips if thresholds wrong
  • Complex state management
🔁

Retry with Backoff

Retry transient failures with exponential backoff and jitter. Jitter prevents thundering herd (all clients retrying simultaneously after an outage).

  • Handles transient faults
  • Simple to implement
  • Can amplify load
  • Increases latency
🪣

Bulkhead

Isolate resources (thread pools, connection pools, semaphores) per dependency so one slow downstream can't consume all threads and affect unrelated services.

  • Blast radius containment
  • Resource isolation
  • More resources needed
  • Config overhead
🕐

Timeout

Always set timeouts on external calls. A slow dependency that never times out will exhaust threads. Set aggressive timeouts at the business layer, not just TCP.

  • Bounded wait time
  • Frees resources
  • Partial execution risk
  • Hard to calibrate
🛡️

Rate Limiting

Token bucket or sliding window limits requests per client/IP/API key. Protects services from overload, DDoS, and misbehaving consumers.

  • Protects availability
  • Fair usage enforcement
  • Legitimate traffic rejected
  • Distributed state needed
↩️

Fallback / Degraded Mode

Return cached data, default responses, or reduced functionality when a dependency fails. Prefer a degraded experience over a total outage.

  • Maintains user experience
  • Reduces total outages
  • Stale data risk
  • Business logic complexity
Circuit Breaker — Implementation (opossum library / Node.js)
import CircuitBreaker from 'opossum';

const options = {
  timeout:              3000,   // 3s — if action takes longer, it fails
  errorThresholdPercent: 50,   // trip when 50% of requests fail
  resetTimeout:         30000, // try again after 30s (half-open)
  volumeThreshold:      10,    // need at least 10 calls to trip
};

const breaker = new CircuitBreaker(paymentService.charge, options);

// Fallback: return a pending state instead of failing completely
breaker.fallback((order) => ({
  status:  'pending',
  message: 'Payment queued — system under load',
  orderId: order.id,
}));

// Events for monitoring
breaker.on('open',     () => metrics.increment('circuit.open'));
breaker.on('halfOpen', () => metrics.increment('circuit.halfOpen'));
breaker.on('close',    () => metrics.increment('circuit.close'));

// Use like the original function
const result = await breaker.fire(order);
Exponential Backoff with Jitter
async function withRetry<T>(
  fn: () => Promise<T>,
  { maxRetries = 3, baseDelay = 100, maxDelay = 10000 } = {}
): Promise<T> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === maxRetries || !isRetryable(err)) throw err;

      // Exponential backoff: 100ms, 200ms, 400ms, 800ms...
      const exponential = baseDelay * Math.pow(2, attempt);
      // Full jitter: spread retries to avoid thundering herd
      const jitter      = Math.random() * exponential;
      const delay       = Math.min(jitter, maxDelay);

      console.warn(`Attempt ${attempt + 1} failed; retrying in ${delay.toFixed(0)}ms`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

7.2 JWT Validation Pattern

JWT Middleware — Verification Best Practices
import jwt   from 'jsonwebtoken';
import jwks  from 'jwks-rsa';

const jwksClient = jwks({ jwksUri: `https://auth.example.com/.well-known/jwks.json` });

const authMiddleware = async (req, res, next) => {
  const token = req.headers.authorization?..split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Missing token' });

  try {
    // Fetch public key from JWKS endpoint (cached internally)
    const decoded = jwt.decode(token, { complete: true });
    const key     = await jwksClient.getSigningKey(decoded.header.kid);

    // Verify signature, expiry, issuer, audience
    const payload = jwt.verify(token, key.getPublicKey(), {
      algorithms: ['RS256'],          // reject HS256 (symmetric)
      issuer:     'https://auth.example.com',
      audience:   'https://api.example.com',
    });

    req.user = payload;
    next();
  } catch (err) {
    res.status(401).json({ error: 'Invalid token' });
  }
};
Security Anti-Patterns to Avoid
Never use algorithm: 'none' in JWT verification. Always validate iss (issuer) and aud (audience). Store refresh tokens in HttpOnly cookies, not localStorage. Use short-lived access tokens (15 min) with refresh rotation. Implement token revocation via introspection endpoint or short expiry.

7.3 Zero Trust Architecture

Zero Trust assumes no implicit trust, even inside the corporate network. Every service call is authenticated and authorized regardless of network location. Key principles: verify explicitly (always authenticate), least privilege (minimal access), assume breach (monitor and limit blast radius).

Principal Architect — Interview Key Themes
Always discuss: (1) Why a pattern, not just what — justify against non-functional requirements. (2) CAP theorem trade-offs (Consistency vs Availability vs Partition tolerance). (3) Eventual consistency and how you handle it in UX. (4) Idempotency — every message handler and API endpoint should be idempotent. (5) Schema evolution strategy — backward/forward compatibility (Avro, Protobuf with schema registry). (6) Observability: structured logging, distributed tracing (OpenTelemetry), metrics (RED method: Rate, Errors, Duration). (7) Domain-Driven Design as the foundational tool for identifying service boundaries (bounded contexts).

8.4 Key Principles Summary

🎯

Design for Failure

Everything fails in distributed systems. Use Circuit Breakers, timeouts, retries with backoff, and bulkheads. Test failure paths with chaos engineering (Chaos Monkey, Gremlin).

🔄

Embrace Idempotency

Design all operations to be safely retried. Use idempotency keys on POST APIs, deduplication IDs in queues, and idempotent handlers for event processing.

📐

Schema-First Contracts

Define API contracts before implementation (OpenAPI, AsyncAPI, Protobuf, Avro). Use schema registries. Apply backward compatibility rules — never remove fields, add defaults.

🔭

Observability by Default

Instrument every service with the three pillars: structured logs (JSON), distributed traces (OpenTelemetry + Jaeger/Tempo), and metrics (Prometheus + Grafana). Build dashboards before incidents.
