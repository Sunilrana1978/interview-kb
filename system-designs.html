01 URL Shortener
Use Case
  • Shorten long URLs into compact shareable links (e.g. bit.ly, t.co)
  • Track click analytics per short link
  • Support optional expiry and custom aliases
  • Handle 100M URL creations/day and 10B redirects/day
  • Redirect latency must be under 10ms globally
Solution

Generate unique short codes using a Snowflake ID (timestamp + machine ID + sequence) encoded in Base62 — no coordination needed, no collisions possible. Serve redirects from a tiered cache: CDN at the edge for the hottest links, then Redis LRU for the long tail, with PostgreSQL as source of truth only on cache miss. Separate read/write paths allow independent scaling. Top 1% of URLs serve ~50% of traffic — caching is the key lever.

Base62 encoding Snowflake ID Redis LRU CDN edge cache Read replicas
02 Ride-Sharing App
Use Case
  • Match riders to the nearest available driver in real time
  • Track 500K concurrent driver locations updating every 5 seconds
  • Calculate ETA-based ranking, not just raw distance
  • Apply dynamic surge pricing based on supply vs demand per zone
  • Handle 5M rides/day with <2s matching latency
Solution

Drivers stream GPS over WebSockets into a Location Service that writes to Redis Geo using GEOADD — enabling O(log N) radius queries via geohash. Matching Service calls GEORADIUS, filters available drivers, scores by OSRM ETA, and offers the top candidate with a 10s timeout. Location events also flow into Kafka → Flink, which aggregates supply/demand per geohash cell every 30s to compute the surge multiplier. Redis is sharded by city/region for locality.

Redis Geo Geohash sharding Kafka + Flink OSRM ETA scoring Surge pricing
03 Distributed Cache
Use Case
  • Store and retrieve key-value data with sub-millisecond latency
  • Scale horizontally across many nodes without downtime
  • Survive individual node failures with automatic failover
  • Support TTL-based expiry and configurable eviction policies
  • Optionally persist data to disk for durability (RDB / AOF)
Solution

Partition the keyspace into 16,384 hash slots via CRC16(key) % 16384. Each node owns a slot range — adding a node only remaps ~1/N keys. Every primary has 2 async replicas. On failure, replicas elect a new leader via Raft consensus in ~30s. The client library handles slot routing transparently. Eviction is configurable: allkeys-lru for general caches, allkeys-lfu for skewed workloads. Persistence combines RDB snapshots for fast restarts with AOF for near-zero data loss.

Consistent hashing 16,384 slots LRU / LFU eviction Raft election RDB + AOF
04 News Feed (Twitter/X)
Use Case
  • Serve personalized feeds to 500M users with <200ms load time
  • Handle 500M tweets/day from users with varying follower counts
  • Celebrity accounts with 100M+ followers create extreme fan-out pressure
  • Support chronological and ML-ranked feed ordering
  • Paginate efficiently without offset-based queries
Solution

Use a hybrid fan-out strategy: for regular users (<10K followers), pre-compute feeds on write by pushing tweet IDs into each follower's Redis Sorted Set (score = timestamp, max 800 entries). For celebrities (>10K followers), skip the push entirely — fetch their posts at read time and merge. This avoids writing one tweet to 50M feeds. Kafka drives async fan-out workers. An ML ranking pipeline on Flink scores candidates by engagement signals and re-ranks the top 500 → serve 20.

Hybrid fan-out Redis Sorted Set Kafka async workers ML re-ranking Cursor pagination
05 Distributed Job Scheduler
Use Case
  • Schedule millions of one-time and recurring (cron) jobs reliably
  • Guarantee exactly-once execution — no double-firing on retries
  • Support DAG dependencies (Job B only runs after A and C complete)
  • Recover automatically from worker crashes with exponential backoff
  • Achieve 100K jobs/sec throughput with horizontal worker scaling
Solution

A single scheduler leader is elected via ZooKeeper advisory lock — only one instance fires jobs at a time. It polls PostgreSQL every second using SELECT FOR UPDATE SKIP LOCKED to atomically claim jobs without contention, then enqueues them to Kafka/SQS. Workers pull jobs, heartbeat while running, and update state. Crashed workers let their lease expire — the job resets to pending. Idempotency keys prevent double execution. DAG resolution checks all upstream dependencies on each completion event before enqueuing the next job.

Leader election ZooKeeper lock SKIP LOCKED Idempotency keys DAG resolution Exponential backoff
