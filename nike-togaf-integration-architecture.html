Domain Context — Nike Product Creation

Understand the business landscape before architecture begins

Foundation
👟
What is Product Creation?
The end-to-end process of taking a footwear / apparel idea from concept → design → development → manufacturing → launch. Involves 12–18 month cycles with global teams across Portland HQ, EMEA, APAC design studios, and contract manufacturers.
🔗
Key Integration Pain Points
Data silos between PLM, ERP, and CAD systems · Manual hand-offs from design to costing to sourcing · Late changes causing rework across 5+ systems · No single source of truth for product specifications.
🏗️
Core Application Landscape
Flex PLM Product Lifecycle Mgmt · SAP S/4HANA ERP / Finance · Adobe Substance 3D Design · Optitex Pattern / CAD · Arena PLM BOM management · Coupa Supplier collaboration · Salesforce Go-to-market CRM · MuleSoft Integration Platform
👥
Stakeholder Map

Domain Owners: Design, Costing, Sourcing, Manufacturing
Technical: Enterprise Architects, Platform Engineering, Security
External: Tier-1 Suppliers, Logistics Partners

Phase A — Architecture Vision

Define the target state and secure executive buy-in

Vision
🎯
Problem Statement
Nike's product creation cycle is delayed 4–6 weeks per season due to manual data re-entry across disconnected systems. Costing teams re-key BOM data from Flex PLM into SAP manually — causing errors, rework, and missed launch windows.
🌟
Target State Vision
Seamless product data flow from concept (Flex PLM) → costing (SAP) → supplier collaboration (Coupa) → go-to-market (Salesforce) with zero manual re-keying. Enable real-time visibility into product cost and availability across 200+ suppliers.
📊
Business Value & KPIs
Reduce time-to-market by 30% · Cut data errors in costing by 90% · Enable same-day BOM updates to suppliers · Increase season line plan accuracy to 95%+ · Save ~$12M/yr in rework costs
📄
Statement of Architecture Work (Key Deliverable)
Formal document approved by the Architecture Governance Board. Scopes the integration architecture engagement to the Product Creation domain covering: PLM ↔ ERP integration, Supplier Collaboration Integration, Design ↔ PLM integration. Timeline: 6-month architecture engagement. Budget: $2.4M integration platform investment.

Infrastructure, middleware, and NFR decisions

Technology
☁️
Integration Platform Stack
iPaaS: MuleSoft Anypoint Platform (cloud-hosted)
Messaging: Confluent Kafka (AWS MSK) for async events
API Gateway: MuleSoft API Manager + Anypoint Exchange
Service Mesh: AWS App Mesh for internal microservices
Monitoring: Dynatrace APM + Anypoint Monitoring
⚡
Non-Functional Requirements
Availability: 99.95% for PLM↔SAP integration
Throughput: 50,000 BOM events/day at peak season
Latency: Costing sync <500ms P99 · Async events <2s P99
Scalability: Auto-scale Mule workers during peak periods
RPO/RTO: 15min / 1hr for critical integration flows
🛡️
Security Architecture
AuthN/AuthZ: OAuth 2.0 + JWT for all APIs · Okta as IdP
mTLS: Mutual TLS between Mule workers and SAP
API Throttling: Rate limits per consumer client ID
Secrets: AWS Secrets Manager — no credentials in config
Network: VPC peering, no public internet for SAP traffic
🔁
Resilience Patterns
Circuit Breaker: On SAP BAPI calls (Resilience4j)
Dead Letter Queue: Kafka DLQ for failed BOM events
Idempotency: StyleID + version as idempotency key
Retry Policy: Exponential backoff 3× before DLQ
Bulkhead: Separate thread pools per integration flow

Migration Roadmap — 3 Horizons
H1 (0–3 months): Observability setup, PLM→SAP sync MVP (Kafka), decommission CSV process
H2 (3–6 months): Design→PLM integration, Coupa supplier API, canonical data model
H3 (6–12 months): Salesforce GTM feed, API productization, supplier self-service portal
🔀
Transition Architecture
During H1→H2 transition, both the legacy CSV process and new Kafka flow run in parallel (shadow mode). Reconciliation jobs compare outputs daily. Cutover to Kafka-only only after 2-week zero-discrepancy window. This de-risks the migration with no hard cutover.

Key Architecture Decisions to Articulate

These show depth of thinking — crucial for senior roles

Interview Gold

ADR-01 · MuleSoft over Custom Middleware

Context: Nike needed to integrate 8+ systems with different protocols (REST, SOAP, BAPI). Decision: Adopt MuleSoft Anypoint Platform. Rationale: Pre-built connectors for SAP BAPI and Flex PLM reduce build time by 60%. API governance, Exchange marketplace, and monitoring built-in. Trade-off: High licensing cost justified by reduced engineering headcount and faster time-to-market.

ADR-02 · Event-Driven for BOM Sync over Polling

Context: 50,000 BOM change events/day with legacy polling every 15 minutes causing 15-min lag and missed changes. Decision: Kafka-based event streaming. Rationale: Kafka provides durable event log, supports replay for new consumers, decouples producer (Flex PLM) from consumers (SAP, Coupa). Trade-off: Operational complexity of Kafka management — mitigated by using Confluent Cloud managed service.

ADR-03 · Canonical Data Model for Product Entity

Context: Each system had its own product schema (Style in Flex PLM, Material in SAP, SKU in Salesforce). Decision: Define a canonical ProductSpec schema as the integration lingua franca. Rationale: Reduces transformation logic from O(n²) to O(n). Schema governed in Anypoint Exchange as the contract. Trade-off: Upfront modeling effort; worth it for maintainability at scale.

ADR-04 · API-Led Connectivity Layering

Context: Multiple consumer teams (internal analytics, external supplier portal) needed access to the same underlying data via different shapes. Decision: MuleSoft's 3-layer API-Led pattern (System/Process/Experience). Rationale: Allows SAP's System API to be reused across multiple Process APIs without re-implementation. Experience APIs tailored per channel. Trade-off: More APIs to manage — offset by reuse dividend.

Interview Talking Points & Tips

How to structure your answers and impress the panel

Interview Prep
Opening Statement "I'd approach this using TOGAF ADM but adapted pragmatically — I start by understanding the business capability gaps and integration trigger events before touching any technology. For Nike's Product Creation domain, the key is identifying the system of record boundaries and the integration style — event-driven vs. request-reply — appropriate for each hand-off."
On Technology Choices Always justify with trade-offs. Don't just say "we used Kafka." Say: "We chose Kafka over a simple message queue because we needed replay capability for the audit trail requirement and to support onboarding new consumers without replaying from source systems."
On Governance Mention ADRs proactively — it signals maturity. "I document every significant architectural decision as an ADR with context, decision, and trade-offs. This prevents the architecture from drifting during implementation and gives future teams the reasoning, not just the decision."
On Stakeholder Management "In Phase B, I ran capability workshops with the Product Creation VP and domain leads. My job as integration architect is not to impose technology — it's to translate business events into integration patterns. The business says 'BOM Approved' — I say that's an event, who needs to react to it?"
On NFRs & SLAs Quantify everything. "The costing approval flow needed to be synchronous and return in under 500ms P99 because the costing team operates in real-time during line review meetings. That drove the decision to keep that specific flow request-reply, not async."
On Migration Risk "I advocate for parallel-run transition architectures. For the BOM sync, we ran both the CSV legacy process and the new Kafka flow in shadow mode for 2 weeks with a reconciliation job. Only after zero discrepancies did we cut over. This de-risked the migration entirely."
On API Strategy "I follow a contract-first approach — API schemas defined in OpenAPI/AsyncAPI specs and published in the API Exchange before implementation begins. This unblocks parallel development between integration teams and consuming teams."
On Observability "An integration architecture is only as good as its observability. From day one, I instrument every flow with structured logs, distributed traces, and SLA dashboards. Silent failures in async flows are the most dangerous — they corrupt downstream data without anyone knowing."
