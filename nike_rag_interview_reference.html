PHASE 6
RAG query loop — what happens at runtime
1
User submits query via PLM app
PLM frontend posts to the API Gateway endpoint. API Gateway triggers the Lambda function with a proxy event payload.
PLM app → API Gateway (HTTP API) → Lambda trigger
2
Mangum translates event → FastAPI request
Mangum adapter converts the Lambda event dict into an ASGI-compatible request. FastAPI routes it to the correct endpoint and validates via Pydantic.
Mangum (ASGI adapter) → FastAPI router → Pydantic model
3
Query embedded with Titan Embeddings
FastAPI service calls Bedrock Titan Embeddings via boto3 to convert query text to a dense vector — same model used at ingestion time for consistent vector space.
Amazon Bedrock · Titan Embeddings · boto3
4
k-NN retrieval from Amazon OpenSearch
Query vector is used for k-NN similarity search on the OpenSearch vector index. Top-K most relevant PLM support documents and incident records are returned.
Amazon OpenSearch · k-NN index · opensearch-py
5
Grounded prompt constructed, Claude 3 invoked
Retrieved documents injected into structured prompt template. Claude 3 (fine-tuned, on Bedrock) generates a grounded answer via InvokeModel API. Answer is traceable to retrieved source material.
Python prompt template → Bedrock InvokeModel → Claude 3 Sonnet
6
Mangum returns response → API Gateway → PLM app
FastAPI returns JSON. Mangum converts it back to a Lambda response dict. API Gateway translates to HTTP and returns to the PLM app chatbot UI.
FastAPI JSON → Mangum → Lambda response → API Gateway → PLM UI

03 · Code

FastAPI + Mangum on Lambda

Key insight: Mangum is a single-line addition to a standard FastAPI app. Your application code is identical to what you'd write for any other deployment target — Mangum just handles the Lambda ↔ ASGI translation at the boundary.
FastAPI app + Mangum adapter
Standard FastAPI application with Pydantic request/response models. The only Lambda-specific line is handler = Mangum(app) at the bottom — Lambda invokes this function.
# main.py from fastapi import FastAPI, Depends, HTTPException from fastapi.security import OAuth2PasswordBearer from pydantic import BaseModel from mangum import Mangum from rag import retrieve, invoke_claude3 from auth import verify_token app = FastAPI() oauth2 = OAuth2PasswordBearer(tokenUrl="/token") class QueryRequest(BaseModel): question: str class QueryResponse(BaseModel): answer: str sources: list[str] @app.post("/query", response_model=QueryResponse) async def query_plm(req: QueryRequest, token: str = Depends(oauth2)): verify_token(token) # validate Bearer token chunks = retrieve(req.question) answer = invoke_claude3(req.question, chunks) return QueryResponse(answer=answer, sources=chunks) # Lambda entry point — one line handler = Mangum(app)
Python 3.12 FastAPI Mangum Pydantic v2
RAG core — OpenSearch + Claude 3
Titan Embeddings vectorises the query. OpenSearch k-NN returns top-K neighbours. Claude 3 InvokeModel generates a grounded answer from the retrieved context.
# rag.py import boto3, json from opensearchpy import OpenSearch bedrock = boto3.client("bedrock-runtime") os_client = OpenSearch(os_endpoint) def retrieve(query: str) -> list: vec = titan_embed(query) resp = os_client.search( index="plm-vectors", body={"knn": {"vector": vec, "k": 5}} ) return [h["_source"]["text"] for h in resp["hits"]["hits"]] def invoke_claude3(q, chunks) -> str: prompt = build_prompt(q, chunks) r = bedrock.invoke_model( modelId="anthropic.claude-3-sonnet", body=json.dumps({"prompt": prompt, "max_tokens": 1024}) ) return json.loads(r["body"].read())
Bedrock InvokeModel Claude 3 Sonnet OpenSearch k-NN boto3 opensearch-py

Provisioned Concurrency (cold start fix)
RAG pipeline has init overhead — boto3 clients, OpenSearch connection. Provisioned Concurrency keeps N Lambda containers pre-warmed, eliminating cold starts for PLM users. Defined in CloudFormation.

04 · Infrastructure as Code

AWS CloudFormation stack

All infrastructure is defined in CloudFormation YAML templates. This means every resource — Lambda, API Gateway, OpenSearch domain, S3 buckets, IAM roles, and Provisioned Concurrency — is version-controlled, repeatable, and deployable via a single stack update from Jenkins.
Lambda + API Gateway resources
Lambda function defined with container image URI from ECR. API Gateway HTTP API with Lambda proxy integration. IAM execution role grants Bedrock and OpenSearch access.
# cfn/lambda.yaml (excerpt) RagFunction: Type: AWS::Lambda::Function Properties: FunctionName: plm-rag PackageType: Image Code: ImageUri: !Sub "${ECRRepo}:${ImageTag}" MemorySize: 3008 Timeout: 30 Role: !GetAtt RagExecutionRole.Arn Environment: Variables: OS_ENDPOINT: !GetAtt OpenSearchDomain.DomainEndpoint OAuth2Authorizer: Type: AWS::ApiGatewayV2::Authorizer Properties: ApiId: !Ref RagHttpApi AuthorizerType: JWT IdentitySource: ["$request.header.Authorization"] JwtConfiguration: Audience: ["plm-rag-api"] Issuer: !Sub "https://auth.${Domain}"
AWS::Lambda::Function AWS::ApiGatewayV2::Api AWS::IAM::Role CloudFormation
OpenSearch + S3 resources
OpenSearch domain for the k-NN vector index. S3 bucket for source documents. Both provisioned via CloudFormation so infra is reproducible across environments (dev, staging, prod).

Stack organisation
CloudFormation templates are split into nested stacks by concern — data layer, compute layer, API layer, IAM. A root stack references each nested stack, enabling independent updates to individual layers.
# cfn/ directory structure cfn/ root.yaml # master stack stacks/ data.yaml # S3 + OpenSearch compute.yaml # Lambda + concurrency api.yaml # API Gateway + routes iam.yaml # roles + policies # Jenkins deploys via: aws cloudformation deploy \ --template-file cfn/root.yaml \ --stack-name plm-rag \ --capabilities CAPABILITY_IAM \ --parameter-overrides \ ImageTag=$IMAGE_TAG

Rollback strategy
CloudFormation automatically rolls back a stack if the changeset fails. For Lambda-specific rollbacks, re-run the Jenkins job with a previous BUILD_NUMBER (previous ECR image tag) as a parameter override.
# Manual rollback — re-deploy previous build aws cloudformation deploy \ --template-file cfn/root.yaml \ --stack-name plm-rag \ --capabilities CAPABILITY_IAM \ --parameter-overrides \ ImageTag=$PREVIOUS_BUILD_NUMBER # Or: update Lambda directly to prev image aws lambda update-function-code \ --function-name plm-rag \ --image-uri "$ECR_REPO:$PREVIOUS_BUILD_NUMBER" # CloudFormation auto-rollback on failure: # --on-failure ROLLBACK (default behaviour)

06 · Key Decisions

Architectural choices & rationale

chose
FastAPI + Mangum on Lambda
over ECS Fargate or EC2
Serverless — no containers to manage or pay for when idle. Mangum makes it a one-line change from a standard FastAPI app. Provisioned Concurrency removes cold starts for the PLM use case.
chose
CloudFormation IaC
over manual console / Terraform
Native AWS — no third-party state management. All infra (Lambda, API GW, OpenSearch, IAM) is version-controlled YAML. Nested stacks let you update individual layers independently without touching the whole stack.
chose
Jenkins pipeline
over GitHub Actions / CodePipeline
The organisation already runs Jenkins on-premise. Using the existing CI platform means no new tooling, no additional cost, and no changes to developer workflow. Jenkinsfile-as-code keeps pipeline logic in the repo.
chose
Claude 3 on Bedrock
over self-hosted open-source LLMs
Best reasoning quality for complex PLM support scenarios. Bedrock = no GPU infra, enterprise SLA, data stays within the AWS boundary. Lambda + Bedrock both serverless — consistent cost model.
chose
Amazon OpenSearch
over Pinecone / external vector DBs
Data stays within the AWS environment. k-NN + BM25 hybrid retrieval in one service. CloudFormation can provision the domain directly — no external API keys or accounts to manage.
chose
Provisioned Concurrency
over accepting Lambda cold starts
RAG pipeline initialises boto3 clients and OpenSearch connections — 2-5s cold start is unacceptable for an in-app chatbot. Provisioned Concurrency keeps 3 containers warm at a fixed hourly cost, giving consistent response times.
07 · Interview Script (STAR)

What to say — in order

SSituation — set the scene

"The PLM application had no self-serve support layer. Engineers and product teams were hitting the same questions and incident patterns repeatedly with no way to surface answers in context. Support load was high, resolution was slow, and there was no institutional memory of past incidents."

TTask — your ownership

"I was the solutions architect responsible for the end-to-end design — model selection, vector database architecture, serverless API strategy, infrastructure-as-code design, and CI/CD pipeline. I owned the full technical delivery."

AAction — the technical approach

"I selected Claude 3 on AWS Bedrock — best reasoning quality for complex PLM scenarios, fully managed, no GPU ops, data stays within the AWS boundary. For the vector database I chose Amazon OpenSearch with a k-NN index — keeps data inside AWS, and supports hybrid semantic and keyword retrieval in one service. I ingested PLM support docs and incident records into S3, vectorised them with Titan Embeddings, and indexed them in OpenSearch. The model was fine-tuned on that domain content so it understands PLM-specific terminology and known failure patterns. The runtime query flow is RAG — embed the query with Titan, k-NN search OpenSearch for top-K relevant docs, inject them into a structured prompt, invoke Claude 3 via Bedrock InvokeModel. The API layer is FastAPI with Mangum as the ASGI adapter, running on AWS Lambda — so it's completely serverless. I added Provisioned Concurrency to keep three Lambda containers pre-warmed, eliminating cold starts for PLM app users. All infrastructure — Lambda, API Gateway, OpenSearch domain, IAM roles — is defined as CloudFormation nested stacks, version-controlled YAML templates deployed from a Jenkins pipeline. The Jenkins pipeline runs pytest, builds a Lambda container image, pushes to ECR, runs a CloudFormation changeset, and finishes with smoke tests against the live endpoint."

RResult — the business impact

"The result was an AI chatbot embedded directly inside the PLM app delivering real-time, grounded answers — no ticket filing, no context switching. Support load reduced, resolution times improved, and the knowledge base can be kept current without retraining the model. The serverless architecture means we pay only for actual usage, and the entire infrastructure can be re-provisioned from scratch in a single CloudFormation deploy."

08 · Likely Follow-Up Questions

Be ready for these

How does Mangum work with FastAPI on Lambda?
Mangum is an ASGI adapter — it translates the Lambda event payload (which comes from API Gateway) into the ASGI interface that FastAPI expects, and translates the ASGI response back into the Lambda response format. Your FastAPI code is completely unchanged. The only Lambda-specific line is handler = Mangum(app) at the bottom of the file, which is what Lambda invokes.
How did you solve Lambda cold starts for the RAG pipeline?
The RAG pipeline has meaningful init overhead — importing boto3, FastAPI, and establishing the OpenSearch connection takes 2-5 seconds in a cold container. I used Lambda Provisioned Concurrency to keep 3 containers pre-warmed at all times. Those 3 containers handle the PLM chatbot's concurrent load without ever going cold. The tradeoff is a fixed hourly cost for those 3 containers, but for a real-time in-app experience that's the right call.
Why CloudFormation over Terraform?
The organisation is all-in on AWS. CloudFormation is native — no third-party state management, no backend S3 state file to maintain, and no drift between the IaC tool's state and AWS's actual state. It integrates directly with the Jenkins pipeline via aws cloudformation deploy. Terraform would have been a valid choice too — the decision was largely about what the team already knew and what reduced operational overhead.
Why Jenkins over GitHub Actions or CodePipeline?
The team already operates a Jenkins infrastructure. Reusing the existing CI platform meant no new tooling, no new access management, and no changes to developer workflow. The Jenkinsfile is committed to the repo, so the pipeline is version-controlled and reviewable like any other code. GitHub Actions would have worked equally well technically — it was an organisational fit decision.
How do you handle Lambda's 15-minute timeout for long RAG queries?
The RAG pipeline — embed, retrieve, generate — typically completes in 3-8 seconds end-to-end. Lambda's 15-minute timeout is not a practical constraint here. The timeout is set to 30 seconds in CloudFormation to give generous headroom while failing fast on any genuine hang. If we ever needed streaming responses, API Gateway supports Lambda response streaming from Python 3.12 onwards.
How do you roll back a bad deployment?
Two layers. First, CloudFormation automatically rolls back a stack if the changeset execution fails — the infrastructure never lands in a broken state. Second, for an application-level regression that passes deployment but fails smoke tests, the Jenkins pipeline is re-run with the previous BUILD_NUMBER as a parameter override, which points the Lambda function at the previous ECR image tag. No manual console work required.
Why RAG instead of just fine-tuning?
Fine-tuning alone bakes knowledge into model weights — expensive to update and the model can hallucinate PLM-specific facts it wasn't trained on. RAG keeps answers grounded in retrieved source documents — traceable and auditable. When new support docs or incidents come in, we update the OpenSearch index, not the model. Much faster and cheaper update cycle for a knowledge base that changes regularly.
What would you do differently if you rebuilt it today?
Tighter feedback loops — thumbs up/down in the chatbot UI feeding a continuous eval dataset for model improvement. Hybrid OpenSearch retrieval combining k-NN vector search with BM25 keyword search for better recall on specific PLM error codes. Bedrock Guardrails for out-of-scope query handling. Lambda response streaming for perceived latency improvement. And AWS CDK instead of raw CloudFormation YAML for a better developer experience on the IaC side.
