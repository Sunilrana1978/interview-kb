Data Versioning & Freshness
5 questions
Q1 Your RAG data changes every hour. How do you manage versioning without breaking the system? Hard ▼

Strategy: Treat each hourly ingestion as an immutable release, not a mutation.

  • Tag indexes with kb-{version}-{timestamp}; maintain a pointer to the "active" version
  • Use a blue-green switchover: ingest into staging, validate, then atomically flip the pointer
  • Keep 3–5 prior versions in hot storage for instant rollback
  • Lock the embedding model per version — swapping models requires a full rebuild
  • In-flight queries hold a reference to the version at request time — never mutate an active index
# Atomic switchover (pseudocode)
if validate_index(staging_version):
    metadata_store.set("active", staging_version)
    cleanup_old_versions(keep=3)
else:
    alert("Validation failed — keeping prior version")
OpenSearchBedrockBlue-Green
Q2 How do you detect and handle embedding drift when your source documents evolve over time? Hard ▼

Drift occurs when the semantic space shifts due to model updates or domain vocabulary changes.

  • Track cosine similarity of a fixed golden query set across versions — significant drops signal drift
  • Store embedding model version in document metadata; fail fast on mismatch at query time
  • If embedding model changes, trigger a full index rebuild rather than partial re-embedding
  • Monitor retrieval recall using held-out Q&A pairs as ground truth
EmbeddingMonitoringObservability
Q3 How do you handle deleted or retracted documents — ensuring stale content doesn't surface in answers? Medium ▼
  • Tombstone records: Mark deleted docs with a status: deleted metadata flag; filter at retrieval time
  • Hard delete pipeline: Scheduled job purges tombstoned vectors from the index
  • Immutable versioned indexes: Re-build a clean index each cycle — no tombstoning needed
  • For compliance (GDPR/CCPA): hard deletes must propagate within SLA; tombstone approach requires strict cleanup guarantees
ComplianceGDPRIndex Management
Q4 What's your strategy for partial vs. full index rebuilds — and when does each make sense? Medium ▼
  • Partial rebuild: Efficient for append-only or low-churn data; upsert changed docs by ID
  • Full rebuild: Required when embedding model changes, schema evolves, or >30–40% of corpus changes
  • Track a content_hash per document; only re-embed on hash change
  • Cost trade-off: full rebuild is expensive but guarantees consistency; partial is cheap but risks index fragmentation
PerformanceCostIndex
Q5 How do you validate a new KB version before promoting it to production traffic? Core ▼
  • Document count check: New version within expected ± threshold of prior version
  • Schema validation: All required metadata fields present and typed correctly
  • Golden query suite: Run 20–50 canonical queries; verify top-k docs match expected IDs
  • Canary traffic: Route 5–10% of real queries to staging; compare latency and answer quality
  • Embedding dimension check: Assert dim == 1536 (or model-specific value) before any queries
ValidationCanaryTesting

Retrieval Quality & Architecture
6 questions
Q6 Dense vs. sparse retrieval — when would you use hybrid search and how do you tune it? Hard ▼
  • Dense (vector): Excellent for semantic similarity; poor on exact-match keywords, product codes, IDs
  • Sparse (BM25): Strong on keyword/lexical match; misses paraphrases and synonyms
  • Hybrid: Combines both with a reciprocal rank fusion (RRF) or linear score blend — best of both worlds
  • Tuning: adjust the alpha weight (dense vs. sparse) using offline evaluation against labeled queries
  • Use hybrid when the domain has precise terminology (SKUs, legal citations, error codes) alongside free-form queries
BM25Hybrid SearchRRFOpenSearch
Q7 How does re-ranking improve retrieval, and what's the latency trade-off? Medium ▼

Initial retrieval (ANN search) optimizes for speed — it can miss subtle relevance distinctions. A re-ranker is a cross-encoder that scores query+doc pairs with higher accuracy.

  • Pattern: Retrieve top-50 via ANN → re-rank to top-5 → send to LLM
  • Adds 50–300ms latency depending on model size and doc length
  • AWS Bedrock offers native re-ranking; alternatively use Cohere Rerank or a fine-tuned cross-encoder
  • Worth it when precision matters more than latency (legal, compliance, medical RAG)
Cross-EncoderRerankingLatency
Q8 How do you handle multi-hop queries that require connecting information across multiple documents? Hard ▼
  • Iterative retrieval: First retrieval → LLM identifies what's missing → second targeted retrieval → final answer
  • Graph-based RAG: Build a knowledge graph alongside the vector index; traverse entity relationships
  • Query decomposition: Break the complex query into sub-questions, retrieve for each, synthesize
  • HyDE (Hypothetical Document Embeddings): Generate a hypothetical answer first, embed it, use that vector for retrieval — often surfaces better evidence
Multi-HopAgentic RAGHyDEGraph RAG
Q9 What causes context stuffing and how do you decide how many retrieved chunks to pass to the LLM? Medium ▼
  • Passing too many chunks increases cost and introduces noise — the LLM may ignore or contradict relevant sections (Lost-in-the-Middle effect)
  • Rule of thumb: 3–7 chunks for focused Q&A; up to 15 for summarization tasks
  • Set a token budget per chunk to guarantee all fit within the context window
  • Use relevance score thresholds: only pass chunks above a minimum cosine similarity score
  • Track: does adding more chunks improve answer faithfulness? Use RAGAS faithfulness metric
Context WindowRAGASToken Budget
Q10 How do you handle queries where no relevant document exists — avoiding confident hallucinations? Hard ▼
  • Relevance threshold guard: If the highest similarity score < 0.65 (tunable), return "I don't know" rather than passing low-signal chunks
  • Citation grounding: Instruct the LLM to cite specific passages; if it can't, it should say so
  • Explicit "no context" prompt path: A separate prompt branch when retrieval returns empty/low-confidence results
  • Confidence score injection: Pass retrieval scores into the prompt so the model has signal quality context
HallucinationGroundingConfidence
Q11 Describe metadata filtering in vector search. When does it help, and when can it hurt recall? Medium ▼
  • Helps: Tenant isolation, date-range filtering, document type scoping — avoids leaking irrelevant or unauthorized content
  • Hurts recall: Over-filtering shrinks the candidate pool; ANN search becomes less accurate at very small corpus sizes
  • Prefer post-filtering for coarse filters; pre-filtering only when strict access control requires it
  • In OpenSearch, use knn_score_script with a filter clause or Bedrock Knowledge Base filter expressions

Chunking Strategies
5 questions
Q12 Fixed-size vs. semantic chunking — what are the trade-offs and when do you choose each? Core ▼
  • Fixed-size (512 tokens + 64 overlap): Simple, predictable latency, works well for homogeneous prose
  • Semantic chunking: Splits on sentence boundaries, semantic similarity drops — better coherence but variable size and higher preprocessing cost
  • Structural chunking: Split on headings, sections — ideal for technical docs, PDFs with clear structure
  • Choose semantic/structural for legal, medical, or policy docs where a split mid-sentence can destroy meaning
ChunkingPreprocessingSemantic
Q13 What is the "lost context" problem and how does chunk overlap help — or not? Medium ▼

A chunk boundary can split a sentence or argument across two chunks, losing the connecting context in each.

  • Overlap (e.g., 10–15% of chunk size): Reduces boundary artifacts but inflates index size and embedding cost proportionally
  • Parent-child chunking: Store small chunks for retrieval precision; embed a larger parent chunk in the LLM context — best of both worlds
  • Overlap beyond ~20% yields diminishing returns and increases similarity collisions between adjacent chunks
OverlapParent-ChildContext
Q14 How do you chunk and embed tables, code blocks, or structured data vs. prose? Hard ▼
  • Tables: Serialize to Markdown or JSON; keep entire table as one chunk — splitting rows destroys meaning
  • Code blocks: Chunk at function/class boundaries; include docstrings + signature in the same chunk
  • Multi-modal (images/diagrams): Extract alt text or use a vision model to generate a text description for embedding
  • Consider late chunking (embed full doc first, then pool chunk embeddings) for very structured documents
TablesCodeMulti-Modal
Q15 How does chunk size affect retrieval recall vs. answer precision — what's your optimization approach? Medium ▼
  • Smaller chunks (128–256 tokens): Higher recall precision for targeted queries; may lack full context for generation
  • Larger chunks (512–1024 tokens): Better generative context; retrieval precision suffers as chunks are diluted
  • Optimize using offline eval: sweep chunk sizes over a labeled query set, measure RAGAS context precision and context recall
  • In practice: 512 tokens, 10% overlap is a solid default starting point
RecallPrecisionEval
Q16 What metadata should you attach to each chunk and why does it matter at query time? Core ▼
  • source_url / doc_id: for citation and deduplication
  • chunk_index, total_chunks: enables parent-doc reconstruction
  • ingestion_timestamp: for freshness filtering and version rollback
  • embedding_model: fast-fail on mismatch at query time
  • access_level / tenant_id: multi-tenant RBAC enforcement
  • content_hash: deduplication during incremental ingestion
MetadataRBACDeduplication

Evaluation & Quality
5 questions
Q17 Walk me through the RAGAS evaluation framework — what does each metric measure? Core ▼
  • Faithfulness: Is the answer supported by the retrieved context? (Hallucination detector)
  • Answer Relevancy: Does the answer address the original question?
  • Context Precision: Are the retrieved chunks actually relevant to the question?
  • Context Recall: Does the retrieved context contain all info needed to answer?
  • Answer Correctness: Does the answer match the ground-truth reference answer?

Run RAGAS continuously in CI/CD — a drop in Faithfulness is an early hallucination warning signal.

RAGASFaithfulnessEval
Q18 How do you build a ground-truth evaluation set for a RAG system when you have no labeled data? Hard ▼
  • LLM-generated synthetic Q&A: Use a strong model (Claude Opus, GPT-4) to generate question+answer pairs from your actual documents
  • User query mining: Sample real queries from prod logs; have SMEs annotate ground-truth answers
  • Adversarial set: Craft edge-case queries (out-of-scope, ambiguous, negation) to probe failure modes
  • Maintain a minimum of 200–500 labeled pairs across query types; version the eval set alongside the KB
Synthetic DataEval SetLLM-as-Judge
Q19 How do you use LLM-as-a-judge for RAG evaluation, and what are its limitations? Medium ▼

Use a separate, stronger LLM to score the output on dimensions like faithfulness, completeness, and coherence.

  • Strengths: Scalable, human-like judgment, no labeled test set required for quality scoring
  • Limitations: Position bias (favors first answer), verbosity bias (favors longer answers), self-preference bias if using same model family
  • Mitigations: Swap answer order, use different model families as judge, calibrate against human annotations
LLM JudgeBiasCalibration
Q20 What metrics do you track in production to monitor RAG system health beyond latency? Core ▼
  • Retrieval hit rate: % of queries where at least 1 relevant chunk retrieved (target >90%)
  • Mean reciprocal rank (MRR): How high in the ranking does the relevant chunk appear?
  • Faithfulness score: LLM-as-judge, track trend over time — drops signal KB or model regression
  • Null retrieval rate: % of queries below relevance threshold — spikes indicate KB gaps
  • Token usage per query: Directly maps to cost; unexpected spikes indicate chunking or retrieval issues
  • User feedback signals: Thumbs down, follow-up clarification requests as implicit relevance signals
MonitoringMRRCloudWatch
Q21 How do you run A/B tests for RAG pipeline changes without disrupting production users? Hard ▼
  • Route a small traffic percentage (5–10%) to the variant pipeline using feature flags or API Gateway stage variables
  • Ensure the split is deterministic (e.g., hash on user ID) — same user sees same variant within an experiment
  • Measure: latency, faithfulness, user satisfaction, cost-per-query
  • Run for sufficient traffic volume before concluding — avoid early stopping bias
  • Shadow mode: run both pipelines on all requests, compare outputs without serving variant answers — zero user risk

Infrastructure & Scalability
5 questions
Q22 How do you scale a RAG system to handle 10,000 queries per minute without proportionally increasing cost? Hard ▼
  • Semantic caching: Cache LLM responses for semantically similar queries (cosine similarity > 0.95); eliminates redundant LLM calls
  • ANN index replication: Horizontal scaling of the vector index with read replicas — retrieval is the high-throughput bottleneck
  • Async ingestion pipeline: Decouple ingestion from query serving; ingest via SQS/Lambda, serve via dedicated compute
  • Query batching: Batch embedding requests to reduce Bedrock API overhead
  • Tiered retrieval: Fast coarse ANN retrieval first → fine-grained re-ranking only for top candidates
CachingScalingANNSQS
Q23 OpenSearch Serverless vs. provisioned OpenSearch for vector workloads — how do you choose? Medium ▼
  • Serverless: Zero ops overhead, auto-scaling, no capacity planning — ideal for variable or unpredictable workloads; cold-start latency is a concern
  • Provisioned: Predictable latency, fine-grained control, better cost efficiency at consistent high throughput (>500 QPS sustained)
  • Bedrock Knowledge Bases integrates natively with OSS Serverless — lowest barrier to entry for RAG
  • For production at scale: provisioned with dedicated master nodes + warm tier for cost optimization
OpenSearchServerlessBedrock
Q24 How do you design a multi-tenant RAG system where one tenant's data must never be visible to another? Hard ▼
  • Index isolation (hard): Separate index per tenant — total isolation, higher ops overhead and cost
  • Namespace isolation (medium): Shared index with tenant_id metadata filter enforced on every query — simpler ops but requires airtight filter logic
  • Encryption isolation: Encrypt each tenant's vectors with a separate KMS key
  • Always validate tenant context at the API layer before passing to retrieval — never trust client-supplied tenant ID without server-side verification
  • Audit log every retrieval with tenant_id + user_id for compliance
Multi-TenantRBACKMSCompliance
Q25 How do you build a disaster recovery plan for a production RAG knowledge base? Hard ▼
  • Source of truth is the raw document store (S3): The vector index is always rebuildable — protect the corpus, not just the index
  • RTO target: How fast can you rebuild the index? Pre-warm a standby index in another region
  • RPO target: How much data can you lose? Continuous S3 replication + versioning ensures near-zero RPO
  • Runbook: Document the full rebuild pipeline; test it quarterly with a chaos drill
  • Cross-region Bedrock KB: use S3 CRR (Cross-Region Replication) + a secondary OSS cluster in the DR region
DRRTORPOS3
Q26 How does semantic caching work and what are the cache invalidation challenges in a RAG context? Medium ▼
  • Embed the incoming query; search a cache store (Redis + vector index) for a semantically similar prior query above a threshold
  • If cache hit: return cached response instantly without hitting the LLM — major cost/latency savings
  • Invalidation challenge: When the underlying KB version changes, cached answers may be stale — tie cache TTL to KB version and invalidate on KB promotion
  • Use short TTLs (15–60 min) for high-churn data; longer TTLs for stable knowledge bases
  • Store kb_version with each cache entry; reject hits where versions don't match current active

Security & Compliance
5 questions
Q27 How do you prevent prompt injection attacks in a RAG system where user queries touch retrieved content? Hard ▼
  • Input sanitization: Strip or escape prompt-like patterns in both user query and retrieved chunks before concatenation
  • Structural separation: Use XML/delimiter tags to clearly delineate user input from retrieved context in the prompt
  • Output validation: Run the LLM response through a classifier before returning — detect if it's following injected instructions
  • Privilege minimization: The RAG system prompt should not have access to actions (API calls, tool use) unless explicitly needed
  • Monitor for anomalous output patterns — sudden instruction-following language in responses is a signal
Prompt InjectionSecurityGuardrails
Q28 How do you comply with GDPR's right to erasure in a vector database where data is encoded in embeddings? Hard ▼
  • Design for deletion: Tag all PII-containing chunks with the subject's user ID at ingestion time
  • Hard delete pipeline: On erasure request, delete vectors matching the user ID tag from the index within the SLA window
  • Index rebuild as fallback: If soft-delete + cleanup isn't sufficient, a full index rebuild from a PII-scrubbed corpus is the clean-room approach
  • PII should ideally not be embedded at all — use anonymization or pseudonymization at ingestion before chunking
  • Document and audit the deletion pipeline for regulator evidence
GDPRPIIRight to Erasure
Q29 What is data poisoning in RAG and how do you defend against it? Hard ▼

Data poisoning: an adversary inserts malicious documents into the corpus that cause the RAG system to consistently return incorrect or harmful answers.

  • Source whitelisting: Only ingest from trusted, authenticated data sources — validate source identity at ingestion time
  • Content validation: Run toxicity classifiers and anomaly detectors on incoming documents before ingestion
  • Ingestion access control: Treat the ingestion pipeline with the same IAM rigor as write access to a production database
  • Human review for high-risk domains: Medical, legal, financial RAG — require SME approval for new document sources
  • Monitor: if answers on a stable topic suddenly change, investigate new document ingestions first
Data PoisoningAdversarialIAM
Q30 How do you implement Bedrock Guardrails and what gaps remain that you need to address at the application layer? Medium ▼
  • Bedrock Guardrails covers: content filtering (hate/violence/PII), topic denial lists, grounding checks (hallucination detection), and word filters
  • Gaps: Business-logic violations (wrong product info, pricing errors), contextual policy rules, complex multi-turn jailbreaks, RAG-specific prompt injection via poisoned chunks
  • Application-layer additions: output schema validation, business rule classifiers, session-level abuse detection
  • Layer Guardrails + application logic + monitoring — don't rely on any single control
Bedrock GuardrailsComplianceDefense in Depth
Q31 How do you audit and explain which retrieved sources contributed to a RAG answer for compliance? Medium ▼
  • Source attribution: Return the source document IDs and chunk indices alongside every response
  • Structured audit log: Log query, kb_version, retrieved_chunks[], scores[], llm_response, user_id, timestamp to an immutable store (S3 + Athena)
  • Citation grounding in prompt: Instruct the LLM to reference specific passages — makes attribution machine-readable
  • Store the full context window sent to the LLM in the audit log — enables full replay and investigation
  • Separate audit log access from application access — compliance teams need read-only access without prod access

Cost Optimization
4 questions
Q32 What are the main cost levers in a RAG system and how do you instrument them? Core ▼
  • Embedding cost: Tokens ingested × embedding model price — optimize chunk size, avoid redundant re-embeds
  • Vector index cost: Storage + query OCU hours (OpenSearch Serverless) — right-size index, use cold tier for old versions
  • LLM input cost: Retrieved context tokens + prompt — reduce chunk count, use smaller model for simple queries
  • LLM output cost: Max tokens per response — set appropriate limits by query type
  • Instrument: tag all Bedrock API calls with cost_center + query_type tags; view in AWS Cost Explorer by dimension
FinOpsCost ExplorerBedrock
Q33 How does model routing (using smaller LLMs for simple queries) work in a RAG system? Medium ▼
  • Query classifier: A lightweight model (or rule-based logic) classifies each query as simple/complex/high-stakes
  • Simple factual queries → Haiku/Lite model (10–50× cheaper)
  • Multi-step reasoning → Sonnet; high-stakes compliance → Opus
  • Can reduce LLM costs by 60–80% with minimal quality degradation on routine queries
  • Monitor quality per route: if Haiku faithfulness drops below threshold, escalate to Sonnet
Model RoutingFinOpsClaude Haiku
Q34 How do you use Bedrock prompt caching to reduce latency and cost for RAG workloads? Medium ▼
  • Prompt caching saves the KV cache of the static prompt prefix — if multiple queries share the same system prompt + context, subsequent calls skip re-processing those tokens
  • Best for: shared system prompts, static document context that many queries use, multi-turn conversations
  • Mark the cacheable prefix with cache_control: ephemeral — Bedrock reuses it for up to 5 minutes
  • Can reduce input token cost by up to 90% and latency by up to 85% for cache hits
  • Ensure the cacheable portion is truly static — any variation breaks the cache hit
Prompt CachingBedrockLatency
Q35 How do you right-size vector index capacity for a corpus that grows 20% monthly? Medium ▼
  • Baseline: calculate storage from num_docs × avg_chunks_per_doc × embedding_dim × 4 bytes
  • Project 6-month growth; provision for 12 months to avoid disruptive re-sharding
  • OpenSearch: use index.number_of_shards based on 30–50GB/shard target; over-sharding hurts performance
  • Move old KB versions to warm/cold tier (UltraWarm) — read-only, 90% cheaper than hot tier
  • Set CloudWatch alarms on FreeStorageSpace and JVMMemoryPressure — scale before you hit limits
Capacity PlanningShardingUltraWarm

Failure Modes & Debugging
5 questions
Q36 You deploy a new KB version and answer quality drops sharply. Walk me through your RCA process. Hard ▼
  • Immediate: Rollback to prior KB version — restore quality first, investigate second
  • Layer 1 — retrieval: Run golden query suite; check if retrieved chunks changed or relevance scores dropped
  • Layer 2 — chunking: Did chunk size, overlap, or splitter logic change in this ingestion run?
  • Layer 3 — embedding: Was a different embedding model or API endpoint used? Check embedding_model metadata
  • Layer 4 — source data: Did the raw documents change format, language, or structure?
  • Layer 5 — LLM: Was a model version change deployed simultaneously? Isolate variables
RCADebuggingRollback
Q37 What happens to RAG quality when the LLM context window is exceeded, and how do you handle it? Medium ▼
  • Most APIs truncate silently or return an error — both are catastrophic for answer quality without guardrails
  • Defensive token budgeting: Calculate system_prompt + retrieved_chunks + user_query + max_output at construction time; truncate chunks if over budget
  • Chunk prioritization: If truncating, remove lowest-scoring chunks first
  • Detect and alert: Log when token budget is exceeded — it's a signal that retrieval is returning too much or chunks are too large
  • Consider a summarization step: summarize retrieved context before passing to the final generation LLM
Context WindowTruncationToken Budget
Q38 How do you debug "retrieval succeeds but generation fails" — when the right chunks are retrieved but the answer is still wrong? Hard ▼
  • Prompt template issues: Retrieved content is correct but the prompt doesn't instruct the LLM to use it — check prompt structure and instructions
  • Lost-in-the-Middle: Relevant chunk is retrieved but buried in position 4–6 of 7 — move higher-scoring chunks to the beginning and end of context
  • Conflicting chunks: Two retrieved chunks contradict each other — the LLM may hallucinate to resolve — add deduplication and contradiction detection
  • Model capability gap: The reasoning required exceeds the model's ability — try a stronger model or chain-of-thought prompting
  • Format mismatch: Chunks are in a format the LLM can't parse well (e.g., HTML tables, garbled PDF extraction)
GenerationPrompt EngineeringDebugging
Q39 How do you handle Bedrock throttling errors at scale without user-visible failures? Medium ▼
  • Exponential backoff with jitter: Standard retry pattern — avoid thundering herd after throttle events
  • Provisioned Throughput: Pre-purchase model units for predictable high-volume workloads — eliminates on-demand throttling
  • Request queuing: SQS buffer between your application and Bedrock — absorbs spikes, smooths throughput
  • Multi-model fallback: If primary model is throttled, route to a secondary model (e.g., Haiku as Sonnet fallback)
  • Graceful degradation: If all retries fail, return a meaningful "service temporarily unavailable" — never surface raw API errors
ThrottlingRetryBedrockSQS
Q40 Your RAG system is returning consistent answers but they're 6 hours stale. How do you diagnose the ingestion pipeline? Medium ▼
  • Check active_version pointer timestamp — is the KB version stuck on an old ingestion?
  • Inspect ingestion pipeline CloudWatch logs — did the hourly trigger fire? Did it fail silently?
  • Check SQS dead-letter queue — failed ingestion events may be accumulating there
  • Validate that the switchover logic ran — validation passing but pointer not updated is a common bug
  • Add a freshness alarm: alert if the active KB version timestamp is > 90 minutes old — catch stale KB before users notice
