01

Kubernetes, the way it actually runs

Know the moving parts before you talk about EKS — interviewers test whether "managed control plane" means something concrete to you.

A cluster splits into a control plane (the brain that decides what should run) and a data plane of worker nodes (where containers actually run). In EKS, AWS runs and scales the control plane for you across availability zones — you never SSH into it. You own the worker nodes.

Control plane components

  • kube-apiserver — the front door. Every kubectl call, every controller, every node talks to it. It's the only component that reads/writes etcd.
  • etcd — the distributed key-value store holding the entire cluster state (the "source of truth" for what exists). AWS manages and backs this up in EKS.
  • kube-scheduler — decides which node a new pod lands on, using resource requests, taints/tolerations, and affinity rules.
  • kube-controller-manager — runs reconciliation loops (Deployment, ReplicaSet, Node controllers) that drive actual state toward desired state.
  • cloud-controller-manager — the AWS integration point: provisions load balancers, EBS volumes, and node lifecycle against the AWS API.

Data plane (your nodes)

  • kubelet — the agent on each node; it talks to the API server, starts/stops containers, and reports health.
  • container runtime — containerd on modern EKS, which pulls images and runs containers.
  • kube-proxy — programs the node's networking rules so traffic to a Service reaches the right pods.

The objects you'll actually deploy

  • Pod — smallest unit; one or more containers sharing a network namespace. You rarely create these directly.
  • Deployment → ReplicaSet → Pods — for stateless apps; handles rolling updates and rollbacks.
  • StatefulSet — for stateful apps needing stable identity and ordered startup (databases, brokers).
  • Service — stable virtual IP/DNS in front of a changing set of pods.
  • Ingress — HTTP(S) routing rules into the cluster, fulfilled by an ingress controller.
  • ConfigMap / Secret — externalized configuration and sensitive values.
Interview gold

The single sentence that ties it together: "Kubernetes is a set of reconciliation loops — controllers continuously compare desired state in etcd to observed state and act to close the gap." That framing is exactly what ArgoCD extends out to Git.

02

Choosing EKS over ECS

The classic opening question. Lead with the decision criteria, then the concrete limits — not "Kubernetes is better."

ECS is AWS's own orchestrator: simpler, tightly integrated, low operational overhead. EKS is managed Kubernetes: portable, standards-based, richer ecosystem, but more to operate. The honest framing is that most AWS-only teams don't need EKS — so be ready to justify when it's actually the right call.

DimensionECSEKS
Operational overheadLow — AWS owns most of itHigher — you own networking, RBAC, add-ons
PortabilityAWS-locked task definitionsStandard k8s manifests, multi-cloud capable
SchedulingSimple task placementAffinity, anti-affinity, topology spread, taints
EcosystemAWS-native add-ons onlyHelm, operators, service mesh, ArgoCD, etc.
Stateful workloadsYou wire it up manuallyStatefulSets + CSI drivers natively
Team skills neededAWS familiarityKubernetes expertise

What ECS genuinely can't do well (beyond multi-cloud)

  • Fine-grained scheduling — pod affinity/anti-affinity, topology spread, and node taints for heterogeneous workloads. ECS placement is far coarser.
  • Ecosystem & extensibility — Custom Resources and operators let you extend the API (ArgoCD, cert-manager, External Secrets, service meshes). ECS has no equivalent extension model.
  • Stateful orchestration — StatefulSets give stable network identity and ordered rollout; on ECS you're hand-rolling that.
  • Declarative GitOps — the whole ArgoCD model assumes a Kubernetes API to reconcile against. This is likely the reason a GitOps shop like the one you're interviewing with runs EKS.
They'll probe

"When would you not pick EKS?" — Have an answer ready: a small team, a handful of stateless services, no multi-cloud need, and no appetite for k8s operational burden. Showing you'd choose ECS or Fargate there proves you're picking tools for the problem, not résumé-driven.

03

Standing up the cluster

Be able to narrate provisioning end-to-end and name the IAM roles involved — this is bread-and-butter for the role.

The prerequisites

  • VPC across ≥2 AZs with public subnets (for load balancers / NAT) and private subnets (for worker nodes). Subnets need the right tags so the cluster can place load balancers correctly.
  • Cluster IAM role — assumed by the EKS control plane to manage AWS resources on your behalf.
  • Node IAM role — attached to worker nodes; needs policies for the worker kubelet, the VPC CNI, and ECR pull access.
  • OIDC provider — enables IRSA so pods can assume IAM roles (covered in §05).

Compute options for the data plane

  • Managed node groups — AWS-managed EC2 Auto Scaling Groups with handled upgrades/draining. The common default.
  • Self-managed nodes — full control, more work; for custom AMIs or special requirements.
  • Fargate — serverless pods, no nodes to manage; great for spiky or isolated workloads, with cost and feature trade-offs.
  • Karpenter — modern just-in-time node provisioner that picks right-sized instances per pending pod (see §07 and §09).

Provision with eksctl (fast path)

# Creates VPC, control plane, and a managed node group
eksctl create cluster \
  --name mugen-prod \
  --region us-west-2 \
  --version 1.30 \
  --nodegroup-name app-workers \
  --node-type m6i.large \
  --nodes 3 --nodes-min 2 --nodes-max 6 \
  --with-oidc --managed
Interview gold

In production you'd do this in Terraform (or CDK) — likely the terraform-aws-eks module — so the cluster itself is version-controlled and reviewable. Mention that infra-as-code is the foundation that makes GitOps for apps coherent: everything is declarative, just at two layers.

04

Networking

The layer interviewers push hardest on, because it's where EKS differs most from vanilla Kubernetes.

The AWS VPC CNI — the key EKS detail

EKS uses the AWS VPC CNI by default, which gives every pod a real IP address from your VPC subnet via secondary ENIs on the node. The upside is native VPC routing and security-group-level control; the catch is that pod density is bounded by how many IPs an instance type can attach, so IP/subnet planning is a real design concern.

Service discovery & traffic

  • CoreDNS resolves Service names to cluster IPs — in-cluster DNS.
  • kube-proxy programs each node so traffic to a Service IP load-balances across healthy pods.
  • Service types: ClusterIP (internal), NodePort (port on every node), LoadBalancer (provisions an AWS NLB).
  • AWS Load Balancer Controller turns Ingress objects into ALBs (L7 routing) and Service type=LoadBalancer into NLBs (L4). This is how external traffic typically enters.

Things they may dig into

  • Security groups for pods — apply SGs at pod granularity, not just node level.
  • NetworkPolicies — k8s-native east-west traffic rules (allow/deny between pods/namespaces).
  • ALB vs NLB — ALB for HTTP(S) path/host routing and TLS termination; NLB for raw TCP/UDP, ultra-low latency, static IPs.
  • IP exhaustion — a real EKS failure mode; mitigations include prefix delegation and well-sized subnets.
They'll probe

"A pod is stuck in ContainerCreating with a network error — what's your first guess?" Strong answer: likely IP exhaustion on the node/subnet from the VPC CNI, or a failing CNI add-on. You'd check kubectl describe pod events, available IPs on the ENI, and the aws-node daemonset.

05

Identity: IAM ↔ RBAC

Two separate permission systems meet here. Being crisp about which is which is a strong signal.

There are two distinct layers, and conflating them is a common stumble:

  • AWS IAM controls what can call the AWS API (read S3, pull from ECR, get a Secret).
  • Kubernetes RBAC controls what can call the Kubernetes API (create deployments, read pods in a namespace).

IRSA — pods that need AWS access

IAM Roles for Service Accounts (IRSA) is the right way to give a pod AWS permissions. Via the cluster's OIDC provider, a Kubernetes ServiceAccount is mapped to an IAM role, so the pod gets scoped, temporary credentials — no static keys on nodes. (EKS Pod Identity is the newer, simpler alternative worth name-dropping.)

# ServiceAccount annotated with the IAM role it assumes
apiVersion: v1
kind: ServiceAccount
metadata:
  name: checkout-sa
  namespace: payments
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::1234:role/checkout-irsa

Mapping humans & nodes into the cluster

  • EKS access entries (modern) or the legacy aws-auth ConfigMap map IAM principals to Kubernetes RBAC groups — that's how an IAM user/role becomes a cluster admin or read-only viewer.
  • RBAC objects: Role/ClusterRole define permissions; RoleBinding/ClusterRoleBinding attach them to users, groups, or service accounts.
Common pitfall

Don't attach broad IAM policies to the node role so "pods can reach AWS" — that gives every pod on the node those rights. The whole point of IRSA is least-privilege per workload. Calling this out unprompted reads as production maturity.

06

Storage & state

Know the access-mode trade-offs and which CSI driver fits which need.

EBS CSIEFS CSI
Access modeReadWriteOnce — one node at a timeReadWriteMany — many pods/AZs at once
ScopeSingle AZ (volume is AZ-bound)Multi-AZ shared file system
Best forDatabases, per-pod block storageShared assets, content read by many pods
PerformanceLow-latency blockNetwork file system; higher latency

How it fits together

  • StorageClass defines how volumes get provisioned (which driver, type, encryption) and enables dynamic provisioning.
  • PersistentVolumeClaim (PVC) is a pod's request for storage; it binds to a PersistentVolume (PV).
  • StatefulSet uses volumeClaimTemplates to give each replica its own stable EBS volume that survives reschedules.
They'll probe

"Your pod with an EBS volume won't schedule after a node failure — why?" EBS volumes are AZ-bound; if the scheduler tries to place the pod in a different AZ from the volume, it can't attach. You'd use topology-aware scheduling or pin to the volume's AZ. Naming this is a strong production tell.

07

Autoscaling, two layers

Separate scaling pods from scaling nodes — interviewers love when candidates keep these distinct.

  • HPA (Horizontal Pod Autoscaler) — adds/removes pod replicas based on CPU, memory, or custom metrics. Needs metrics-server installed.
  • VPA (Vertical Pod Autoscaler) — adjusts a pod's CPU/memory requests; usually for right-sizing, not run alongside HPA on the same metric.
  • Cluster Autoscaler — adds/removes nodes in a node group when pods can't be scheduled or nodes sit idle.
  • Karpenter — the modern replacement: provisions right-sized nodes directly (any instance type/AZ) in seconds, consolidates underused nodes, and natively favors Spot. Increasingly the production default and great to mention.
Interview gold

The clean mental model: "HPA scales pods to meet demand; Cluster Autoscaler or Karpenter scales nodes to fit the pods." Two loops at two layers — and both depend on accurate resource requests, which is also your main cost lever (§09).

08

Observability & debugging

Expect a "production is broken, walk me through it" scenario. Have a calm, ordered method.

The stack

  • Metrics — CloudWatch Container Insights, or Prometheus + Grafana (the common k8s-native combo). AWS offers managed Prometheus/Grafana.
  • Logs — Fluent Bit daemonset ships container logs to CloudWatch Logs or another backend.
  • Traces — OpenTelemetry / AWS Distro for OpenTelemetry for distributed tracing across services.

The debugging sequence (say it out loud in this order)

# 1. What's the pod's status and recent events?
kubectl get pods -n payments
kubectl describe pod checkout-7f9 -n payments   # events at bottom

# 2. What is the app itself saying?
kubectl logs checkout-7f9 -n payments --previous   # crash loop? prior logs

# 3. Cluster-wide events & node pressure
kubectl get events -n payments --sort-by=.lastTimestamp
kubectl top nodes

Failure states to recognize on sight

  • CrashLoopBackOff — app starts then exits; check logs and the readiness/liveness probes.
  • ImagePullBackOff — can't pull from ECR; usually IAM/ECR permission or a bad tag.
  • Pending — scheduler can't place it; insufficient resources, taints, or no node matches (ties to §07).
  • OOMKilled — exceeded its memory limit; revisit requests/limits.
09

GitOps with ArgoCD

The center of gravity for this role. Be fluent in the model, the repo layout, and how drift is handled.

The principle

GitOps treats infrastructure and app config like source code: Git is the single source of truth for desired state. ArgoCD runs in-cluster, continuously watches the config repo, compares it to the live cluster, and reconciles — extending Kubernetes' own control-loop idea out to Git. No one runs kubectl apply by hand; you change the cluster by merging a PR.

The two-repo pattern (you described this exactly)

  • App repo — application source code. On merge, CI builds the image and pushes it to ECR with a new tag.
  • Config / manifest repo — Kubernetes manifests (or Helm/Kustomize). A pipeline step or ArgoCD Image Updater bumps the image tag here.
  • ArgoCD watches the config repo, sees the new tag as a diff, and syncs it to EKS.

Separation of concerns is the "why": developers own code and CI; platform owns deployment config; the cluster is never changed by hand, so the audit trail is just Git history. Rollback = revert a commit.

Drift detection & self-healing

Drift is when live cluster state diverges from Git — someone hand-scales a deployment, or something mutates a resource. ArgoCD continuously diffs and marks the app OutOfSync. You can either alert on it or enable self-heal so ArgoCD forces the cluster back to match Git. With auto-sync + prune + self-heal, Git is genuinely authoritative.

An ArgoCD Application with auto-sync, prune, and self-heal

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: checkout
  namespace: argocd
spec:
  project: payments
  source:
    repoURL: https://git/mugen/manifests.git
    path: apps/checkout/overlays/prod
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: payments
  syncPolicy:
    automated:
      prune: true        # delete resources removed from Git
      selfHeal: true     # revert manual drift

Patterns they'll likely name-check

  • App-of-apps — one parent Application that points at a folder of child Applications, so ArgoCD bootstraps and manages many apps from one root.
  • ApplicationSet — templates Applications across many targets (per-environment, per-cluster, per-team) from a generator — the scalable answer for fleets.
  • Helm vs Kustomize — ArgoCD renders both; Helm for packaged/parameterized charts, Kustomize for overlay-based env differences. Be ready to defend a preference.
  • Sync waves & hooks — order resource application (e.g., run a DB migration before the app rolls out).

Secrets — the question that separates people

You never commit plaintext secrets to Git. The two standard answers:

  • External Secrets Operator — pulls from AWS Secrets Manager / SSM Parameter Store into k8s Secrets at runtime. Common with EKS because it reuses IRSA.
  • Sealed Secrets / SOPS — encrypt secrets so the ciphertext can live safely in Git and is decrypted in-cluster.
Interview gold

Multi-environment promotion in their language: "Dev/staging/prod are separate overlays (or branches) in the config repo. Promotion is a PR that moves a tested image tag from one overlay to the next — so every environment change is reviewed, audited, and trivially revertible."

10

Cost optimization

DevOps owns the bill. A few sharp levers signal you operate, not just deploy.

  • Right-size with requests/limits — the biggest lever. Over-requested pods strand node capacity and inflate node count; accurate requests pack nodes efficiently.
  • Spot instances — big savings for fault-tolerant/stateless workloads; Karpenter handles Spot interruptions and diversification well.
  • Karpenter consolidation — actively bin-packs and replaces underused nodes with cheaper right-sized ones.
  • Fargate for spiky/bursty — pay per pod, no idle nodes, when traffic is uneven.
  • Scale to zero — dev/preview environments shut down off-hours (a natural ArgoCD/ApplicationSet fit).
  • Visibility — Kubecost or AWS split-cost allocation tags to attribute spend per namespace/team.
Interview gold

Tie cost back to scheduling: "Cost in EKS is mostly a bin-packing problem — accurate resource requests plus Karpenter consolidation and Spot is where the real savings are, well before reserved instances."

11

Rapid-fire rehearsal

Tap each to reveal a tight answer. Say them aloud — fluency beats memorization.

QWhat does AWS manage in EKS vs. what do you own?

AWS manages the control plane — API server, etcd, scheduler, controller-manager — across AZs, including patching and backups. You own the worker nodes (or use Fargate/Karpenter), networking config, RBAC, add-ons, and the workloads.

QWhy does the VPC CNI matter for capacity planning?

It gives each pod a real VPC IP via node ENIs, so the number of pods per node is capped by the instance's IP allotment, and large clusters can exhaust subnet IPs. You plan subnet sizing accordingly and can use prefix delegation to raise density.

QIRSA in one breath?

It maps a Kubernetes ServiceAccount to an IAM role via the cluster's OIDC provider, giving a pod scoped, temporary AWS credentials — least privilege per workload, no static keys on nodes.

QHow does ArgoCD know something drifted?

It continuously diffs live cluster state against the desired state in Git. Any divergence flips the Application to OutOfSync; with self-heal on, it reapplies Git to revert the drift automatically.

QWhy two repos instead of one?

Separation of concerns and clean audit trails. Devs own app code + CI in one repo; platform owns deployment config in the other. The cluster only ever changes via the config repo, so Git history is the deployment record and rollback is a revert.

QPod stuck Pending — how do you diagnose?

kubectl describe pod and read the events. Usual causes: insufficient CPU/memory on any node (autoscaler should add one), a taint with no matching toleration, a node-selector/affinity mismatch, or an unbound PVC in the wrong AZ.

QHPA vs Cluster Autoscaler vs Karpenter?

HPA scales pod replicas on metrics. Cluster Autoscaler adds/removes nodes within fixed node groups. Karpenter provisions right-sized nodes just-in-time across instance types/AZs and consolidates idle capacity — the modern node-scaling choice.

QHow do you handle secrets in GitOps?

Never plaintext in Git. Either External Secrets Operator pulling from AWS Secrets Manager/SSM at runtime (reuses IRSA), or Sealed Secrets/SOPS where only encrypted ciphertext lives in Git and is decrypted in-cluster.

QApp-of-apps vs ApplicationSet?

App-of-apps is a parent Application managing a folder of child Applications — good for bootstrapping a set. ApplicationSet templates Applications from a generator across many environments/clusters/teams — the scalable choice for fleets.

QBiggest EKS cost lever?

Accurate resource requests/limits — it's a bin-packing problem. Right-sized requests plus Karpenter consolidation and Spot capacity for fault-tolerant workloads beat reserved-instance discounts as a starting point.
